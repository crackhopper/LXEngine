# 06 · 演进路径

> 阅读前提：[05](05-LX当前状态对照.md) 已经盘点 LX 现状与教科书形态的差距。本文给出推进路径切分。
>
> 注：本文给出的 REQ-A..G 编号是 *候选编号*，正式立项时按 `notes/requirements/` 实际可用号顺延。

## 6.1 总体策略

**结论先行：不是一次大重构，而是 7 个可独立合 PR 的 REQ**。

每个 REQ：

- 有清晰的输入输出
- 完成后工程仍可编译可测
- 不依赖后续 REQ 即可单独评估
- 风险点（barrier 推导 / aliasing）单独立 REQ，便于聚焦验证

**前置条件已锁定**：[REQ-042](../../../requirements/042-render-target-desc-and-target.md) 把 `RenderTarget` 拆为 `RenderTargetDesc` + `RenderTarget`。在 REQ-042 落地之前不要开 REQ-A。

## 6.2 七步演进序列

### REQ-A · FrameGraphResource 数据结构 + 跨 pass 资源命名

**目标**：引入"跨 pass 资源"这个概念。还不做 JSON，先用代码 API。

**改动**：

- 新增 `FrameGraphResource` 结构：`type / resource_info / producer / output_handle / ref_count / name`
- `FramePass` 加 `inputs: vector<FrameGraphResourceHandle>` 和 `outputs: vector<FrameGraphResourceHandle>` 字段
- `FrameGraphBuilder`（新类）持资源池 + 名字 → handle 索引
- `FrameGraph::addPass(builder, pass)` 注册资源到 builder

**验收**：可以用代码 API 表达"node A 写 X，node B 读 X"，但暂不强求自动推断顺序（pass 顺序仍由 addPass 顺序决定）。

**风险**：低。纯数据结构 + 一层间接索引。

### REQ-B · compile 阶段 v1 — 构边 + 拓扑 + cycle 检测

**目标**：让 pass 顺序由 DAG 决定，不再由 `addPass` 顺序硬定。

**改动**：

- `FrameGraph::compile()` 新方法
- 内部按 [04 章 4.1](04-compile-阶段.md#41-构边compute-edges) 算法 resolve producer + 构边
- 内部按 [04 章 4.2](04-compile-阶段.md#42-拓扑排序dfs--三态-visited) 算法做 DFS + 三态 visited 拓扑排序
- cycle 检测：visited[child] == 1 时 `FATAL + terminate`，错误信息带 cycle 路径

**验收**：

- 给一组测试 graph（包括环、菱形、链），拓扑序正确
- cycle 触发清晰错误信息

**风险**：低。教科书算法。

### REQ-C · 自动 renderpass / framebuffer 创建

**目标**：把现在散在 backend 里的 renderpass / framebuffer 代码搬到 frame graph compile 阶段。

**改动**：

- `FrameGraph::compile()` 后半段：根据每个 node 的 attachment 类型 inputs/outputs，自动建 `VkRenderPassCreateInfo` 和 `VkFramebuffer`
- backend 提供 builder.create_render_pass() / create_framebuffer() 接口
- `FramePass` 加 `render_pass: RenderPassHandle` 和 `framebuffer: FramebufferHandle` 字段
- 移除 backend 里现有的手写 renderpass / framebuffer 代码

**验收**：现有 forward demo 仍工作，但 renderpass / framebuffer 由 frame graph compile 产生。

**风险**：中。需要小心 backend 接口的设计，避免 leak Vulkan 类型到 core 层。

### REQ-D · 自动 barrier 推导（**风险点 #1**）

**目标**：消除手工 barrier 的整类 bug。每两个相邻 pass 之间扫资源转换，自动插入 barrier。

**改动**：

- `FrameGraph::compile()` 增加 barrier 推导阶段
- 对每条资源边，确定起点 layout / access mask 和终点 layout / access mask
- 把推导结果存进 node 元数据，render 阶段 replay
- backend 提供 `add_image_barrier()` / `add_buffer_barrier()` 工具

**验收**：

- 启用 Vulkan validation layer 跑 demo 全程无 warning
- 手动构造跨 pass 资源场景（写后读、读后写、reference 模式），barrier 正确

**风险**：**高**。barrier 推导细节多（srcStage / dstStage / srcAccess / dstAccess 的组合 + queue family ownership transfer），错了的症状是花屏 / validation error 但功能"看起来"对，难调试。

**配套要求**：必须配 Vulkan validation layer 强测试 + 跨 pass 资源场景 fixture。

### REQ-E · JSON parser + schema

**目标**：让 frame graph 真正数据驱动。把 graph 定义从代码搬到 JSON。

**改动**：

- 定义 JSON schema（节点 / 资源类型 / external 标记 / load/store ops）
- 实现 parser：JSON → `FrameGraph` 结构
- 错误模式：字段缺失 / 类型错 / 引用未定义资源 / cycle，全部 fail-fast 带行号
- 保留代码 API 作为底层（parser 调代码 API），便于测试

**验收**：

- demo 的 frame graph 改为 JSON 描述
- 改 attachment format / 增删 pass 不需重编
- 错误 JSON 触发清晰错误信息

**风险**：中。工程量大但风险低。

### REQ-F · attachment aliasing（**风险点 #2**）

**目标**：让 attachment 复用显存。

**改动**：

- compile 阶段算 ref_count + 维护 free_list
- 按 [04 章 4.3](04-compile-阶段.md#43-resource-aliasing) 算法做 greedy 分配
- 桌面端默认开启，提供 `--no-aliasing` 关闭做调试 / 对比

**验收**：

- 显存占用对比：典型场景节省 30%+
- aliasing 后画面与不 aliasing 画面像素一致
- 没有 use-after-free 或 race condition（依赖 REQ-D barrier 正确性）

**风险**：**高**。aliasing 错了的症状是 *偶发* 画面错误（具体取决于驱动 alias 策略），极难复现和定位。

**配套要求**：

- 必须先有 REQ-D barrier 自动推导（否则无法保证 aliasing 安全条件）
- 强 fuzz 测试 + 跨 GPU 厂商验证

### REQ-G · runtime register_pass 策略热替换

**目标**：支持运行期换 pass 实现，编辑器 / A-B 测试用。

**改动**：

- `FrameGraphRenderPass` 抽象基类（用户实现）
- `FrameGraphBuilder::register_render_pass(name, impl)` 接口
- `FrameGraph::render()` 通过 name 查策略
- 提供 swap API，运行期替换某个 pass 的实现

**验收**：

- 一个 pass 在运行期切换两个实现，画面立即响应
- swap 不需要重 compile

**风险**：低。架构层面的 polish，不是新算法。

   ↓

## 6.6 工业级延伸（不在本路线范围）

完成 REQ-G 后离工业级 frame graph 还有以下方向，建议不进 v1 路线：

- 多 queue 调度（graphics / compute / transfer 异步执行 + queue ownership transfer）
- subpass merging（连续 attachment-only pass 合并成 Vulkan 多 subpass，省中间 barrier）
- 增量 invalidate（resize 不重 compile 整图，只失效受影响子图）
- pipeline 兼容性放宽（Vulkan 允许 attachment formats 相同的 pipeline 跨 renderpass 复用）
- conditional pass（运行期决定是否启用某些 pass，比如 debug overlay）
- 更优 aliasing 算法（bin packing / 约束求解）

这些都是 *优化层* 的能力，等 v1 跑稳定且有真实瓶颈再考虑。
