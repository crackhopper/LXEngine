# 02 · 数据驱动方式

> 阅读前提：[01](01-frame-graph-是什么.md) 已经讲清 frame graph 是 DAG。本文讨论"用什么方式来描述这个图"。

## 2.1 两种构建路径

### 代码驱动（LX 当前 + 多数小项目）

```cpp
FrameGraph fg;
fg.addPass(shadow_pass);
fg.addPass(gbuffer_pass);
fg.addPass(light_pass);
```

- ✅ 类型安全，编译器查
- ✅ IDE 跳转方便
- ❌ 改 attachment 大小 / 格式都要重编
- ❌ 增删 pass 要改多个文件
- ❌ 整张图的样子要把代码读一遍才能看清
- ❌ 美术 / TA 无法参与

### 数据驱动（书 + 多数中大型引擎）

```json
{
  "passes": [
    { "name": "shadow_pass", ... },
    { "name": "gbuffer_pass", ... },
    { "name": "light_pass", ... }
  ]
}
```

- ✅ 改 attachment 不重编
- ✅ 一个文件看清整张图
- ✅ 可上 visual editor，美术 / TA 能改
- ✅ 运行期 reload 调试不同方案
- ❌ 引入运行期 parse 失败模式（schema 校验 / 引用解析）
- ❌ IDE 不能跳转 JSON 字符串
- ❌ 需要写一套 schema + parser

## 2.2 节点的 JSON 结构

教科书示例：

```json
{
  "name": "gbuffer_pass",
  "inputs": [
    { "type": "attachment", "name": "depth" }
  ],
  "outputs": [
    {
      "type": "attachment",
      "name": "gbuffer_colour",
      "format": "VK_FORMAT_B8G8R8A8_UNORM",
      "resolution": [1280, 800],
      "op": "VK_ATTACHMENT_LOAD_OP_CLEAR"
    },
    {
      "type": "attachment",
      "name": "gbuffer_normals",
      "format": "VK_FORMAT_R16G16B16A16_SFLOAT",
      "resolution": [1280, 800],
      "op": "VK_ATTACHMENT_LOAD_OP_CLEAR"
    }
  ]
}
```

每个节点三个核心字段：

### `name` — 节点的唯一身份

字符串名字。后续 `register_render_pass(name, impl)` 通过它找到节点；调试 log 也用它。

LX 等价物：`StringID`（编译期常量化的字符串）。运行期 string 与编译期 StringID 是等价语义的两种实现，性能 / 类型安全权衡不同。

### `inputs` — 我消费什么资源

列表，每项是"我需要的某个资源"。**这些资源必须由别的 node 产生**（少数例外：external resource，比如 swapchain image 从外面注入）。

inputs 表达"我等谁完成"。frame graph 通过 inputs 反推依赖。

### `outputs` — 我产生什么资源

列表，每项是"我会写出的某个资源"。每个 output 必须带完整描述（format / resolution / loadOp 等），因为它是 *规范资源声明*。

后续 pass 在 inputs 里只引用 `name`（不重复 format / resolution），元数据已在原始 output 处声明。

## 2.3 四种资源类型

资源类型决定它怎么参与 renderpass 构造、怎么插 barrier。这是 frame graph 里最关键的概念。

### 2.3.1 attachment

**它是 framebuffer 的一部分**。一个 pass 把它列在 outputs 里，意味着这个 pass 在 GPU 上会写到这块 image，并且 *作为 renderpass 的 attachment*。

具体到 Vulkan：决定 `VkRenderPassCreateInfo.pAttachments` 列表，决定 `VkFramebuffer` 绑哪些 image view。

**attachment 也可以做 input**：表示 *复用上一个 pass 写的 attachment*。比如 depth pre-pass 写 depth → gbuffer 复用 depth（`LOAD` 而不是 `CLEAR`）。

LX 等价物：`RenderTarget.colorFormat / depthFormat` 描述 *output attachment*。但目前没办法表达 *input attachment*（"复用上个 pass 的"），因为没有跨 pass 资源概念。

### 2.3.2 texture

**它在 pass 里被 shader 采样读取**。区别于 attachment 的关键：

- attachment：framebuffer 写入目标，layout `*_ATTACHMENT_OPTIMAL`
- texture：shader 通过 descriptor set 绑定，`texture(sampler, uv)` 读取，layout `SHADER_READ_ONLY_OPTIMAL`

为什么要拆这两种？因为同一张 GPU image 在两个 pass 里可以分别扮演这两种角色，而 GPU image 的 *layout* 不同。frame graph 必须在两次使用之间插入一个 *layout transition barrier*。如果不区分类型，frame graph 不知道该不该插 barrier。

例：SSAO pass 写出 `ssao_texture`（output 是 attachment）；下一个 lighting pass 把它作为 texture input 采样。frame graph 看到"SSAO output type=attachment, lighting input type=texture, name 相同" → 在 lighting 之前插一个 attachment→shader_read 的 transition。

LX 等价物：完全没有这条路径。`RenderingItem::descriptorResources` 里有 sampled texture，但跟 frame graph 资源依赖无关 —— `FrameGraph` 不知道某张 texture 是不是上个 pass 刚写的。

### 2.3.3 buffer

storage buffer，可以被 compute shader 写、被后续 shader 读，或反之。也需要 memory barrier 跨 pass 同步。

相对 image 简单：只有 access mask 和 stage，没有 layout 概念。

LX 等价物：UBO / vertex / index buffer 走 `IGpuResource`，但都不参与 frame graph 边。它们的同步由 `IGpuResource::setDirty()` 触发的隐式上传完成，不是 pass 间同步。

### 2.3.4 reference — 最微妙的一种

教科书示例：

```json
{
  "name": "transparent_pass",
  "inputs": [
    { "type": "attachment", "name": "lighting" },
    { "type": "attachment", "name": "depth" }
  ],
  "outputs": [
    { "type": "reference", "name": "lighting" }
  ]
}
```

理解关键：transparent_pass 在 GPU 上做什么？

1. 把 `lighting` 作为 input attachment 读入（前一 pass 已画了不透明物体）
2. 继续 *往 lighting 里画* 透明物体（混合到已有像素之上）
3. 读 `depth` 做 depth test 但不写深度

第 2 步是关键：transparent_pass 的 *写动作* 改变了 lighting 的内容，但 *没有产生新资源*。下一个用 lighting 的 pass（比如 post-process）必须排在 transparent_pass 之后，否则会拿到没有透明物体的 lighting 结果。

如果 transparent_pass 的 outputs 不写任何东西，frame graph 无从知道"transparent 也修改了 lighting"，下游 pass 会以为自己直接依赖 forward shading。

为什么不在 outputs 里写一个 `attachment` 类型的 lighting？因为：

- attachment 类型会被 frame graph 算入 framebuffer 创建逻辑 → lighting 会被算两次
- 形式上像在"产生新资源"，语义不对

**reference 类型就是为这种 read-modify-write 场景设计的**：表达"我没产生新资源，但我修改了它，请把任何后续读 lighting 的人排在我之后"。它只用于构边，不参与 framebuffer 构造。

这是 frame graph 里 *容易踩坑、容易讲漏* 的点。

## 2.4 什么时候选数据驱动

不是所有项目都该走数据驱动。判断标准：

| 触发条件 | 推荐方式 |
|---------|---------|
| pass 数 ≤ 5、无美术参与 | 代码驱动 |
| pass 数 > 10、需要美术 / TA 改图 | 数据驱动 |
| 需要 visual editor | 数据驱动 |
| 需要 hot reload 调试不同 pipeline | 数据驱动 |
| 强类型化 pipeline 配置（编译期检查） | 代码 DSL（C++ lambda） |

LX 当前 pass 数极少（基本只 forward + 预留），无美术参与。从这个判断看，*现在* 不需要数据驱动；但路线图（[REQ-119](../../main-roadmap/phase-1-rendering-depth.md#req-119--g-buffer--延迟渲染路径) deferred / [REQ-103](../../main-roadmap/phase-1-rendering-depth.md#req-103--shadow-pass--depth-only-pipeline) shadow / [REQ-107](../../main-roadmap/phase-1-rendering-depth.md#req-107--bloom-pass) bloom 落地后）pass 数会快速突破 10，需要的时点会到。
