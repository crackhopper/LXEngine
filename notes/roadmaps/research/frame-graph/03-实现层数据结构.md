# 03 · 实现层数据结构

> 阅读前提：[02](02-数据驱动方式.md) 已经讲清 JSON 描述形态。本文讨论 parse 完后内存里长什么样。

## 3.1 `FrameGraphResource` — 资源句柄

每个节点的每个 input 和每个 output 都是一个 *独立* 的 `FrameGraphResource` 实例（**关键认知**，下面会展开）。

```cpp
struct FrameGraphResource {
  FrameGraphResourceType type;
  FrameGraphResourceInfo resource_info;
  FrameGraphNodeHandle producer;
  FrameGraphResourceHandle output_handle;
  i32 ref_count = 0;
  const char* name = nullptr;
};
```

### 字段拆解

#### `type`

`image` 还是 `buffer`。一级 dispatch，决定后续 barrier 插入的代码路径：

- `image` → 需要 layout transition + access mask
- `buffer` → 只需要 access mask

#### `resource_info`

按 `type` 解出具体内容（format / size / loadOp 等），通常是 union 或 variant 实现。

#### `producer` — 反向回指"谁产生我"

构边的关键字段。流程：

1. parse 阶段每个 input/output 创建 `FrameGraphResource`，但 `producer` 字段空
2. compile 阶段对每个 input 资源，按 name 查"哪个 output 同名" → 找到 output 所在 node → 写入本 input 的 `producer` 字段
3. 构边：本 node 的 `edges` 里加上所有 input 的 producer node

整个 DAG 都从 `producer` 字段反向推导出来。

#### `output_handle` — input 资源回指对应 output

这是 *"为什么 input 和 output 各自独立 resource 实例"* 的答案。

考虑同一张 GPU image 在两个 pass 里的不同用法：

- pass A 的 output 资源："lighting"，type=attachment（写时是 framebuffer attachment）
- pass B 的 input 资源："lighting"，type=texture（读时是 sampled image）

是同一张 GPU image，但 *用法不同*。两个独立 `FrameGraphResource` 实例描述同一底层 image 的两种角色。`output_handle` 让 input 端能找回原始 output 端。

为什么需要回指：

- compile 算 layout transition 时，知道起点（output 端 layout）和终点（input 端 layout）
- ref_count 应该挂在规范资源（output 端）上，input 只是引用

如果 input 和 output 共享同一个 resource 实例，`type` 字段就没法分别表达"产出时是 attachment、消费时是 texture"。

#### `ref_count` — aliasing 用的引用计数

aliasing 是这样的：让生命周期不重叠的多个逻辑资源共享同一块 GPU memory。

例：

- depth_prepass 写 `depth`
- gbuffer 读 `depth`
- 这之后没人再用 `depth` 了

那么 `depth.ref_count = 2`（被两个 node 引用）。compile 阶段反向扫一遍 graph 算出每个资源的 ref_count；render 阶段每用完一次减 1，归零就标记内存可复用。

#### `name`

JSON 里写的字符串名字。两用：

- 调试 log
- parser 用 name 查 handle（"depth" → 资源池里的某个 index）

## 3.2 `FrameGraphNode` — pass 节点

```cpp
struct FrameGraphNode {
  RenderPassHandle render_pass;
  FramebufferHandle framebuffer;
  FrameGraphRenderPass* graph_render_pass;
  Array<FrameGraphResourceHandle> inputs;
  Array<FrameGraphResourceHandle> outputs;
  Array<FrameGraphNodeHandle> edges;
  const char* name = nullptr;
};
```

### 字段拆解

#### `render_pass` 和 `framebuffer`

**parse 阶段空，compile 阶段自动构造**。

具体：根据本 node 的 attachment 类型 inputs 和 outputs 列表，自动构造 `VkRenderPassCreateInfo`（决定 attachment slots / loadOp / storeOp / subpass）和 `VkFramebufferCreateInfo`（绑实际 image view）。

**这是数据驱动 frame graph 的核心收益之一**：写 JSON 描述 attachment 配置，框架自动建 renderpass / framebuffer。

LX 等价物：暂无。当前 backend 自己手工创建 renderpass / framebuffer。

#### `graph_render_pass` — 用户实现的渲染逻辑

接口指针，指向用户实现的 `FrameGraphRenderPass` 子类。子类里实现 `pre_render / render / on_resize` 等回调，里面写具体 draw call、descriptor 绑定。

frame graph 提供骨架（按拓扑顺序遍历、begin/end renderpass、插入 barrier），用户提供策略（每条 pass 内部具体内容）。**经典策略模式**。

`register_render_pass("gbuffer_pass", &gbuffer_pass)` 就是把策略实例挂到对应 name 的节点的 `graph_render_pass` 字段。

LX 等价物：渲染逻辑散在 `VulkanRenderer` 里，没有"per-pass 实现"的抽象。`FramePass` 只持数据（`queue`），不持行为。

#### `inputs / outputs`

注意类型是 `FrameGraphResourceHandle`（句柄 / index），不是直接的 `FrameGraphResource`。

为什么用 handle：

- 句柄是定长的（一般 32-bit index），数组紧凑
- 资源池统一管理，资源生命周期独立于 node
- 删除某个 node 不会让别的 node 的引用悬空

#### `edges` — 拓扑排序后的依赖链

**parse 阶段空，compile 阶段填**。

填法：扫本 node 所有 inputs，找每个 input 的 `producer` node，加入 `edges`。

注意 *边的方向*：`parent_node.edges` 含"依赖 parent 的 node 列表"，即 *从 producer 指向 consumer*（A 产生资源 → A 的 edges 里有 B → B 消费 A 的资源）。

#### `name`

策略查找的 key（`register_render_pass(name, ...)`）+ 调试。

## 3.3 Builder 模式

```cpp
FrameGraphBuilder builder;
builder.init(&gpu);
FrameGraph graph;
graph.init(&builder);
```

Builder 模式把两件事分开：

| 谁 | 持有什么 |
|----|---------|
| `FrameGraph` | 逻辑结构（节点 + 资源 + 边构成的 DAG） |
| `FrameGraphBuilder` | backend 资源构造（VkRenderPass / VkFramebuffer / VkImage），通过它持有的 `gpu` handle 实施 |

好处：

1. `FrameGraph` 类不直接耦合 GPU API。换 backend（D3D / Metal）只需换 builder
2. builder 同时是名字 → handle 的查询入口（`builder.get_resource("depth")`）
3. 测试时可 mock builder，让 `FrameGraph` 不接触真 GPU 也能测

LX 思路对齐：`IGpuResource` + backend 适配器是同构的设计哲学（core 层薄、backend 层厚）。

## 3.4 四步生命周期

整个 frame graph 走完一次的流程是固定的四步：

### Step 1: parse

```cpp
frame_graph.parse(json_path, allocator);
```

- 读 JSON
- 每个 node 创建 `FrameGraphNode`
- 每个 input/output 创建 `FrameGraphResource`
- `producer / edges / render_pass / framebuffer` 全部空

**完全不接触 GPU**。纯数据结构构造。

### Step 2: compile

```cpp
frame_graph.compile();
```

书里说 "magic happens" 的地方。详见 [04 compile 阶段](04-compile-阶段.md)。要做的事：

1. resolve producer
2. 构边
3. 拓扑排序 + cycle 检测
4. 算 ref_count
5. 建 VkRenderPass
6. 建 VkFramebuffer（按 ref_count 决定 alias）
7. 推 barrier

**只依赖 graph 结构，不依赖 scene 数据**。所以结果可缓存 —— 同一个 graph 只 parse + compile 一次。

### Step 3: register render pass

```cpp
builder->register_render_pass("gbuffer_pass", &gbuffer_pass);
builder->register_render_pass("lighting_pass", &light_pass);
```

把用户实现的策略类挂到对应 name 的 node 的 `graph_render_pass` 字段。

**为什么 register 在 compile 之后才发生**：因为 compile 已经把节点骨架（renderpass / framebuffer）建好了，register 只是给已有骨架挂策略实例。这意味着不需要重新 compile 就能换实现 —— 运行期可以 swap 策略做 A/B 测试。

### Step 4: render

```cpp
frame_graph.render(gpu_commands, scene);
```

按拓扑排序顺序遍历所有 nodes：

```pseudo
for node in topological_order:
    insert barriers (compile 算好的)
    begin renderpass(node.render_pass, node.framebuffer)
    call node.graph_render_pass->render(commands, scene)
    end renderpass
```

**这是每帧调用的部分**。compile 的所有结果重用，只产生运行期开销。
