## 5.2 字段级对照

### `FramePass` vs 教科书 `FrameGraphNode`

| 字段 | LX `FramePass` | 教科书 `FrameGraphNode` | 差距 |
|------|----------------|-------------------------|------|
| 身份 | `name: StringID` | `name: const char*` | 等价（实现选择不同） |
| 输出形状 | `target: RenderTarget` | 通过 outputs 列表表达 | LX 只能表达单一 attachment 形状 |
| pass 内容 | `queue: RenderQueue`（已经填好的 RenderingItem 列表） | `graph_render_pass`（策略指针，运行期发 draw call） | 完全不同抽象 |
| 资源依赖 | **无** | `inputs / outputs` 列表 | 完全缺失 |
| 边 | **无** | `edges` 列表 | 完全缺失 |
| backend 资源 | **无**（散在 backend 里） | `render_pass / framebuffer` | 完全缺失 |

### `FrameGraph` vs 教科书 `FrameGraph`

| 能力 | LX | 教科书 | 差距 |
|------|------|---------|------|
| 构造方式 | `addPass` 代码 | JSON parse | 数据驱动缺失 |
| 编译期处理 | 无 compile 阶段 | compile 算 edges / barriers / aliasing | 全缺失 |
| 资源依赖 | 不感知 | 通过 inputs/outputs 推导 | 缺失 |
| barrier 自动插入 | 不做（散在 backend） | 自动 | 缺失 |
| aliasing | 不做 | 算 ref_count + alias | 缺失 |
| 拓扑排序 | 按 addPass 顺序硬走 | 按 DAG 拓扑序 | 缺失 |
| dead pass elimination | 不做 | 可选 | 缺失 |
| 跨 pass PipelineKey 去重 | ✅ `collectAllPipelineBuildDescs` | 默认无（去重在别处） | LX 多一层 |

### 资源类型对照

| 教科书类型 | LX 现状 |
|-----------|---------|
| attachment | `RenderTarget.colorFormat / depthFormat` 描述 *output*；无 *input attachment* 概念 |
| texture | `RenderingItem.descriptorResources` 里有 sampled texture，但跟 frame graph 边无关 |
| buffer | UBO / vertex / index 走 `IGpuResource`，跟 frame graph 边无关 |
| reference | 完全无对应概念 |

## 5.3 LX 当前位置

用书里术语描述：**LX 当前 `FramePass + FrameGraph` 大致对应 *parse 完但还没 compile* 的最早期状态**。

- 节点已经有了
- 节点的 name 有了
- 节点的 attachment 形状有一点点（`RenderTarget`，但还很薄；REQ-042 在补）
- 节点的 inputs / outputs / edges：完全没有
- compile 阶段：完全没有
- aliasing：完全没有
- 自动 barrier：完全没有
- 数据驱动：还是代码驱动

距离教科书形态有 6 个 REQ 的距离（详见 [06 演进路径](06-演进路径.md)）。

## 5.4 REQ-009 target 轴的占位现状

[REQ-009](../../../requirements/finished/009-multi-camera-multi-light.md) 引入了 `Scene::getSceneLevelResources(pass, target)` 这一 *两轴筛选*：

- pass 轴：决定哪些 light 参与（`light->supportsPass(pass)`）
- target 轴：决定哪些 camera 参与（`camera->matchesTarget(target)`）

目前 *target 轴几乎是占位 hook* —— 全工程实际只用一种默认构造的 `RenderTarget`，所有 pass 与 seed Camera 默认值相同，`matchesTarget` 永远返回 true，没有真实筛选。

这不是设计成果，是因为 `RenderTarget` 还没长出足够字段（MRT、layer、自定义 extent、load/store ops 都缺）来产生真实差异。REQ-042 落地后 `RenderTargetDesc` 会长出真字段，target 轴才开始有真实筛选行为。

## 5.5 LX 已经做对的（不需要重新设计的）

| 能力 | LX 实现 | 在 frame graph 路线里的作用 |
|------|---------|---------------------------|
| StringID + GlobalStringTable interning | `src/core/utils/string_table.*` | 资源名 / pass 名的稳定身份；比教科书 `const char*` 更稳定 |
| `IGpuResource` 统一契约 | `src/core/rhi/gpu_resource.hpp` | 未来 `FrameGraphResource.resource_info.texture` 直接挂 `IGpuResourceSharedPtr`，attachment / texture / buffer 全统一 |
| `PipelineKey` 三级 compose（REQ-042 后） | `src/core/pipeline/pipeline_key.hpp` | 跨 pass pipeline 兼容性的 cache key |
| `RenderTargetDesc`（REQ-042 后） | 待落地 | attachment 形状的标准描述，对应教科书 `FrameGraphResourceInfo` 中 image 部分 |
| 单 pass 的 `RenderQueue` 内部排序去重 | `src/core/frame_graph/render_queue.*` | 未来作为 node 的"内置默认策略"，不需要每个 node 都从零写 |
| `Scene` 扁平容器 | `src/core/scene/scene.*` | scene-level 资源（camera / light）的来源，被每个 node 的 strategy 调用 |

未来 frame graph 不需要推翻这些，只需要在它们之上加 inputs/outputs/edges/compile/barrier 这套机制。
