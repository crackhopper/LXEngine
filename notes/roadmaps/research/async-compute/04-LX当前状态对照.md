# 04 · LX 当前状态对照

> 阅读前提：[01](01-async-compute-是什么.md) ~ [03](03-compute-shader基础.md) 已经讲清完整 async compute 形态。本文逐项对照 LX 当前实现，盘点差距。

## 4.1 LX 当前的 compute 能力总盘点

**几乎全空白**。逐项盘点：

### Compute queue 与 queue family

LX 的 `VulkanRenderer` 只初始化 graphics queue：

- ❌ 没有 compute queue handle
- ❌ 没有 compute queue family index 查询
- ❌ 没有独立 compute family 的优先选择策略
- ❌ 没有 transfer queue 单独使用
- ❌ 没有硬件 fallback 检测逻辑

### Compute pipeline

`PipelineBuildDesc` 描述的是 graphics pipeline（vertex layout / render state / topology）：

- ❌ 没有 `ComputePipelineDesc` 抽象
- ❌ 没有 `vkCreateComputePipelines` 路径
- ❌ Shader stage 列表默认假定 vertex + fragment

### Compute shader 加载

需要查代码确认，但大概率：

- ❌ Shader loader 默认 `*.vert.spv` / `*.frag.spv`，不识别 `*.comp.spv`
- ❌ `ShaderStage` 枚举可能没有 `Compute`（待查）
- ❌ Reflection 不提取 local group size（compute 编译期常量）
- ❌ Reflection 不识别 storage block 与 storage image

### Dispatch 命令

`RenderingItem` 是 draw call 中心：

- ❌ 没有 `DispatchItem` 抽象
- ❌ 没有 `vkCmdDispatch` 命令封装
- ❌ Command buffer recorder 只支持 `bind_pipeline / draw*`

### Storage buffer 角色

`IGpuResource` 已有 `ResourceType::ParameterBuffer` / `VertexBuffer` 等，但：

- ❌ Storage buffer（compute writeable SSBO）未单独建模
- ❌ Storage image 未单独建模
- ❌ Buffer 创建时未声明 `STORAGE_BUFFER` usage flag

### Frame Graph 的 queue 概念

`FramePass` 没有"画到哪条 queue"字段：

- ❌ 默认所有 pass 走 graphics queue
- ❌ 没有 queue type 字段
- ❌ FrameGraph 无法表达跨 queue 依赖

### Timeline

multi-threading/08 已经设计了 `IGpuTimeline` 接口，预留了 `computeTimeline` 字段：

- ✅ **接口设计完成**
- ❌ Backend 实现未接入
- ❌ `computeTimeline` 没有对应 `VkSemaphore`

这是 LX 在 async compute 上 *最成熟* 的部分，但也仅停留在接口层。

### Queue family ownership transfer

完全没碰过：

- ❌ 资源创建时 sharing mode 未显式声明（默认 `EXCLUSIVE` 隐含）
- ❌ 没有 ownership transfer barrier 工具
- ❌ 没有跨 queue 资源生命周期跟踪

## 4.2 字段级 / 能力级 gap 表

| 维度 | LX 现状 | 教科书 / async compute 需要 | gap 大小 |
|------|---------|---------------------------|---------|
| Device queue | 仅 graphics queue handle | + compute queue handle、+ transfer queue handle | 大 |
| Queue family 选择策略 | 只查 graphics-capable | + 查独立 compute family、+ fallback 到 graphics-capable | 中 |
| Pipeline 类型 | 只有 graphics pipeline | + compute pipeline | 大 |
| Shader stage 枚举 | Vertex / Fragment / ... | + Compute（可能已有，需查） | 小 |
| Shader loader | `*.vert.spv` / `*.frag.spv` | + `*.comp.spv` | 小 |
| Shader reflection | vertex input / uniform / texture | + local group size / storage block / storage image | 中 |
| 命令录制 | `bind_pipeline / draw*` | + `bind_pipeline(compute) / dispatch` | 中 |
| RenderingItem / RenderQueue | draw 中心 | 需要"DispatchItem"抽象 或 同一 item 类型加 dispatch 字段 | 中 |
| FramePass 队列亲和性 | 默认 graphics | + queue type 字段 | 中 |
| 跨 queue barrier | 不做 | 必须做（ownership transfer） | 大 |
| Storage buffer | 没单独建模 | 单独 ResourceType + descriptor type | 中 |
| Async 触发框架 | 无 | submit 双 queue + timeline 协调（multi-threading/08 已设计） | 已设计未实现 |
| Timeline backend 接入 | `IGpuTimeline` 接口存在但 backend 未实现 | `computeTimeline` 对应的 `VkSemaphore` 创建 + signal/wait 实现 | 中 |
| 硬件 fallback | 无 | 必须（移动 / 集显 / 老硬件） | 中 |

整体距离：**比 frame graph 演进路径还更早期**。frame graph 至少有 `FrameGraph / FramePass / RenderQueue` 骨架；async compute 连骨架都没有。

## 4.3 LX 已经做对的、能直接对接的部分

| LX 已有 | 在 async compute 路线里的作用 |
|--------|----------------------------|
| `IGpuTimeline` 接口设计（multi-threading/08） | 直接复用 `computeTimeline` 字段，graphics ↔ compute 同步天然就绪 |
| 资源 RetirePoint 模型（multi-threading/08） | compute 写的资源退休点挂 `computeTimeline@T`，graphics 读完可回收 |
| `IGpuResource` 统一契约 | compute 用的 storage buffer 可以走同一接口，只是新增一个 ResourceType |
| FrameGraph 候选 REQ-035..041 | compute pass 自然作为 graph node 的一种，只是有"队列亲和性"标签 |
| StringID + interning | compute pipeline 的 pipelineKey 可以走同一套 compose，只是 sig 内容不同 |
| Shader 反射框架（基础） | 扩展支持 compute stage，不是重写 |

## 4.4 几个特殊耦合点

### async compute ↔ frame graph
**强耦合**。理由：

1. compute pass 必须能进 frame graph DAG（否则没法表达跨 queue 依赖）
2. frame graph 的 barrier 推导（候选 REQ-038）必须知道 source / dest queue family（否则推不出 ownership transfer barrier）
3. frame graph 的拓扑排序需要扩展 — 同 queue 内偏序已经有，跨 queue 的"同步点"是新维度

**结论**：async compute 的 frame graph 集成 *不应该* 在 frame graph 演进完成前推进太深。详细分析见 [05 与 frame graph 的耦合](05-与frame-graph的耦合.md)。

### async compute ↔ multi-threading
**已经预留**。multi-threading/08 §8.4 就预留了 `computeTimeline`。async compute 落地时不需要重新设计 timeline 模型，直接 *消费* 这套接口。

### async compute ↔ shader system
**中等耦合**。compute shader 的反射结果跟 graphics 不同：

- 没有 vertex input layout
- 通常没有 push constant（或有但用法不同）
- 主要 binding 是 storage buffer + storage image
- 有 local group size 这个编译期常量

需要扩展 `ShaderReflector`，但不需要重写。

### async compute ↔ material system
**低耦合**。compute 一般 *不走材质系统* — 它是"系统级 GPU 计算"，跟"材质 = 一组 shader + 参数"概念正交。直接走 `Pipeline + DescriptorSet` 裸接口即可。

但有例外：如果未来引入 "compute-based material variant"（比如 ray-traced material），那就要把 material 抽象扩展到 compute path。短期不考虑。

### async compute ↔ REQ-034 RenderTarget
**无关**。REQ-034 拆 RenderTarget 的工作不影响 async compute 路径。compute 不写 framebuffer attachment，写的是 storage buffer / storage image。

## 4.5 LX 当前位置（用层级描述）

参照 [01 §1.5](01-async-compute-是什么.md#15-lx-当前的位置) 给的能力层级：

```
Level 0: 没 compute shader 路径               ← LX 在这里
Level 1: 有 compute shader，单 queue 串行
Level 2: compute 提到独立 queue，但手工 barrier
Level 3: frame graph 自动 barrier + ownership transfer
Level 4: 自动 workload 调度（哪些 pass 适合 async 自动决定）
```

距离 Level 1 ≈ REQ-A（基础设施）的工作量
距离 Level 2 ≈ + REQ-B + REQ-C 的工作量
距离 Level 3 ≈ + REQ-D（依赖 frame-graph REQ-038）的工作量
Level 4 不在本调研范围。
