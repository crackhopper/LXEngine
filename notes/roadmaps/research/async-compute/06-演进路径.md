# 06 · 演进路径

> 阅读前提：[04](04-LX当前状态对照.md) + [05](05-与frame-graph的耦合.md) 已经盘清 LX 现状与时序约束。本文给出推进路径切分。
>
> 注：本文给出的 REQ-A..F 编号是 *候选编号*，正式立项时按 `notes/requirements/` 实际可用号顺延。

## 6.1 总体策略

**结论先行：6 个可独立合 PR 的 REQ，分两大族**：

- **基础设施族（A / B / F）**：纯 backend 能力，不依赖 frame graph
- **集成族（C / D / E）**：与 frame graph 演进协同，必须按 frame graph 节奏推进

每个 REQ：

- 有清晰的输入输出
- 完成后工程仍可编译可测
- 不依赖后续 REQ 即可单独评估
- 风险点（跨 queue barrier 推导）单独立 REQ，便于聚焦验证

**前置条件**：multi-threading/08 设计的 `IGpuTimeline` 接口要在 backend 落地（这本身可作为 multi-threading 调研下的独立 REQ，不属于本调研但是硬前置）。

## 6.2 六步演进序列

### REQ-A · Compute queue + compute pipeline 基础设施

**目标**：让 LX 能跑起最简单的 compute shader，提交到独立 compute queue（或 fallback 到 graphics queue），不参与 frame graph。

**改动**：

- `VulkanDevice` 查 compute queue family（区分独立 vs 共享 graphics family）+ 创建 compute queue handle
- 引入 `ComputePipeline` / `ComputePipelineDesc`
- 扩展 `ShaderReflector` 支持 compute stage（local group size 提取、storage block 识别）
- Shader loader 支持 `*.comp.spv`
- 加 `dispatch` 命令封装
- 扩展 `IGpuResource` 加 `StorageBuffer` 角色（或确认已有可复用）
- 扩展 timeline backend 实现 `computeTimeline`（multi-threading/08 接口的具体实现）

**验收**：脱离 frame graph，写一个独立 demo（比如 GPU 上计算 mandelbrot 写入 texture），跑通 compute submit + graphics 显示。

**风险**：低 - 中。基础设施铺设。

**可独立合 PR**：是。

### REQ-B · Queue family ownership transfer 基础

**目标**：让 buffer/image 能在 graphics queue 和 compute queue 之间正确流转，barrier 暂时手工。

**改动**：

- 资源创建时显式声明 `VK_SHARING_MODE_EXCLUSIVE`
- backend 添加 `transferOwnership(resource, srcQueue, dstQueue)` 工具函数
- 加测试：cross-queue resource access 不触发 validation error
- 文档：手工 ownership transfer 的使用模式

**前置依赖**：REQ-A

**验收**：compute 写一个 buffer，graphics 读它，barrier + ownership transfer 正确，validation 全绿。

**风险**：中。barrier 细节多，但单点验证。

**可独立合 PR**：是。

### REQ-C · Async-aware FramePass（队列亲和性字段）

**目标**：让 frame graph 知道每条 pass 走哪条 queue。

**改动**：

- `FramePass` 加 `queue: QueueKind` 字段（默认 Graphics，可选 Compute / Transfer）
- `FrameGraph::buildFromScene` / `compile`（取决于 frame-graph 演进进度）按 queue 分流 submit
- 暂不做跨 queue 自动 barrier（手工维护，沿用 REQ-B 工具）
- `FrameGraph::compile` 拓扑排序考虑 queue 归属

**前置依赖**：REQ-A、REQ-B、frame-graph REQ-035（FramePass 数据结构扩展）

**验收**：写一个布料模拟样例，compute pass 走 compute queue，graphics pass 走 graphics queue，手工 barrier 协调，frame graph 正确分流提交。

**风险**：中。对 FrameGraph 的侵入式扩展。

**可独立合 PR**：是，但与 frame-graph REQ-035 协同最经济。

### REQ-D · 跨 queue 自动 barrier 推导（**风险点 #1**）

**目标**：把 REQ-B 的手工 barrier 升级为 frame graph compile 阶段自动推导。

**改动**：

- 扩展 frame graph compile 阶段的 barrier 推导逻辑，处理跨 queue 资源
- 自动插入 ownership transfer release barrier + acquire barrier
- 自动插入 timeline signal/wait
- 资源 lifetime 跟踪扩展到多 queue

**前置依赖**：**frame-graph 候选 REQ-038（barrier 自动推导）必须先完成**

**验收**：

- 去掉 REQ-C 的所有手工 barrier，frame graph 自动推导
- Vulkan validation layer 全绿
- 手动构造跨 queue 资源场景（compute → graphics、graphics → compute、双向），barrier 正确

**风险**：**高**。多 queue 的 barrier 推导比单 queue 复杂一个数量级。

**配套要求**：必须配 Vulkan validation layer 强测试 + 跨硬件验证（NVIDIA + AMD + Intel）。

### REQ-E · 实战 demo - GPU culling 或后期 compute pass

**目标**：选一个真实工业场景验证整套机制。

**候选**（按推荐度排序）：

- **GPU frustum culling**（compute 算可见性 + indirect draw）— 收益清晰、可见
- **Compute bloom**（compute mip downsample / upsample）— 工程简单
- **GPU 粒子系统**（compute 模拟 + draw indirect）— 跟 Phase 5 物理对接

**前置依赖**：REQ-A 到 REQ-D（理想），或 REQ-A 到 REQ-C（可接受）

**验收**：

- 性能对比 vs CPU 实现 / vs graphics queue compute
- GPU 利用率提升（用 NSight / Radeon GPU Profiler 验证）
- 画面正确（与非 async 路径像素一致）

**风险**：低。是已有机制的应用。

### REQ-F · 硬件 fallback（无独立 compute queue 时降级）

**目标**：对低端硬件 / 集成显卡 / 移动 GPU 的兼容性。

**改动**：

- 启动期检测：是否有 *独立 compute queue family*
- 如果没有，async compute 降级为 *graphics queue 上 compute submit*（仍能跑，但失去并行收益）
- 通过 `IGpuTimeline` 抽象，调用方不感知（仍然用 `computeTimeline`，只是底层与 `graphicsTimeline` 是同一个 `VkSemaphore`）
- 启动 log 标注 fallback 状态

**前置依赖**：REQ-A

**验收**：在只有 1 个 graphics queue 的设备上 demo 仍能跑，画面正确，性能跟纯 graphics 路径相近（无显著回归）。

**风险**：低。

**可独立合 PR**：是。

## 6.3 依赖图

```
multi-threading/08（已设计）
   ↓ 提供 IGpuTimeline + RetirePoint
   ↓ + computeTimeline 字段预留
   ↓
REQ-A (compute queue + pipeline)
   ↓
REQ-B (queue ownership transfer 工具) ── 平行 ──→ REQ-F (fallback)
   ↓
   ↓ 等 frame-graph REQ-035 (FramePass 扩展)
   ↓
REQ-C (FramePass 队列亲和性)
   ↓
   ↓ 等 frame-graph REQ-038（barrier 推导）
   ↓
REQ-D (跨 queue 自动 barrier) ← 风险点 #1
   ↓
REQ-E (实战 demo)
```

REQ-A / REQ-F 可以最早做。REQ-B 紧随。REQ-C 等 frame-graph REQ-035。REQ-D 等 frame-graph REQ-038。

## 6.4 何时触发 REQ-A 立项

按 [`README §"何时触发本调研立项"`](README.md#何时触发本调研立项) 的判断，触发条件：

- **强触发**：[Phase 5 物理](../../main-roadmap/phase-5-physics.md) 决定走 GPU 实现（如 GPU 布料 / 粒子 / 流体）
- **强触发**：性能瓶颈分析显示 GPU 利用率有大量空闲窗口 + 有 compute-suitable workload 可以并入
- **中触发**：引入 GPU culling
- **中触发**：引入 compute-based post-process（compute bloom、tonemap 等）
- **弱触发**：纯学习 / 探索目的

任一条强触发条件命中，REQ-A 应紧随其后立项。当前路线图（[Phase 1](../../main-roadmap/phase-1-rendering-depth.md)）的 PBR / shadow / IBL / bloom 都是 graphics-only 工作，**没有强触发条件**。

## 6.5 中间里程碑

各 REQ 完成后的可交付：

| 阶段 | 完成 REQ | 可交付 |
|------|---------|--------|
| M-AC.α | REQ-A | LX 能跑 compute shader（脱离 frame graph，单点 demo） |
| M-AC.β | + REQ-B | LX 能手工跨 queue 同步，单元测试通过 |
| M-AC.γ | + REQ-F | LX 在低端硬件 / 集显上 fallback 正常 |
| M-AC.δ | + REQ-C | compute pass 进入 frame graph 调度，分流 submit；手工 barrier |
| M-AC.ε | + REQ-D | 跨 queue barrier 自动推导，手工代码归零 |
| M-AC.ζ | + REQ-E | 真实 GPU workload 落地（GPU culling / compute bloom / 粒子） |

每个里程碑都是可独立交付、可独立验证的状态。

## 6.6 工业级延伸（不在本路线范围）

完成 REQ-E 后离工业级 async compute 还有以下方向，建议不进 v1 路线：

- **多 compute queue 调度**（同 family 多 queue 实例并行）
- **Async compute 的自动 workload 调度**（自动决定哪些 pass 适合 async）
- **Subgroup 操作优化**（warp-level 原语：ballot / broadcast / reduce）
- **VK_KHR_synchronization2 完全替换 sync 1.0 API**
- **GPU-driven rendering**（compute 完全控制 draw indirect）
- **Async compute 与 raytracing 的协同**

这些都是 *优化层 / 高级特性*，等 v1 跑稳定且有真实瓶颈再考虑。

## 6.7 风险与不确定性

### 风险 1：硬件碎片化

不同 GPU 厂商对 async compute 的硬件支持差异大：

- NVIDIA：从 Pascal 开始 async 支持好
- AMD：GCN 架构原生 async 优势大
- Intel：Arc 才有真正 async；老集显基本无收益
- 移动 GPU：差异更大，多数没有独立 compute queue

需要 REQ-F 的 fallback 设计才能保证可移植性。

### 风险 2：性能不一定提升

Async compute 不是免费的。错误的工作分配可能反而 *降低* 性能：

- compute 工作太短：submit overhead > 收益
- 资源竞争：跨 queue 用同 cache，cache thrashing
- ownership transfer barrier 开销

需要配套 GPU profiling 工具（NSight / Radeon GPU Profiler）才能验证收益。

### 风险 3：调试难度

跨 queue 同步 bug 的症状是 *偶发的画面错误* 或 *罕见 hang*，跟 aliasing bug 类似 — 极难复现。

配套要求：

- 强 validation layer 测试
- 跨硬件验证（至少 NVIDIA + AMD + Intel）
- timeline 断言机制（multi-threading/08 已经有 RetirePoint 检查的基础）

## 6.8 一句话总结

**Async compute = 让 GPU 多硬件引擎并行跑。LX 当前 Level 0（无 compute）；推到 Level 3（自动 barrier）需要 6 个 REQ + frame-graph 演进配合。基础设施 REQ-A/B/F 可早做，集成 REQ-C/D 必须等 frame-graph 协同。Phase 5 物理走 GPU 是真正的强触发。**
