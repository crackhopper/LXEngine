# 08 · 演进路径

> 阅读前提：[07 LX 当前状态对照](07-LX当前状态对照.md) 已经盘清 LX 现状与跟前面 7 个调研的耦合。本文给出推进路径切分。
>
> 注：本文给出的 REQ-A..H 编号是 *候选编号*，正式立项时按 `notes/requirements/` 实际可用号顺延。

## 8.1 总体策略

**结论先行：8 个候选 REQ，按 *基础设施 → 单点验证 → 完整特性 → 通用 denoiser* 四层切分**：

- **基础设施层（REQ-A、B）**：AS + ray query 路径，desktop-only，验证 Vulkan extension 能跑通
- **单点验证层（REQ-C）**：朴素 RT shadow，给现有 lighting shader 加 4 行
- **完整特性层（REQ-D、E、F）**：RT pipeline + SBT，DDGI + reflection + SVGF
- **优化层（REQ-G、H）**：advanced RT shadow + 双路径 fallback

每个 REQ：

- 有清晰的输入输出
- 完成后工程仍可编译可测
- *可独立合 PR*，避免大爆炸式提交
- *desktop-only*，不影响 Web / 跨平台路径（Web fallback 在 REQ-H 收口）

**前置条件**：

- frame-graph 候选 REQ-035 + REQ-038（FramePass + barrier 推导）必须先就位
- multi-threading/08 `IGpuTimeline` + `RetirePoint` 必须落地
- temporal-techniques REQ-A（motion vector + history）必须先做
- gpu-driven-rendering 候选 REQ-F（cluster lighting）+ bindless texture 必须先做
- shadows 候选 REQ-A..C（cubemap shadow）必须先做（fallback 路径）

**关键架构决策**：本调研的所有特性 *desktop-only*，Web / 移动端走 raster fallback。这必须在 REQ-A 就明确设计 *双路径开关*——不能让 RT 代码渗透到跨平台共享层。

## 8.2 八步演进序列

### REQ-A · RT 基础设施（AS + 扩展启用）

**目标**：让 LX 能创建 BLAS / TLAS，但暂不调 ray。

**改动**：

- 启用 4 个 Vulkan extension：
  - `VK_KHR_acceleration_structure`
  - `VK_KHR_ray_query`
  - `VK_KHR_buffer_device_address`
  - `VK_KHR_deferred_host_operations`
- `VulkanDevice` 加 RT feature 检测（`VkPhysicalDeviceRayTracingPipelinePropertiesKHR` 等）
- `IGpuBuffer::get_device_address()` 接口
- 新增 `IBlas` / `ITlas` 资源类型（继承 `IGpuResource`）
- AS scratch buffer pool（构建时复用临时 memory）
- BLAS 构建工具：从 mesh vertex/index buffer → BLAS
- TLAS 构建工具：从 instance list → TLAS（每帧 rebuild）
- AS compaction 流程
- BLAS lifetime 跟 mesh asset 联动（mesh load → BLAS build；mesh unload → BLAS free）
- TLAS 双 buffer + RetirePoint 跨帧调度
- *双路径开关*：`LX_ENABLE_RAY_TRACING` 编译时常量 + 运行时 feature 检测；非 RT 平台所有 RT 代码路径完全编译外掉

**前置依赖**：

- frame-graph REQ-035（FramePass）— 跨帧资源表达
- multi-threading/08 `IGpuTimeline` + `RetirePoint`

**验收**：

- 写一个 demo：场景内多个 mesh 各建 BLAS + TLAS
- 启动期 log 显示 BLAS / TLAS 构建时间 + 内存占用
- AS compaction 后内存减少 ~50%
- TLAS 跨帧 retire 通过 validation
- 在不支持 RT 的设备上 fallback 流程不 crash

**风险**：中。基础设施面广，但每块都是孤立机制。

**可独立合 PR**：是。

### REQ-B · Ray Query 路径

**目标**：让 fragment / compute shader 能调 ray query，但还不接 lighting。

**改动**：

- ShaderReflector 兼容 `GL_EXT_ray_query` 扩展
- shader system 让 fragment shader 能 declare `accelerationStructureEXT` uniform
- frame graph 加 *AS read* 资源依赖类型
- 写一个独立 demo shader：fragment shader cast 1 ray，可视化命中 / miss

**前置依赖**：REQ-A

**验收**：

- demo：渲染一个三角形，fragment shader 调 ray query 测自己跟 TLAS 是否相交（应该相交）
- Validation layer 全绿
- 性能数字：单 ray query 在 RTX 30 上 < 1 μs

**风险**：低。ray query 是简单单点扩展。

**可独立合 PR**：是。

### REQ-C · 朴素 RT Shadow（killer first use case）

**目标**：用 ray query 实现 [Ch13 §4.2](04-RT-shadow.md#42-朴素版4-行代码) 的朴素 shadow——给现有 lighting shader 加 4 行。

**改动**：

- Lighting shader 加 ray query shadow 函数
- `Camera::cullMask` 跟 ray query cullMask 整合
- frame graph 把 lighting pass 标记为 *AS read*
- 跟 cubemap shadow path *并存*（feature flag 切换）
- demo：单光源场景对比 cubemap shadow vs RT shadow（RT 应该更精细 / 无 acne）

**前置依赖**：REQ-A + REQ-B + shadows REQ-A..C（cubemap shadow 共存路径）

**验收**：

- 单光源 hard shadow，无 self-intersection
- 多光源（5-10 个）性能 acceptable（< 1 ms / fragment lighting pass overhead）
- 切换 RT / cubemap shadow，画面无突变（geometry-correct shadow vs filter-blur shadow 的预期差异）

**风险**：低。Ch13 朴素版只有 4 行 GLSL。

**可独立合 PR**：是，可以在 RT pipeline 之前先获得 visual baseline。

### REQ-D · RT Pipeline + SBT + Hit Shader 框架

**目标**：让 LX 能创建 RT pipeline，跑通 raygen + closest-hit + miss 三件套。

**改动**：

- 新增 `RTPipeline` / `RTPipelineDesc` 类型（跟 GraphicsPipeline / ComputePipeline 并列）
- ShaderReflector 支持 6 个新 shader stage（raygen / miss / closest-hit / any-hit / intersection / callable）
- `VkRayTracingShaderGroupCreateInfoKHR` wrapper
- SBT buffer 资源类型 + handle 拷贝工具
- `vkCmdTraceRaysKHR` command buffer wrapper
- Hit shader 工具：通过 buffer device address 重建 vertex/normal/uv
- Mesh asset 暴露 `device_address` 字段（vertex / index / uv / normal buffer）
- Bindless texture 在 RT shader 里能用（`nonuniformEXT` 限定符）
- frame graph 加 *RayTracing* pass kind
- demo：raygen 投单 ray，closest-hit 返回红色，miss 返回蓝色

**前置依赖**：

- REQ-A
- frame-graph REQ-038（barrier 推导处理 RT access mask）
- bindless-texture（独立调研，必须就位）

**验收**：

- demo 红蓝二色图正确
- RT pipeline cache 工作（第二次启动加载 < 100 ms）
- Hit shader 能采 bindless 贴图

**风险**：**高**。这是 RT 引入的最大复杂度——pipeline cache + shader system + frame graph 都要扩展。

**可独立合 PR**：建议拆 3 个 sub-PR：
- D.1：RT pipeline 类型 + shader stage 枚举
- D.2：SBT + handle 拷贝 + dispatch
- D.3：Hit shader bindless + buffer device address

### REQ-E · DDGI（动态 GI）

**目标**：实现 [Ch14](05-RT-DDGI.md) 的完整 DDGI 5 步算法。

**改动**：

- 新增 3D probe grid 资源（octahedral 编码 patch + visibility）
- Probe ray output 2D texture (`rays_per_probe × num_probes`)
- 5 个 RT/compute pass：
  1. Probe ray trace（RT pipeline）
  2. Irradiance update + border 拷贝（compute）
  3. Visibility update（compute）
  4. Probe offset（compute，仅启动期）
  5. Indirect lighting sample（compute，quarter resolution）
- Octahedral 编码 / spherical Fibonacci / hysteresis blend shader util
- Hit shader 调 cluster lighting 算 direct lighting
- Miss shader 返回 sky color / env cubemap
- 应用到现有 lighting shader 末端

**前置依赖**：

- REQ-D
- gpu-driven-rendering REQ-F（cluster lighting，hit shader 必需）
- bindless-texture（hit shader 采贴图）

**验收**：

- 室内场景：红色地毯把红光反射到墙壁
- 动态光源移动：probe 几十帧内追上变化
- 性能：1.5-2 ms（参考 NVIDIA DDGI 论文）
- 跟 cubemap shadow + 现有 lighting 协同

**风险**：高。第一个真实 RT pipeline 应用，corner case 多（probe 卡墙 / 漏光 / hysteresis 时序）。

**可独立合 PR**：是。

### REQ-F · RT Reflection + SVGF 通用 Denoiser

**目标**：实现 [Ch15](06-RT-reflection与SVGF.md) 的 RT reflection + 把 SVGF 实现成 *可复用通用 denoiser*。

**改动**：

- G-buffer 加两个通道：
  - `mesh_id`（u32 per-pixel，跨帧稳定）
  - `depth_normal_dd`（screen-space derivatives）
- Stable global mesh ID 系统（mesh asset 扩展）
- GGX VNDF / PCG random / importance sampling lights shader util
- Reflection RT pipeline（raygen + closest-hit + miss）
- **SVGF 3-pass 通用 compute chain**：
  - Pass 1: temporal accumulation + 4 道 consistency check
  - Pass 2: variance estimate
  - Pass 3: 5× A-trous wavelet（normal + depth + luminance 三重权重）
- SVGF 接口设计：参数化输入 / 输出 / 权重 σ，多个特性可复用
- Reflection 应用到 lighting shader（specular term）
- 跟 SSR *并存*（feature flag）

**前置依赖**：REQ-D + REQ-E（RT pipeline 完整 + 验证过 1 遍）

**验收**：

- 镜面 / glossy 表面正确反射屏幕外几何
- 动态场景反射跟随物体移动
- SVGF 调参后画面稳定，无明显噪声 / ghost
- SVGF 通用接口验证：用同一 SVGF 实例处理 reflection + 一个简单 RT AO 测试

**风险**：**高**。SVGF 调参 + mesh_id 跨帧稳定 + importance sampling 跟 cluster 整合都是难点。

**可独立合 PR**：建议拆 3 个：
- F.1：mesh_id + depth_dd G-buffer 通道 + GGX VNDF + 1-ray reflection
- F.2：SVGF 3-pass 实现 + reflection 应用
- F.3：SVGF 通用化（参数化 + RT AO 验证）

### REQ-G · Advanced RT Shadow（RT Gems 4-pass）

**目标**：把 REQ-C 的朴素 RT shadow 升级到 [Ch13 §4.3](04-RT-shadow.md#43-rt-gems-advanced-版4-pass) 的 advanced 版——adaptive sample count + Poisson disk + 时空滤波。

**改动**：

- 3D texture (W, H, lights) for visibility / variance / sample_count cache（4 帧 ring）
- Motion vector 特化版 compute pass
- Visibility variance compute pass
- Adaptive ray cast + history update compute pass
- Spatio-temporal filter compute pass（normal-aware skip）
- Poisson disk 资产
- Cluster lighting 整合（per-fragment 只查 cluster 命中的 lights）

**前置依赖**：REQ-C + REQ-F（SVGF 框架可作 reference / 共享）+ gpu-driven REQ-F（cluster lighting）

**验收**：

- Soft shadow + area light 支持
- 多光源（50+）不崩
- 跟 REQ-C 朴素版对比，画面更柔 + 性能更好（per-fragment 平均 ray 数 < 3）

**风险**：中。Ch13 advanced 版规则清晰，但 4-pass 调参陷阱多。

**可独立合 PR**：是。

### REQ-H · 双路径 Fallback + 跨平台收口

**目标**：把所有 RT 特性的 *non-RT fallback* 路径统一收口，让 LX 在 Web / 移动端能 *运行同一场景*（画质降级但不 crash）。

**改动**：

- `RenderConfig` 暴露 *RT 特性开关*：shadow / GI / reflection 各自 RT/raster
- shader 条件编译（per-platform shader variant）
- 启动期 capability detection → 自动选 RT / fallback
- Lighting shader 双路径：`#ifdef LX_RT_SHADOW` / `#ifdef LX_RT_GI` / `#ifdef LX_RT_REFLECTION`
- WebGPU 路径（REQ-112）继承 fallback 默认（全 raster）
- 文档：每个 RT 特性的 fallback 对应表

**前置依赖**：REQ-A..G 全部 + frame-graph 主路径

**验收**：

- 同一场景在 RTX GPU + WebGPU + 集显上都能跑（画质有差异但都正确）
- 启动 log 清晰显示走 RT / raster
- 切换 capability 不需要重启

**风险**：低-中。Fallback 设计早期就该考虑，REQ-A 已经埋坑，这里只是收口。

**可独立合 PR**：是。

## 8.3 依赖图

```
multi-threading/08（IGpuTimeline + RetirePoint）
       │
frame-graph REQ-035 + REQ-038（FramePass + barrier 推导）
       │
temporal-techniques REQ-A（motion vector + history）
       │
gpu-driven-rendering REQ-F（cluster lighting）+ bindless-texture
       │
shadows REQ-A..C（cubemap shadow，fallback 路径）
       │
       ↓
   ┌───────────────────────────────────────┐
   │ REQ-A · RT 基础设施 + AS               │
   │ 4 extension + BLAS/TLAS + scratch     │
   └────────┬──────────────────────────────┘
            │
            ↓
       REQ-B · Ray Query 路径
            │
            ↓
       REQ-C · 朴素 RT Shadow ──→ visual baseline
            │
            │
            ↓
       REQ-D · RT Pipeline + SBT + Hit Shader
            │
            ├──────────────────────┐
            ↓                      ↓
       REQ-E · DDGI            REQ-F · Reflection + SVGF
            │                      │
            └──────────┬───────────┘
                       ↓
                REQ-G · Advanced RT Shadow（升级 REQ-C）
                       │
                       ↓
                REQ-H · 双路径 Fallback + 跨平台收口
```

**关键观察**：

- REQ-A/B/C 是 *desktop-only single point*——不阻塞跨平台路径，可早做
- REQ-D 是 *最大复杂度*，前置依赖最重（bindless 必须先做）
- REQ-E 跟 REQ-F 完全 *并行*（都基于 REQ-D，互不阻塞）
- REQ-H *早期设计、最后收口*——双路径思路 REQ-A 就埋坑

## 8.4 何时触发 REQ-A 立项

按 [`README §"何时触发本调研立项"`](README.md#何时触发本调研立项) 的判断，触发条件：

- **强触发**：[Phase 1 主路径](../../main-roadmap/phase-1-rendering-depth.md)（PBR / shadow / cluster lighting / G-buffer / bindless）+ temporal-techniques REQ-A 全部就位 → 立 REQ-A
- **强触发**：视觉品质要求 AAA 级（动态 GI / 屏幕外反射 / 多光源 soft shadow）→ 立 REQ-A + REQ-B + REQ-C
- **强触发**：项目转向 *desktop-only* 或明确放弃 Web → RT 进入 P0
- **中触发**：shadow map 的 acne / peter-panning 严重 → 立 REQ-A + REQ-B + REQ-C
- **弱触发**：纯学习 / 探索目的

**反触发（需谨慎评估）**：

- LX 强调 Web 兼容性（当前 Phase 1 路线图明确包含 WebGPU）→ RT 应延后或仅作 *desktop-only optional*
- 移动端目标 → RT 几乎不可行

当前 Phase 1 路线图 *把 Web 当作 P0*，所以 RT 调研的所有 REQ 都应延后到 Phase 1 主路径稳定后再考虑。

## 8.6 工业级延伸（不在本路线范围）

完成 REQ-H 后离 *最现代 RT 子系统* 还有以下方向，建议不进 v1 路线：

- **RT AO**（共享 SVGF 框架，REQ-F 已经埋坑）
- **RT 透射 / 折射**（hit shader 分材质 dispatch）
- **Path tracing reference renderer**（离线 ground truth，调试 SVGF 用）
- **ReSTIR GI / DI**（reservoir-based importance sampling）
- **Lumen-style screen+world space 混合**
- **OptiX denoiser 集成**（NN-based denoiser，画质高于 SVGF）
- **TLAS 增量更新**（部分 instance 改动只改部分 TLAS）
- **多 BLAS 异步 build queue**（async-compute REQ-A 升级）
- **Specular 跨帧累积**（高光 temporal stable 是 SVGF 难点）
- **DX12 / Metal RT 跨平台**（Apple Silicon Metal RT API）

这些都是 *优化层 / 高级特性*，等 v1 跑稳定且有真实需求再考虑。

## 8.7 风险与不确定性

### 风险 1：硬件碎片化 + Web 兼容性

最大架构风险。RT 仅 desktop RTX/RDNA2/Arc 支持；Web / 移动 / Apple 全 fallback。

**缓解**：

- REQ-H 双路径设计 *早期就埋坑*（REQ-A 起）
- 文档明确 RT 是 *desktop-only 加分项*，不是 *跨平台基础*
- shader system 必须支持条件编译

### 风险 2：内存代价

```
1000 mesh × 5 MB BLAS = 5 GB（不 compaction）
                     → 2 GB（compaction 后）
+ 多个 RT history texture
+ 3D probe grid
+ DDGI / reflection / shadow cache
─────────────
8 GB VRAM 上 RT 可能吃 30%+
```

**缓解**：

- AS compaction 必做（REQ-A）
- 大场景 streaming（不在 v1 范围）
- 用户可关 RT 特性

### 风险 3：BLAS 构建延迟

骨骼动画 / 形变 mesh 每帧 refit/rebuild → 几 ms。大量动态物体 → 阻塞 frame。

**缓解**：

- BLAS build 走 async compute queue（async-compute REQ-A）
- per-frame 限制 rebuild 数量（budget-driven）
- 静态 mesh 优先 compact，动态 mesh 不 compact

### 风险 4：SVGF 调参陷阱

`α`、`σ_n`、`σ_z`、`σ_l`、`mesh_id` 容忍度等魔法数字 *跟内容相关*。

**缓解**：

- 调试 UI（可视化每个权重 / variance / history）
- 对比测试（reference frame）
- profiling 工具

### 风险 5：RT pipeline 编译时间

RT pipeline 编译 *远比 graphics pipeline 慢*（几百 ms 量级，多 hit shader 时上 s 级）。

**缓解**：

- `VK_KHR_deferred_host_operations` 异步编译
- pipeline cache 强力使用
- 启动期 warm cache（预编译所有材质 hit shader）

### 风险 6：Mesh ID 跨帧稳定性

SVGF 强依赖 `mesh_id` 跨帧稳定。Asset 重新加载 / mesh 替换都会破坏。

**缓解**：

- 全局 stable mesh ID 系统（基于 GUID）
- 重新加载时保持 ID 不变
- SVGF disocclusion 检测兜底

## 8.8 一句话总结

**Ray tracing = LX 引入的最大复杂度调研，desktop-only 加分项。8 个 REQ 分四层：基础设施（A/B）→ 单点验证（C，朴素 shadow）→ 完整 RT pipeline 三大特性（D/E/F：DDGI + reflection + SVGF）→ 优化层（G/H：advanced shadow + 双路径 fallback）。前置依赖最重——必须 frame-graph + multi-threading/08 + temporal + cluster + bindless + shadows 全部就位。Web 兼容性是核心张力，REQ-H 双路径设计必须 REQ-A 就埋坑。**
