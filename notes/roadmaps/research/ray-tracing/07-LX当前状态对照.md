# 07 · LX 当前状态对照

> 阅读前提：[01](01-ray-tracing-范式.md) ~ [06](06-RT-reflection与SVGF.md) 讲完了 RT 范式 + AS + 调用范式 + 三大消费特性。本文综合盘点 LX gap。

## 7.1 LX 现状速查

LX 当前 *零 RT 能力*：

- 未启用任何 RT 相关 Vulkan 扩展
- 没有 BLAS / TLAS 类型
- 没有 RT pipeline 类型
- 没有 SBT 资源类型
- 没有 buffer device address 路径
- 没有 mesh_id / depth_dd 等 SVGF 必需的 G-buffer 通道
- Phase 1 路线图里 *没有* RT 相关 REQ

## 7.3 LX 已经做对的、能直接对接的部分

| LX 已有 | 在本调研里的作用 |

| `IGpuResource` 统一契约 | BLAS / TLAS / SBT / 多张 RT history texture 都走同接口 |

## 7.4 跟现有 LX REQ 的对照

LX Phase 1 路线图里跟本调研 *直接对应* 的 REQ 几乎没有：

| 现有 REQ | 跟本调研关系 |
|---------|------------|
| 无 | RT 完全未在 Phase 1 路线图，需新立 REQ |
| [REQ-118 PBR 完整管线](../../main-roadmap/phase-1-rendering-depth.md#req-118--pbr-完整管线) | RT reflection 跟 PBR 高光强协同（GGX VNDF 复用 BRDF） |
| [REQ-119 G-Buffer / 延迟渲染](../../main-roadmap/phase-1-rendering-depth.md#req-119--g-buffer--延迟渲染路径) | mesh_id / depth_dd 必须在 G-buffer 加通道 |
| [REQ-103 Shadow map](../../main-roadmap/phase-1-rendering-depth.md) | 跟 RT shadow 共存 / fallback |
| [REQ-105 Environment map](../../main-roadmap/phase-1-rendering-depth.md) | RT miss shader 用作 sky |

**重要观察**：本调研引入的能力 *Phase 1 路线图本来没有计划*，需要在路线图里新增条目。

## 7.5 跟前面 7 个调研的耦合

### 跟 multi-threading/08

| 耦合点 | 强度 |
|------|------|
| TLAS 跨帧 retire | **强** |
| BLAS lifetime 跟 mesh asset 联动 | 中 |
| RT shader compile 异步化（VK_KHR_deferred_host_operations） | 中 |

### 跟 frame-graph

| 耦合点 | 强度 |
|------|------|
| RT pass kind in FramePass | **强** |
| AS build pass + AS read pass barrier 推导 | **强** |
| 跨帧 TLAS / history 资源 frame graph 表达 | **强** |
| 自动 barrier 推导（REQ-038）必须 handle RT access mask | **强** |

### 跟 async-compute

| 耦合点 | 强度 |
|------|------|
| BLAS / TLAS build 走 compute queue | **强** |
| RT pass 是 async candidate | 中 |
| Ownership transfer：AS build 写 → graphics 读 | 中 |

### 跟 gpu-driven-rendering

| 耦合点 | 强度 |
|------|------|
| Cluster lighting 给 RT shadow / reflection 限定光源候选 | **极强** |
| Bindless texture（hit shader 必须） | **强** |
| Mesh shader 跟 BLAS 来源 *互斥*（同 mesh 二选一） | 中 |
| Indirect draw 跟 RT 路径 *并存* | 中 |

### 跟 shadows

| 耦合点 | 强度 |
|------|------|
| Cubemap shadow / RT shadow 共存（fallback） | **强** |
| Sparse residency 给 cubemap shadow，RT 不需要 | 弱 |
| 256 光源 importance metric 复用 | 中 |

### 跟 temporal-techniques

| 耦合点 | 强度 |
|------|------|
| Motion vector 复用 + RT shadow 加一层失效检测 | **强** |
| History texture 双 buffer 模式复用 | **强** |
| Blue noise + Poisson disk 风格相似 | 中 |
| TAA + SVGF 协同（dithering 抑制） | 中 |

### 跟 frame-graph + bindless-texture

| 耦合点 | 强度 |
|------|------|
| Hit shader bindless texture 采样 | **强** |
| `nonuniformEXT` GLSL 限定符 | 中 |

## 7.6 LX 当前位置（按 RT 能力层级）

```
Level 0: 无 RT                                     ← LX 在这里
Level 1: AS 基础（BLAS + TLAS + buffer device address）
Level 2: + ray query 路径（fragment / compute 可调 ray）
Level 3: + 简单 RT shadow（朴素版）
Level 4: + RT pipeline + SBT + 完整 hit shader     ← Ch12 教科书目标
Level 5: + DDGI 完整 5 步                          ← Ch14 教科书目标
Level 6: + RT reflection + SVGF 通用 denoiser      ← Ch15 教科书目标
Level 7: + RT AO / 透射 / area light shadow（共享 SVGF）
Level 8: + 跟 Lumen / ReSTIR / path tracing 看齐
```

要从 0 推到 6 需要本调研 *全部* 候选 REQ。

## 7.7 Web 兼容性（关键风险）

| 平台 | RT 支持 | LX 行为 |
|------|--------|--------|
| Desktop NVIDIA RTX 20+ | ✅ | RT 路径 |
| Desktop AMD RDNA 2+ | ✅ | RT 路径 |
| Desktop Intel Arc A | ✅ | RT 路径 |
| Apple Silicon M3+ | Metal RT API（不是 Vulkan） | 暂不支持 |
| 移动 GPU | 极少 | fallback |
| **WebGPU（截至 2026-04）** | **❌** | **必须 fallback** |

**架构含义**：

- 所有 RT 特性必须 *双路径设计*——RT path + non-RT path
- shader system 必须支持 *条件编译*（per-platform variants）
- 渲染管线必须 *能在不同后端跑同一场景*（RT 关时画面降级但不 crash）

这是跟 [Phase 1 跨平台目标](../../main-roadmap/phase-1-rendering-depth.md) 的根本张力——后续路线图必须明确 RT 是 *desktop-only 加分项*，不是 *跨平台基础*。

## 7.8 一句话定位

**本调研是 *最下游消费方* + *最大复杂度引入者*——依赖前面 7 个调研全部就位才能完整跑通**：

- frame-graph 提供多 pass + 跨帧资源 + RT pass kind
- multi-threading/08 提供 TLAS retire 模型
- async-compute 提供 AS build queue
- gpu-driven-rendering 提供 cluster lighting + bindless texture
- shadows 提供 cubemap shadow（共存 fallback）
- temporal-techniques 提供 motion vector + history infrastructure

但 *基础 RT shadow*（ray query 朴素版）可以早做一些，作为单点 demo 验证基础设施。
