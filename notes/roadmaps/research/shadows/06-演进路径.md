# 06 · 演进路径

> 阅读前提：[05](05-LX当前状态对照.md) 盘清 LX gap。本文给出推进路径切分。
>
> 注：本文给出的 REQ-A..G 编号是 *候选编号*，正式立项时按 `notes/requirements/` 实际可用号顺延。

## 6.1 总体策略

**结论先行：7 个候选 REQ 分两族**：

- **基础阴影族（A / B / C / G）**：从单 shadow map 入门，逐步加 cubemap、CSM、filtering
- **Sparse + 多光源族（D / E / F）**：让 N 光源同时投影成为可能

每个 REQ：

- 输入输出明确
- 工程编译可测
- 风险点单独立 REQ

## 6.2 七步演进序列

### REQ-A · Cubemap + Cubemap Array 基础设施

**目标**：让 LX 能创建、写入、读取 cubemap 和 cubemap array 资源。脱离 shadow 主题，通用基础。

**改动**：

- `IGpuResource` 加 cubemap 资源类型
- 资源创建支持 `VK_IMAGE_VIEW_TYPE_CUBE` / `CUBE_ARRAY`
- Layered rendering 支持（framebuffer 绑 layered image view）
- Shader 反射识别 `samplerCube` / `samplerCubeArray`
- 基础 demo：画一个 sky cubemap

**验收**：cubemap demo 跑通，sampling 正确。

**风险**：低。

**前置依赖**：无。

### REQ-B · Directional Shadow Map（基础阴影入门）

**目标**：实现 [REQ-103](../../main-roadmap/phase-1-rendering-depth.md#req-103--shadow-pass--depth-only-pipeline)。单 directional light + 单 2D shadow map + hard shadow。

**改动**：

- Depth-only pipeline 支持
- Shadow pass：从光源视角渲染场景
- 主渲染：投影 + depth 比较
- Bias / slope-scale bias 防 acne
- PBR shader 加 shadow 接收

**验收**：单 directional light 投影到地面，shadow 正确。

**风险**：低。教科书技术。

**前置依赖**：无（不强依赖 mesh shader）。

### REQ-C · Point Light Cubemap Shadow（少光源）

**目标**：单 point light → cubemap shadow。仍走 *传统* 渲染路径（vertex+fragment + 6 face pass）。

**改动**：

- Shadow pass 改成 cubemap layered rendering（`gl_Layer`）
- View / projection 矩阵 per-face
- `vector_to_depth_value` 工具
- Cubemap array（多光源用）但 v1 只画 1-2 光源

**验收**：1-2 个 point light 投影正确。

**风险**：中。Layered rendering + cubemap array 需要 framebuffer 配置正确。

**前置依赖**：REQ-A。

### REQ-D · Sparse Residency 基础设施

**目标**：完全 *跟 shadow 解耦* 的 Vulkan 基础能力。

**改动**：

- 资源支持 `SPARSE_RESIDENCY_BIT` flag
- Backend 提供 page pool 管理（`vmaAllocateMemoryPages`）
- `vkQueueBindSparse` 命令封装
- Bind semaphore 跟 timeline 协同
- Sparse-capable queue 选择策略

**验收**：

- 创建 sparse cubemap，bind 部分 page，sample 已 bind 部分正确
- 动态 unbind 部分 page，重新 bind 不同 page
- Validation layer 全绿

**风险**：中。Vulkan API 复杂，但 algorithm 不难。

**前置依赖**：multi-threading/08 timeline backend 实现。

### REQ-E · Per-Light Importance Metric

**目标**：算每个光源该用多少 shadow 分辨率。

**改动**：

- Cluster 跟 [clustered lighting REQ-F](../gpu-driven-rendering/06-演进路径.md#req-f--clustered-lighting) 共用
- Importance metric 算法（cluster solid angle）
- Per-light resolution 决策上传到 GPU

**验收**：

- 远处光源分辨率 < 近处光源
- 内存占用对比 vs 固定分辨率：50%+ 节省

**风险**：低。

**前置依赖**：clustered lighting REQ-F（需要 cluster 数据结构）。

### REQ-F · Multi-Light Sparse-Bound Shadow（**核心整合**）

**目标**：把 REQ-C / D / E 整合，支持 256 光源同时 cubemap shadow。

**改动**：

- 用 sparse cubemap array 容纳所有光源 shadow
- Per-frame 按 importance 决定 bind
- 用 mesh shader 路径（[gpu-driven REQ-G](../gpu-driven-rendering/06-演进路径.md#req-g--mesh-shader-pipeline--实战)）渲染（可选 compute culling fallback）
- 4 步流水线（[03 mesh shader shadow](03-mesh-shader-shadow.md)）

**验收**：

- 256 光源场景 real-time 阴影
- 内存可控（< 200 MB）
- Validation 全绿
- 跟 REQ-109 多光源合同对接

**风险**：**高**（核心整合点，跟多个调研协同）。

**前置依赖**：

- REQ-C + D + E
- gpu-driven REQ-G（mesh shader）+ REQ-D（compute culling fallback）
- frame-graph REQ-038（barrier 推导）— shadow texture 跨 pass 流转
- async-compute REQ-A（compute pipeline）

### REQ-G · Shadow Filtering + CSM

**目标**：画质升级 — PCF / VSM 软化 + cascaded shadow maps for directional。

**改动**：

- PCF 3×3 / 5×5 实施
- 可选 VSM（variance shadow maps）
- CSM：4 级 cascade，logarithmic + uniform 切分
- Cascade 边界 fade，避免硬边

**验收**：

- Directional shadow 拉远拉近不糊不闪
- PCF 软化锯齿明显
- 性能可接受（PCF 提升 ~20% shading 时间）

**风险**：中。Algorithm 成熟，工程实施细节多。

**前置依赖**：REQ-B。

                   │       ↓                            ↓ ← clustered REQ-F          │

                   │       └────────→  REQ-F (256 lights sparse shadow) [风险点]     │
                   │                       ↑                                          │

- REQ-A / REQ-B / REQ-D 完全独立可并行
- REQ-C 依赖 A
- REQ-E 依赖 clustered REQ-F
- REQ-F 是大整合，依赖 C + D + E + 多个其它调研
- REQ-G 依赖 B

## 6.4 何时触发 REQ-A / B 立项

按 [README §"何时触发本调研立项"](README.md#何时触发本调研立项) 的判断：

强触发：

- [REQ-103 Shadow pass](../../main-roadmap/phase-1-rendering-depth.md#req-103--shadow-pass--depth-only-pipeline) 立项 → 触发 A + B（基础阴影入门）
- [REQ-109 多光源](../../main-roadmap/phase-1-rendering-depth.md#req-109--pointlight--spotlight--统一多光源合同) 立项 + 决定走 256 光源场景 → 触发 C + D + E + F
- 其它项目（streaming texture / virtual geometry）需要 sparse → REQ-D 单独立项

中触发：

- [REQ-104 CSM](../../main-roadmap/phase-1-rendering-depth.md#req-104--cascaded-shadow-maps) 立项 → 触发 G

## 6.6 风险与不确定性

### 风险 1：REQ-F 多调研协同

REQ-F 是整个路线的核心整合点，依赖 4 个其它调研：

- gpu-driven mesh shader / compute culling
- frame-graph 自动 barrier
- async-compute compute pipeline
- multi-threading/08 timeline

任一前置未就位会让 REQ-F 无法落地。需要严格按时序推进。

### 风险 2：Sparse residency 跨硬件兼容

不同 GPU 厂商的 sparse 支持差异大：

- NVIDIA：支持完整
- AMD：基本支持
- Intel：部分支持
- 移动 GPU：通常不支持

需要 fallback 路径：硬件不支持 sparse 时，固定分配最低分辨率（如 64×64）or 限制光源数。

### 风险 3：Cubemap face / multiview 当前是 6× draw call

NV mesh shader 不支持 multiview，6 face = 6 indirect draws。当 KHR mesh shader 完全成熟后能优化为 1 个 multiview draw。

实施时应 *接口预留*，等 KHR 成熟切换。

## 6.7 工业级延伸（不在本路线范围）

完成 REQ-G 后离工业级阴影还有：

- **Contact-hardening shadow**（PCSS / RTGI）
- **Volumetric shadow**（光柱、雾中阴影）
- **Hair shadow**（stochastic shadow）
- **Translucent shadow**（半透明物体）
- **Raytraced shadow**（硬件加速）
- **Shadow caching**（静态光源 cache shadow）

这些是 *优化层 / 高级特性*，等 v1 跑稳定再考虑。

## 6.8 一句话总结

**Shadow + Sparse = 现代多光源渲染的最后一块拼图。LX 当前 Level 0；推到 Level 5（256 光源 sparse shadow）需要 7 个候选 REQ + 前面四个调研协同。基础阴影 REQ-A/B 可以早做（独立），但 REQ-F 整合是高风险节点。Phase 1 REQ-103 + REQ-109 立项是真正的强触发。**
