# 01 · 动机与核心模型

> 阅读前提：了解 Vulkan 的基本概念（command buffer、pipeline、descriptor），但不必用过 bindless。

## 1.1 传统绑定模型的痛点

在传统（非 bindless）的 Vulkan / D3D12 代码里，"让一张贴图给 shader 用" 要经过这么几步：

```
一张贴图想被 shader 看到
    ↓
绑到某个 descriptor set 的某个 binding 上    ← vkUpdateDescriptorSets
    ↓
这个 descriptor set 的 layout 要被 pipeline 知道  ← pipeline layout 里声明
    ↓
draw 之前把 set 绑到 command buffer          ← vkCmdBindDescriptorSets
    ↓
shader 里 sampler2D tex 采样
```

**每一个材质都是这一条链路的一次重复**。如果场景里有 1000 个物体、200 种材质：

- 200 个 descriptor set（一个材质一个）
- 每帧大量 `vkCmdBindDescriptorSets` 调用
- 如果材质用的贴图数量不同（有的带法线，有的不带），descriptor set layout 也不同 → pipeline layout 也得不同 → pipeline 还要重新创建

本质矛盾：**"哪张贴图"这个信息被编码进了 pipeline state**，所以一切"贴图变了"的变化都会传染到 pipeline。

## 1.2 Bindless 的基本思路

核心想法一句话：

> **把所有纹理丢到一个超大的全局数组里，shader 通过"下标"访问。**

形象一点：

| 传统模型 | Bindless 模型 |
|---------|--------------|
| 每个材质一个小柜子（descriptor set），柜子里的抽屉（binding）预先排好 | **一个巨大的公共货架（全局数组）**，所有纹理都放上去 |
| 画某物体前，先"开对应柜子"（bind set），再采样 | 货架永远是敞开的，shader 拿着"货号"（纹理 index）去取 |

这样一来：
- **每帧只需 bind 一次这个全局货架**，不用每个材质切换
- **所有材质共享同一套 descriptor set layout** → pipeline layout 全局收敛
- **"哪张贴图"变成了 shader 运行时的一个 `u32`**，不再是 pipeline 状态

## 1.3 Bindless 的两层模型：binding vs slot

理解 bindless 最关键的一步，是把 **binding** 和 **slot** 这两层分清楚。

```
descriptor set（整个应用只有 1 个）
└── binding = 10, type = COMBINED_IMAGE_SAMPLER, descriptorCount = 65536
    │   ↑ 这是 binding。shader 里写 binding = 10 的那个数字。
    │   ↑ 创建 descriptor set layout 时定死，之后永不变。
    │
    └── 数组内部有 65536 个 slot：
        ├── slot 0:    (VkImageView, VkSampler)  ← 纹理 A
        ├── slot 1:    (VkImageView, VkSampler)  ← 纹理 B
        ├── slot 2:    (VkImageView, VkSampler)  ← 纹理 C
        ├── slot 3:    空（允许不填，见 02 章 PARTIALLY_BOUND）
        ├── ...
        └── slot 65535: 空
```

### 两层的区别

| | 是什么 | 谁决定 | 什么时候变 |
|---|------|-------|---------|
| **binding** | 数组的"地址"，shader 里 `layout (binding = 10)` 这个整数 | descriptor set layout | 创建时定死，**永不变** |
| **slot** | 数组内部某个位置的下标 | 资源管理器 | 运行时 **动态分配、回收** |

### Shader 端长什么样

```glsl
layout (set = 1, binding = 10) uniform sampler2D global_textures[];

// 采样时：
color = texture(global_textures[material.albedoIndex], uv);
//                             ^^^^^^^^^^^^^^^^^^^^^
//                             slot 下标，从 material 数据里读出来
```

`binding = 10` 永远不变。`[]` 里的那个下标才是 slot。

### 数据流：从 draw call 到实际采样

真实引擎里通常是"三层跳表"：

```
drawcall push constant:  object_id (u32, 4 字节)
         ↓
SSBO object_data[object_id]: { materialId, transformId, ... }
         ↓
SSBO material_data[materialId]: { albedoTexIdx, normalTexIdx, flags, ... }
         ↓
bindless 数组 global_textures[albedoTexIdx]: 实际的纹理
```

CPU 端 drawcall 只传一个 32-bit 索引，其他全靠 GPU 端间接查表。这才是 bindless 能喂养 GPU-driven rendering 的前提 —— CPU 连材质信息都不用知道。

## 1.4 核心收益（预告）

Bindless 直接带来的好处：

- ✅ **Descriptor set layout 全局收敛**（通常 1 个）
- ✅ **Pipeline layout 全局收敛**（通常 1~2 个）
- ✅ **大幅减少 `vkCmdBindDescriptorSets` 调用**
- ✅ **顶点数据也能塞进 SSBO**，顺带消除 vertex input state 差异

但要注意：

- ❌ **Bindless 本身不会减少 PSO 数量**
- ❌ **不会让不同的 shader 合并成一个**

让 PSO 从 10 万 打下来到 ~500，需要 bindless **配合** ubershader + GPU-driven。这部分详见 [04 · Pipeline 与 Shader 变量策略](04-Pipeline与Shader策略.md)。

## 1.5 需要的硬件/API 支持

- **Vulkan**: `VK_EXT_descriptor_indexing`（已进 core 1.2）
- **D3D12**: SM 6.6 `ResourceDescriptorHeap` / Dynamic Resources
- **OpenGL**: `ARB_bindless_texture`（NV/AMD 扩展，驱动差异大）
- **Metal**: Argument Buffers（iOS 13+/macOS 10.13+）
- **WebGPU**: 暂未标准化

平台支持率：Windows 桌面 ~42%、Linux ~52%、Android ~1%。桌面端/主机端是标配，**移动端仍然是最大障碍**。
