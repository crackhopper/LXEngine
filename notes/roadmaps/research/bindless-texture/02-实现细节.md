# 02 · 实现细节

> 这一章展示 Vulkan 下一个真实的 bindless 实现怎么搭起来。代码骨架来自 *Mastering Graphics Programming with Vulkan*（Raptor Engine）。
> 本章覆盖：**设备特性启用 → descriptor pool → descriptor set layout → descriptor set → 纹理更新 → shader 端基础代码 → 同步模型**。
> 关于 `nonuniformEXT` 和 warp 执行模型的深入解释，见 [03 · nonuniform 与 warp 执行模型](03-nonuniform与warp执行模型.md)。

## 2.1 启用 `VK_EXT_descriptor_indexing`

创建 `VkDevice` 时把 `indexing_features` chain 到 `VkPhysicalDeviceFeatures2`：

```cpp
VkPhysicalDeviceFeatures2 physical_features2 = {
    VK_STRUCTURE_TYPE_PHYSICAL_DEVICE_FEATURES_2 };
vkGetPhysicalDeviceFeatures2(vulkan_physical_device, &physical_features2);

VkDeviceCreateInfo device_create_info = {};
device_create_info.pNext = &physical_features2;
if (bindless_supported) {
    physical_features2.pNext = &indexing_features;
}
vkCreateDevice(...);
```

需要检查的关键 feature：`descriptorBindingPartiallyBound`、`descriptorBindingUpdateAfterBind`、`runtimeDescriptorArray`、`shaderSampledImageArrayNonUniformIndexing`。缺任何一个都得回退到传统路径。

## 2.2 创建 Descriptor Pool

关键是加上 `VK_DESCRIPTOR_POOL_CREATE_UPDATE_AFTER_BIND_BIT_EXT`。这个 flag 让 pool 能分配"绑定后仍可更新"的 set：

```cpp
VkDescriptorPoolSize pool_sizes_bindless[] = {
    { VK_DESCRIPTOR_TYPE_COMBINED_IMAGE_SAMPLER, k_max_bindless_resources },
    { VK_DESCRIPTOR_TYPE_STORAGE_IMAGE,          k_max_bindless_resources },
};

VkDescriptorPoolCreateInfo pool_info = {};
pool_info.flags = VK_DESCRIPTOR_POOL_CREATE_UPDATE_AFTER_BIND_BIT_EXT;
pool_info.maxSets = k_max_bindless_resources * ArraySize(pool_sizes_bindless);
pool_info.poolSizeCount = ArraySize(pool_sizes_bindless);
pool_info.pPoolSizes = pool_sizes_bindless;
vkCreateDescriptorPool(...);
```

`k_max_bindless_resources` 典型值：4096 / 16384 / 65536。看引擎需求和硬件限制。

## 2.3 创建 Descriptor Set Layout（全局唯一）

这一步是 bindless 架构的"心脏"。整个应用只创建一次。

```cpp
VkDescriptorSetLayoutBinding vk_binding[2];

// binding 0：所有 sampled image（2D 贴图）统一放这里
VkDescriptorSetLayoutBinding& image_sampler_binding = vk_binding[0];
image_sampler_binding.descriptorType  = VK_DESCRIPTOR_TYPE_COMBINED_IMAGE_SAMPLER;
image_sampler_binding.descriptorCount = k_max_bindless_resources;  // ← 这是"数组长度"
image_sampler_binding.binding         = k_bindless_texture_binding; // 比如 10

// binding 1：storage image（compute shader 用）
VkDescriptorSetLayoutBinding& storage_image_binding = vk_binding[1];
storage_image_binding.descriptorType  = VK_DESCRIPTOR_TYPE_STORAGE_IMAGE;
storage_image_binding.descriptorCount = k_max_bindless_resources;
storage_image_binding.binding         = k_bindless_texture_binding + 1;
```

关键点：**`descriptorCount` 不是"descriptor 的个数"，是"这个 binding 内部数组的长度"**。整个 layout 只声明了 2 个 binding，每个 binding 是一个长数组。

然后需要给每个 binding 挂上 bindless 三件套的 flag：

```cpp
VkDescriptorBindingFlags bindless_flags =
    VK_DESCRIPTOR_BINDING_PARTIALLY_BOUND_BIT_EXT |
    VK_DESCRIPTOR_BINDING_UPDATE_AFTER_BIND_BIT_EXT;
VkDescriptorBindingFlags binding_flags[2] = { bindless_flags, bindless_flags };

VkDescriptorSetLayoutBindingFlagsCreateInfoEXT extended_info = {
    VK_STRUCTURE_TYPE_DESCRIPTOR_SET_LAYOUT_BINDING_FLAGS_CREATE_INFO_EXT };
extended_info.bindingCount = 2;
extended_info.pBindingFlags = binding_flags;

VkDescriptorSetLayoutCreateInfo layout_info = {
    VK_STRUCTURE_TYPE_DESCRIPTOR_SET_LAYOUT_CREATE_INFO };
layout_info.bindingCount = 2;
layout_info.pBindings = vk_binding;
layout_info.flags = VK_DESCRIPTOR_SET_LAYOUT_CREATE_UPDATE_AFTER_BIND_POOL_BIT_EXT;
layout_info.pNext = &extended_info;

vkCreateDescriptorSetLayout(...);
```

三个 flag 的分工：

| Flag | 作用 |
|------|------|
| `PARTIALLY_BOUND_BIT` | shader 不访问的 slot 可以不初始化，整个大数组不必填满 |
| `UPDATE_AFTER_BIND_BIT` | descriptor set 绑到 command buffer 之后仍然能更新其中的 slot |
| `UPDATE_AFTER_BIND_POOL_BIT`（layout 级） | 告诉 layout "我要从允许 UPDATE_AFTER_BIND 的 pool 分配" |

## 2.4 分配 Descriptor Set（全局唯一一个）

```cpp
VkDescriptorSetAllocateInfo alloc_info = { VK_STRUCTURE_TYPE_DESCRIPTOR_SET_ALLOCATE_INFO };
alloc_info.descriptorPool     = vulkan_bindless_descriptor_pool;
alloc_info.descriptorSetCount = 1;                              // ← 就 1 个
alloc_info.pSetLayouts        = &vulkan_bindless_descriptor_layout;
vkAllocateDescriptorSets(..., &vulkan_bindless_descriptor_set);
```

⭐ **整个应用生命周期只分配一次这一个 set**。这是 bindless 最颠覆的一点——不是"一个材质一个 set"，是"一个应用一个 set"。

## 2.5 纹理加载与 Update 流程

### 加载时：入队

```cpp
if (gpu.bindless_supported) {
    ResourceUpdate resource_update{
        ResourceDeletionType::Texture,
        texture->handle.index,     // ← 纹理在 bindless 数组里的 slot
        gpu.current_frame
    };
    gpu.texture_to_update_bindless.push(resource_update);
}
```

### 帧开始：集中 flush

```cpp
for (i32 it = texture_to_update_bindless.size - 1; it >= 0; it--) {
    ResourceUpdate& u = texture_to_update_bindless[it];
    Texture* texture = access_texture({ u.handle });

    VkWriteDescriptorSet descriptor_write = { VK_STRUCTURE_TYPE_WRITE_DESCRIPTOR_SET };
    descriptor_write.descriptorCount = 1;
    descriptor_write.dstArrayElement = u.handle;          // ← slot 下标
    descriptor_write.descriptorType  = VK_DESCRIPTOR_TYPE_COMBINED_IMAGE_SAMPLER;
    descriptor_write.dstSet          = vulkan_bindless_descriptor_set;
    descriptor_write.dstBinding      = k_bindless_texture_binding;  // ← 永远是 10

    VkDescriptorImageInfo image_info = {};
    image_info.sampler   = texture->sampler->vk_sampler;
    image_info.imageView = texture->vk_image_view;
    image_info.imageLayout = VK_IMAGE_LAYOUT_SHADER_READ_ONLY_OPTIMAL;
    descriptor_write.pImageInfo = &image_info;

    vkUpdateDescriptorSets(device, 1, &descriptor_write, 0, nullptr);
    texture_to_update_bindless.delete_swap(it);
}
```

`dstBinding` = 10（哪条 binding），`dstArrayElement` = slot（哪个位置），两者合起来定位到一个 descriptor。

## 2.6 Shader 端基础代码

```glsl
#version 450
#extension GL_EXT_nonuniform_qualifier : enable  // ← 每个 shader 都要开

// 不同类型可以别名到同一个 binding
layout (set = 1, binding = 10) uniform sampler2D   global_textures[];
layout (set = 1, binding = 10) uniform sampler3D   global_textures_3d[];
layout (set = 1, binding = 10) uniform samplerCube global_cubemaps[];

void main() {
    uint albedoIdx = material.albedoIndex;
    vec4 color = texture(global_textures[nonuniformEXT(albedoIdx)], uv);
    //                                  ^^^^^^^^^^^^^^^^^^^^^^^^^
    //                  为什么需要这个限定符？见 03 章详解。
    ...
}
```

### 类型别名（alias）技巧

同一个 `(set, binding)` 声明多个不同类型的数组。底层 descriptor 类型仍是 `COMBINED_IMAGE_SAMPLER`，shader 编译器根据访问时用 `global_textures[i]` 还是 `global_textures_3d[i]` 把 slot 里的数据解释成 2D 或 3D。

**代价**：类型安全由应用保证。如果 slot 5 实际是 2D 贴图，shader 里却当 3D 来采，编译通过、运行时 UB。

## 2.7 同步与更新策略

这块是 bindless 最容易踩坑的地方。规范原话：

> descriptors in this binding can be updated after a command buffer has bound this descriptor set, **provided that the descriptor is not used by any command buffers that are in the pending state**.

翻译成人话：**Vulkan 只放宽了规则，不保证安全。应用必须自己保证"不更新正在被读的 slot"**。

### 传统 vs Bindless 的同步粒度

| | 传统 descriptor set | Bindless + UPDATE_AFTER_BIND |
|---|---|---|
| Fence 粒度 | 整个 set | **单个 slot** |
| "何时能 update" | set 未被任何在途 command buffer 引用 | **这个 slot 不被在途 shader 实际读取** |
| 检查者 | 驱动 + validation layer | validation layer（仅 debug），**硬件不查** |

### 三个"结构性"策略保证安全

真实引擎不需要显式锁。靠三条规矩就够：

#### 策略 1：Append-only，永不覆盖活跃 slot

加载新纹理时**取空 slot**，不复用正在用的：

```
slot 0..99    帧 N-1 使用中
slot 100      ← 新纹理写到这里
```

帧 N-1 的 shader 不知道 slot 100 存在，自然不会读。**RAW 冲突不存在**。

等价于 append-only log + GC 的经典并发规避套路。

#### 策略 2：删除走 Frames-in-Flight 延迟回收

```cpp
void deleteTexture(TextureHandle h) {
    deletionQueue.push({ h, currentFrame });  // 排队，不立即释放
}

void beginFrame() {
    while (!deletionQueue.empty() &&
           currentFrame - deletionQueue.front().frame >= FRAMES_IN_FLIGHT) {
        freeSlot(deletionQueue.front().handle);  // 过了 N 帧才真正归还
        deletionQueue.pop();
    }
}
```

frame counter 担任应用级 fence。`FRAMES_IN_FLIGHT` 通常 2 或 3。

#### 策略 3：帧起始集中 flush update

在帧开始、录制任何 command buffer **之前** 集中调一次 `vkUpdateDescriptorSets`：

```cpp
void beginFrame() {
    reclaimDeletedSlots();          // 策略 2
    flushPendingDescriptorWrites(); // 把新纹理一次性写进 bindless set
    // 之后才 record command buffers
}
```

### 为什么这样就够了

把三条合起来看：

1. **新资源** —— 永远拿空 slot，in-flight 帧不可能知道 → 零冲突
2. **删除资源** —— 延迟 N 帧回收，保证 in-flight 帧读完才归还 → 零冲突
3. **更新时机** —— 集中在帧开始前、命令录制之前 → 顺序明确

`UPDATE_AFTER_BIND` 的价值不是"允许并发乱写"，而是"**加一张新纹理不需要 wait idle 所有 GPU**"。

### 唯一需要显式 barrier 的场景

"加载的纹理当帧就要用" —— VRAM 上传（`vkCmdCopyBufferToImage`）要在采样之前完成。这是图像 memory barrier 的事，和 bindless 无关，**传统绑定模型下也要做**。Bindless 没有新增同步义务。
