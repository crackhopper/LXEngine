# 05 · 业界现状对比

> 这一章横向看 bindless 在不同引擎的采用情况，为 LX Engine 的选型提供参考。
> 调研日期：2026-04-23。

## 5.1 AAA / 商业引擎：已是默认方向

| 引擎 | Bindless 状态 | 备注 |
|-----|-------------|------|
| **id Tech 7 (Doom Eternal)** | 2020 起全面 bindless | SIGGRAPH 2020 公开分享的教科书案例，~100 shader / ~350 PSO |
| **Unreal Engine 5** | 5.0 起逐步引入 Bindless Resources，5.5 扩大平台支持 | Nanite 的 visibility buffer 路径本身依赖 bindless |
| **Frostbite / Decima / Snowdrop / Anvil** | 新架构都是 bindless-first | 多数走 GPU-driven 渲染 |

## 5.2 API 层支持情况

| API | Bindless 入口 | 状态 |
|-----|-------------|------|
| **Vulkan** | `VK_EXT_descriptor_indexing` | 已进 core 1.2，Windows ~42% / Linux ~52% / Android ~1% 支持率 |
| **D3D12** | SM 6.6 `ResourceDescriptorHeap` / Dynamic Resources | 写现代 D3D12 几乎必用 |
| **OpenGL** | `ARB_bindless_texture` | NV/AMD 扩展，驱动差异大 |
| **Metal** | Argument Buffers | iOS 13+/macOS 10.13+ |
| **WebGPU** | 暂未标准化 | 浏览器路径目前没有 bindless |

结论：**桌面/主机标配，移动端最大阻碍**。

## 5.3 Godot 的讨论与现状

Godot 作为开源引擎、且架构设计公开透明，是值得重点研究的参照物。

### 5.3.1 当前状态（Godot 4.4 / 4.5）

**Godot 目前不使用 bindless。** Forward+ 渲染器走的是传统 descriptor set 路径：

- `RenderingDevice` 抽象层统一封装 Vulkan / Direct3D 12 / Metal 三套现代 API
- `RendererCompositorRD` 选择具体后端，但对材质仍然采用 "每个 material 一套 descriptor set" 的传统模型
- 官方三档渲染器（Forward+ / Mobile / Compatibility）共享同一套 `.gdshader` 源码，但 Compatibility 档走 GLES3，没有 bindless 能力。这是 Godot 不敢全面 bindless 的根本约束。

### 5.3.2 关键阻塞：`nonuniformEXT` 还没实现

Godot 社区已有明确提案要求补齐 bindless 的前置能力：

- **[godot-proposals#10423](https://github.com/godotengine/godot-proposals/issues/10423)** —— *Add support for non-uniform indexing qualifier (`nonuniformEXT()` in GLSL) in shaders*
  - 状态：**Open，未实现**（2024-08 最后活跃）
  - 场景：地形系统需要根据顶点位置动态索引 `sampler2D` 数组，目前只能用 `switch` 语句笨拙绕过
  - 目标范围：仅限 Vulkan / Forward+ 后端
  - 意义：这是引入真正 bindless 的硬前提 —— 没有 `nonuniformEXT` 就只能做"伪 bindless"（索引必须是 wave-uniform 的常量），触及不到现代 bindless 的核心场景

（为什么必须有 `nonuniformEXT`，见 [03 · nonuniform 与 warp 执行模型](03-nonuniform与warp执行模型.md)。）

### 5.3.3 reduz 的 GPU-driven 渲染器蓝图

Godot 主创 Juan Linietsky (reduz) 在 [GPU Driven Renderer for Godot 4.x](https://gist.github.com/reduz/c5769d0e705d8ab7ac187d63be0099b5) 中给出了将来的 bindless 路径：

- **认为技术门槛不高** —— 原话 "The bindless implementation should be relatively simple to do"
- **纹理方案**：`uniform texture2D textures[MAX_TEXTURES]`，走 unbounded array + 动态索引
- **顶点方案**：vertex pulling via `utextureBuffer`，顶点数据从 buffer 里拉取而非靠 pipeline 的 vertex input layout
- **无公开时间表** —— gist 本身是设计草案，不是 release plan

### 5.3.4 <a name="godot-最有参考价值的设计"></a>Godot 最有参考价值的设计：Shader 语言抽象

Godot 的关键创新不在 bindless 本身，而在**把 bindless 推进"后端实现细节"而非"用户可见 API"**。引用 reduz：

> "a `uniform sampler2D` can be converted to a regular sampler2D in GLES3, a texture2D with predefined samplers in Vulkan, or a bindless access in a GPU driven renderer all behind the user's back."

也就是说，同一份 `.gdshader` 源码，编译器根据目标后端产出完全不同的绑定形式：

| 后端 | 采样器表达 |
|------|-----------|
| GLES3（Compatibility） | 传统 `sampler2D`，按 slot 绑定 |
| Vulkan（Forward+ / Mobile，当前） | `texture2D` + 独立 `sampler`，descriptor set 绑定 |
| GPU-driven（未来） | bindless 数组 + 运行时索引 |

对 LX Engine 的启发是：**如果想给未来的 bindless 改造留后路，应该把 `sampler2D` 式的表达封装在引擎的 material DSL / shader graph 里，而不是让用户直接写裸 GLSL**。否则一旦切到 bindless，用户 shader 全部要重写。

### 5.3.5 Godot 的路径 vs AAA 引擎的路径

| 维度 | Godot | id Tech / UE5 |
|------|-------|---------------|
| 核心矛盾 | 兼容最广硬件（含移动/旧 GPU） | 榨取高端硬件极限 |
| 当前 bindless 状态 | 未实现，在规划中 | 已经是核心架构 |
| 抽象位置 | 在 shader 语言层抽象，用户无感 | 暴露给引擎/材质作者 |
| 推进节奏 | 保守，先落 `nonuniformEXT` 等前置 | 激进，带动整个管线重构 |

**结论**：Godot 把 bindless 当成 GPU-driven 渲染大重构的一环，而不是单独功能点。这和它"长期支持低端/旧设备"的定位一致。

## 5.4 KlayGE 的现状（华人开源渲染实验平台）

KlayGE 是 **龚敏敏 (gongminmin)** 从 2003 年开始维护的开源跨平台引擎，slogan "To arm your engine with cutting-edge technology" 强调的是 **前沿渲染技术集成**（PBR、DDGI、SSR、过程化材质、HDR 管线等），而不是现代底层 API 的激进采用。

### 5.4.1 仓库现状（截至 2026-04）

- 活跃度：仍在维护，最新 push 2025-11-05，约 2.1k ⭐、6000+ commits
- 作者：龚敏敏（原 Microsoft / Tencent 图形工程师，GDC China 常客）
- 默认分支：`develop`，版本迭代持续

### 5.4.2 渲染后端分布

| 后端 | 状态 |
|------|------|
| D3D11 | 有 |
| **D3D12** | 有 |
| OpenGL | 有 |
| OpenGL ES | 有 |
| NullRender | 有 |
| **Vulkan** | **无** |
| Metal | 无 |

关键观察：**KlayGE 没有 Vulkan 后端**。这点在 2026 年已经相当罕见，说明它的渲染架构演进路径与现代 Khronos / bindless 生态不完全同步。

### 5.4.3 Bindless 支持情况：未使用

对仓库全文搜索 `bindless`、`descriptor_indexing` 的结果：

- 所有 `bindless` 匹配都在 **`glloader/xml/` 自动生成的 OpenGL 扩展注册描述文件** 中（如 `GL_ARB_bindless_texture.xml`、`GL_NV_bindless_multi_draw_indirect.xml`）。这些只是 glloader 代码生成器扫描的扩展清单，**不代表引擎实际使用了这些扩展**
- `VK_EXT_descriptor_indexing` / `descriptor_indexing` **零命中**（也符合 —— 没有 Vulkan 后端）
- D3D12 后端未见 SM 6.6 `ResourceDescriptorHeap` 风格的 bindless 编码

结论：**KlayGE 目前没有采用 bindless 渲染架构**。它的 D3D12 路径仍是传统的每材质 descriptor table 绑定模型。

### 5.4.4 KlayGE 的定位差异

KlayGE 关注的"前沿"偏 **渲染算法层面**（光照、GI、去噪、材质模型）而不是 **API/绑定模型层面**。它的架构在"先进算法可以跑起来"这件事上已经足够，没有为 bindless 做主动迁移。

## 5.5 三方对比总览

| 维度 | AAA (UE5 / id Tech) | Godot | KlayGE |
|------|-------------------|-------|--------|
| 核心定位 | 商业 AAA 工具链 | 通用型开源引擎 | 前沿渲染技术研究平台 |
| API 策略 | 激进、全覆盖 | Vulkan + D3D12 + Metal | D3D 优先 + GL/GLES，**无 Vulkan** |
| Bindless 状态 | **已成为核心架构** | **规划中，shader 层抽象已就位** | **未采用** |
| 推动力 | 商业需求 + 项目预算 | 社区 + 多人维护 | 个人技术追求 + 长期积累 |
| Shader 抽象 | 材质作者可见 | `.gdshader` 后端无感 | 裸 HLSL/GLSL |
| 推进节奏 | 最快 | 中等（卡在前置能力） | 最慢 |

## 5.6 对 LX Engine 的启示

- **想做技术研究 / 演示**：**KlayGE 路线**（不必强求 bindless，把时间投入渲染算法）
- **想做通用开源引擎**：**Godot 路线**（在 shader 抽象层留接口，逐步演进）
- **想做面向"商业级"/AI 工作流的生产级渲染**：**更像 UE5 / id Tech 路线**，bindless 该上就上

LX Engine 目前定位偏向第三种（见 `notes/roadmaps/README.md` 的 AI-Native 愿景），所以可以比 Godot 更激进地采用 bindless。但要避免 KlayGE 的反面陷阱：**别把时间都砸在单个高端 feature 上而推迟架构层面的现代化**。

具体演进路径见 [06 · LX Engine 演进路径](06-LXEngine演进路径.md)。
