# 01 · 概念与选型

> 从"为什么需要多线程"讲到"现在主流用哪种架构、为什么选它"，面向没做过引擎多线程的读者。

## 1.1 为什么游戏引擎必须多线程

CPU 单核频率多年没有明显增长了，收益全在"核数更多"。游戏引擎如果不多线程：

- 渲染命令录制跑满一个核，其他 15 个核闲着
- 资源加载阻塞主线程 → 关卡切换卡死
- 物理 / 动画 / 音频 / gameplay 排成串行，帧时间累加
- AI agent（LX 的 Phase 10 愿景）产生额外计算需求，想插队没地方放

所以 multi-threading 不是"锦上添花"，是**可持续演进的底座**。

## 1.2 三种主流架构

历史上引擎多线程架构演进大致经过三代：

### (a) Thread-per-subsystem（老架构）

每个子系统一个专属线程：一个线程做渲染、一个做物理、一个做音频、一个做 I/O。

- ✅ 实现简单直观
- ❌ 粒度粗，负载不均：物理帧跑完了、渲染还没开始，物理线程闲着
- ❌ 扩展性差：双核够用，16 核浪费 12 核
- ❌ 子系统之间同步复杂，生命周期容易出错

代表：早年 Unreal 2 / 3 时代的"Game thread + Render thread"。

### (b) Task-based（现代主流）

代码切成小块、独立、不可中断的 **task**，一堆 worker thread 从队列里抢任务执行。依赖关系通过"task A 依赖 task B 完成"显式声明。

- ✅ 粒度细、负载均衡好，核数越多收益越大
- ✅ **不可中断** —— 每个 task 是原子执行块，不存在"跑到一半被抢占"的情况，调试直接
- ✅ 平台无关，依赖操作系统线程但不依赖特定调度
- ✅ 成熟开源库多：enkiTS / TaskFlow / Intel TBB / oneAPI

代表：Destiny（Bungie）、Unreal 4+、Unity Job System、id Tech 6+。

### (c) Fiber-based（追求极致的架构）

**fiber** 是用户态的协程 —— 一段代码可以主动 `yield`、被调度器 `resume`。和 task 的区别在于 fiber 可以**被中断**（在 yield 点），然后被重新排到别的线程。

- ✅ 细粒度任务 + 阻塞操作（如等 I/O）可以挂起而不占线程
- ✅ 理论上能表达更复杂的依赖结构
- ❌ **写对非常难**：栈 / TLS / 锁在 fiber 切换时的行为容易踩坑
- ❌ 调度器实现复杂，调试器通常不支持 fiber 栈追踪
- ❌ 库少，多数工作室自己写

代表：Naughty Dog（《神秘海域》、《最后生还者》）、部分 id Tech 实现。

## 1.3 三种架构的本质区别

| 维度 | Thread-per-subsystem | Task-based | Fiber-based |
|------|---------------------|-----------|-------------|
| 粒度 | 子系统级（粗） | 函数级（中） | 语句块级（细） |
| 可中断 | 否（线程级抢占由 OS） | **否**（task 原子执行） | **是**（显式 yield） |
| 调度单位 | 线程 | task | fiber |
| 库成熟度 | 不需要库 | 成熟、多选 | 几乎没有通用库 |
| 调试 | 容易 | 容易 | 困难 |
| 负载均衡 | 差 | 好 | 最好 |
| 实现复杂度 | 低 | 中 | 高 |

**一句话记住两个关键判据**：

1. Task **不能中断**，fiber **可以中断** —— 这是最本质的区别
2. 不能中断带来的代价是：**遇到阻塞只能占着线程等**，或者把阻塞任务挂到专用线程（书里的 pinned task 异步 I/O 就是这个思路）

## 1.4 为什么主流是 task，不是 fiber

即使 fiber 理论上更强，主流选择仍然是 task，原因有三：

1. **简单性原则**：fiber 的诡异 bug（跨 fiber 的锁、TLS 混乱、析构顺序）会吃掉它带来的性能优势
2. **库生态**：task scheduler 有现成好用的（enkiTS / TaskFlow），fiber 你得自己写
3. **现代硬件的改进**：OS 调度、锁实现、cache coherency 都改善了，task 的粒度对大部分工作负载已经够细

除非你的工作流 **天然充满阻塞点**（比如大量 I/O、网络、等 GPU 结果），fiber 的收益才明显 —— 这也是 Naughty Dog 选 fiber 的理由（他们的 gameplay 逻辑很多"等触发器"的场景）。

## 1.5 业界案例

| 工作室 / 引擎 | 架构 | 备注 |
|-------------|-----|------|
| **Bungie (Destiny)** | Task-based | 2015 GDC 公开分享，教科书案例 |
| **Epic (Unreal 4+)** | Task-based（主流），保留少量专用线程 | Game / Render / RHI 线程仍在，task graph 处理其他 |
| **Unity** | Task-based (Unity Job System / Burst) | 现代 DOTS 架构 |
| **id Tech 6+** | Task-based（部分 fiber 化） | 持续演进 |
| **Naughty Dog** | **Fiber-based** | 从《神秘海域 4》起全面 fiber 化 |
| **Frostbite / Decima / Anvil** | Task-based | 有各自定制调度器 |

主流选 task-based。Naughty Dog 是"工作室规模 + 工作流特性"的特例，不是普适范式。

## 1.6 Work-stealing queue —— task 调度的关键机制

现代 task scheduler 的核心数据结构是 **work-stealing queue**：

- 每个 worker thread 有**自己的本地队列**
- 新 spawn 的子 task 默认塞进"我自己的"队列（局部性好、无锁）
- 本地队列空了，就从"邻居" worker 的队列**末端偷一个**（cache 友好 + 负载均衡）

这样：

- **无竞争路径**：大部分 task 在自己的队列里来去，不需要全局锁
- **自动负载均衡**：忙的 worker 被偷走工作，闲的 worker 自己找活
- **局部性好**：spawn 出的子任务通常还在同一个核的 L1/L2 cache 里

enkiTS / TaskFlow / TBB / Rust 的 Rayon 都是这个思路。这是理解"为什么 task-based 能扩展到 16 核"的关键。

## 1.7 选型结论：LX Engine 选 task-based + enkiTS

基于以上分析，对 LX Engine 未来的多线程架构，**选 task-based + enkiTS**。理由：

- **简单性**：fiber 的坑不适合单人 / 小团队维护
- **库成熟**：enkiTS 轻量（~3k 行 C++）、单头文件、跨平台、MIT 许可
- **和书对齐**：书里的 Raptor Engine 也用 enkiTS，学习材料直接可用
- **够用**：LX 目前目标没到 Naughty Dog 级的 fiber 复杂度
- **可替换**：如果将来要切到 TaskFlow / TBB / 自家 scheduler，task 的抽象不变，迁移成本可控

具体 enkiTS 怎么用、怎么配合 Vulkan 做异步 I/O，见 [02](02-enkiTS与异步IO模式.md)。
