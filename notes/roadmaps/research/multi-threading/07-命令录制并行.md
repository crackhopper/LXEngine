# 07 · 命令录制并行

> 多线程渲染的第二条主线。[02](02-enkiTS与异步IO模式.md) / [05](05-enkiTS与Vulkan-queue的分工.md) 讲的是 **资源加载并行**，这一章讲 **命令录制并行**。
>
> 前置阅读：[06 · CPU-GPU 分层架构草案](06-CPU-GPU分层架构草案.md)（`GpuJob` 抽象、FrameGraph 是生产者）。
>
> 核心洞察：**命令录制并行分三个层级 —— 帧并行 / pass 并行 / pass 内并行**，三层都有 task。

## 7.1 问题定位：CPU 录制并行，不是 GPU 多线程

书里这一节讲的是怎么 **并行录制 command buffer**。但很容易误解 —— 它**不是让 GPU 多线程**。

```
CPU 侧                             GPU 侧
────────────────                   ────────────────
worker 1 → 录一段命令    ┐
worker 2 → 录一段命令    ├──►  主线程按顺序汇总   ──► GPU 按提交顺序执行
worker 3 → 录一段命令    ┘                          （看到的是单命令流）
```

- **并行的是**：CPU 录制 command buffer 的时间
- **不并行的是**：GPU 执行命令的顺序 —— GPU 仍然按提交 order 串行执行（除非走多 queue）

一句话：**让 CPU 更快喂 GPU，而不是让 GPU 自己多线程**。

## 7.2 Vulkan 的硬约束：pool 必须外部同步

Vulkan 里 CommandPool / DescriptorPool / QueryPool 都有同一条规则：**同时只能被一个线程访问**。最简方案：**每条录制线程绑定一个私有 pool**。

```
Thread A ↔ CommandPool A
Thread B ↔ CommandPool B
Thread C ↔ CommandPool C
```

这样每个 worker 从自己的 pool 分配 command buffer，不会互相踩。

这和 [01 · 概念与选型](01-概念与选型.md) 的 task-based 心智一致：
- enkiTS 决定"谁来干活"
- Vulkan pool 约束"干活时不能共享哪些底层对象"

## 7.3 为什么要 F × T 个 pool

书里的分配策略是 **F × T**：

- **F** = in-flight frame 数（通常 2 或 3）
- **T** = 允许录制命令的线程数

两个维度都要隔离：

| 隔离维度 | 原因 |
|---------|------|
| **线程隔离** | 同一时刻两个线程不能抢同一个 pool |
| **帧隔离** | 上一轮的 command buffer 可能还在 GPU 里执行，pool 不能立刻 reset |

所以：

```
frame 0: [thread 0 pool] [thread 1 pool] [thread 2 pool] ...
frame 1: [thread 0 pool] [thread 1 pool] [thread 2 pool] ...
frame 2: [thread 0 pool] [thread 1 pool] [thread 2 pool] ...
                                                              ↑
                                按 (frameIndex, threadIndex) 找 pool
```

这本质上是"**线程局部 + 帧局部**"的资源分配器。

## 7.4 Reset pool 而不是单独 free command buffer

每帧开始时，把对应帧的所有 pool 整批 reset：

```cpp
void CommandBufferManager::reset_pools(u32 frame_index) {
    for (u32 i = 0; i < num_pools_per_frame; ++i) {
        const u32 pool_index = pool_from_indices(frame_index, i);
        vkResetCommandPool(vulkan_device, vulkan_command_pools[pool_index], 0);
    }
}
```

原因：

- `vkFreeCommandBuffers` 一个个 free / realloc，成本高
- `vkResetCommandPool` 批量 reset，让驱动复用底层内存
- 语义上不是真的"释放"，而是"把 pool 里所有 command buffer 回到初始态"

所以每帧开始时一次 reset，整帧内部的 command buffer 直接重录。

## 7.5 Primary vs Secondary command buffer

Vulkan 独有的一个区分：

| 维度 | Primary | Secondary |
|------|---------|----------|
| 能做什么 | 所有命令（draw / compute / copy） | **只能在 render pass 内录 draw 命令** |
| 能提交到 queue？ | ✅ 可以 | ❌ 不行，必须被 primary 用 `vkCmdExecuteCommands` 插入 |
| 粒度 | 粗（pass / 帧） | 细（pass 内部的 draw chunk） |
| 继承什么 | 无继承（自包含） | 继承 primary 的 `renderPass` 和 `framebuffer` |
| 起状态 | 全新 | 每次都要重新 `vkCmdSetViewport` / `vkCmdSetScissor` |

使用时机：

- **Primary 适合**：组织整段流程、整个 pass 作为一整块录制
- **Secondary 适合**：单个 render pass 内 draw call 很多（GBuffer / Forward 大场景），想在 pass 内进一步拆并行

Secondary 的 magic 在于：

```cpp
vkCmdBeginRenderPass(primary, ..., VK_SUBPASS_CONTENTS_SECONDARY_COMMAND_BUFFERS);
// 多个 worker 并行录 secondary buffer（都继承 primary 的 renderPass/framebuffer）
// 主线程汇总：
vkCmdExecuteCommands(primary, count, secondaryBuffers);
vkCmdEndRenderPass(primary);
```

这样一个"大 pass"的 draw call 就被拆到多个 worker 线程录。

## 7.6 ⭐ 录制并行的三个层级

这是这一章最关键的洞察：**命令录制并行实际上分三层，每一层都有 task**。

```
层级 1：帧级并行（CPU / GPU pipeline）
  └── Frame N-2 在 GPU 执行
      Frame N-1 在 present
      Frame N   CPU 在录制
      Frame N+1 CPU 在做 pre-record 准备

层级 2：pass 级并行（同一帧内）
  └── FrameGraph DAG:
        ShadowPass       ┐
        GBufferPass      ├──> LightingPass ──> PostProcessPass
        DepthPrepassPass ┘

        (没有依赖关系的 pass 可以并行)

层级 3：pass 内并行（单个 pass 内部）
  └── GBufferPass 里有 1000 个 draw call:
        draw [0..250]   → secondary cb 0
        draw [250..500] → secondary cb 1
        draw [500..750] → secondary cb 2
        draw [750..1000]→ secondary cb 3

        (主线程 vkCmdExecuteCommands 汇总到 primary)
```

### 每一层的 task 粒度

| 层级 | Task 类型 | 产出 | 并行度 |
|------|----------|------|-------|
| 帧级 | FrameTask | 整帧的命令 + present | ≤ maxFramesInFlight (2-3) |
| pass 级 | PassRecordingTask | 一个 primary command buffer | 看 DAG 可并行 pass 数 |
| pass 内 | SecondaryRecordingTask | 一个 secondary command buffer | 任意（看 pass 内 draw call 数） |

**三层都由 enkiTS 调度**。task 的产物是 `VkCommandBuffer`，不是直接"执行 GPU"。

## 7.7 架构师的原始心智模型

> "一个 pass 指令的录制，对应一个 primary command buffer。一帧由多个 pass 组成，不同 pass 之间有依赖关系，这里由 FrameGraph 定义。pass 层面本身也可以并行。这里在 core 层，每个 pass 会启动对应的 task。这样 task 调度器就会按照顺序来组织 pass 的执行。一个 pass 具体的执行，封装到 backend 中，不过这里还能进一步并行，pass 内部如果由可以拆分的，就用 secondary command buffer，随后使用任务调度器创建并行的任务，同时进行录制。录制结束后，由当前的 pass 执行器插入到对应的位置里。"

大方向正确，但有 **三处需要校正**，不然后面实现时会把调度边界搞混。

## 7.8 三处关键校正

### 校正 1：task scheduler 不是"顺序执行器"

❌ 不对：**task scheduler 按顺序组织 pass 执行**
✅ 正确：**FrameGraph 定义依赖关系，scheduler 在依赖满足时并行执行**

```
顺序不来自 enkiTS，来自 FrameGraph DAG。
enkiTS 只是消费这张依赖图，做"依赖满足后的并行执行器"。
```

具体表现：

- 如果 B 依赖 A，A 完成前 B 不能开始
- 如果 C 不依赖 A/B，C 可以与 A/B 并行

这和 [06 §6.5](06-CPU-GPU分层架构草案.md#65-最小接口草案) 的 `addDependency(parent, child)` 吻合。

### 校正 2："一 pass 一 primary" 是**策略**，不是硬规则

你最初的模型把它当成了必然关系，实际上：

| 可能的对应关系 | 适用场景 |
|-------------|---------|
| 1 pass = 1 primary | ✅ 最清晰的第一版实现策略（推荐） |
| 1 pass = 1 primary + N secondary | pass 内 draw call 多，需要进一步拆 |
| N pass = 1 primary | 工程上不常见但合法；小 pass 合批可以省开销 |

**建议：LX 第一版就用 "1 pass = 1 primary"，清楚、对齐 FrameGraph、好调试**。

### 校正 3：secondary 的适用条件比想象的**更窄**

❌ 不对：**pass 内部如果可以拆分，就用 secondary**
✅ 正确：**只有"同一 render pass 内有大量 draw recording"才值得 secondary 化**

Secondary 适合的场景：

- GBuffer pass 有大量 mesh draw
- Forward pass 有大量独立 draw item
- shadow map 某一层内 draw 很多

Secondary **不适合** 的场景：

- 一个 copy 命令（primary 直接做）
- 短的 compute dispatch（primary 直接做）
- 很少量 draw call 的 pass（拆 secondary 反而有 overhead）

所以**判断**：只有录制成本足够高、且能在同一 render pass 内分块时，才值得 secondary。否则 primary 就够。

## 7.9 多帧并行：不是"更多 task"，是"帧 slot 流水线"

你原本描述："多帧并行则每个帧有自己的一套资源集……有多少 frameInFlight，task 就是多少个"。

这句**方向对**，但**重点要换**：核心不是"任务数"，而是 **"资源生命周期"**。

### 正确的心智模型

```
时间线：

Frame N-2 : GPU 正在执行
Frame N-1 : 已提交，等 present
Frame N   : CPU 正在录制
Frame N+1 : CPU 在做 pre-record 准备（culling / queue build）
```

`maxFramesInFlight` 不是 "task 的上限"，而是 **"GPU frame-local 资源能同时在飞的上限"**。

### 两类工作的区分

| 类别 | 示例 | 是否受 frame slot 约束 |
|------|------|------------------|
| **CPU 逻辑任务** | scene update / culling / build draw list / pass packet 构造 | ❌ 不受约束，可以提前跑 |
| **GPU frame-local 任务** | 分配该帧的 cmd pool / 写 descriptor arena / 录 command buffer / submit | ✅ 必须拿到可用 frame slot |

也就是说 CPU 端可以"**跑在前面**"，但 GPU frame-local 的工作必须等到 slot 空闲。

### 真正被 fence 保护的资源

- 该 frame slot 的 command pool
- descriptor pool / transient allocations
- per-frame uniform / upload arena
- swapchain 同步对象

这就是你说的"每帧有自己的一套资源集，会锁定，完成后再释放" —— 但强调 **这是 slot 级别的"池"，不是每帧一个完整独立的资源宇宙**。

## 7.10 Task × Vulkan 的映射

整合 [05 · enkiTS 与 Vulkan queue 的分工](05-enkiTS与Vulkan-queue的分工.md) 和本章，把两条并行主线放在一张图里：

```
enkiTS task scheduler
│
├── 长 / 阻塞型任务（资源加载主线）
│   └── 文件 I/O → stbi 解码 → staging 写入 → transfer queue submit
│       (对应 pinned I/O thread + transfer queue)
│
└── 短 / 每帧任务（命令录制主线）
    ├── 帧级 task       (Frame N 的整体编排)
    ├── pass 级 task    (每个 ready pass 一个)
    └── pass 内 task    (重 pass 内部的 secondary 录制)

各类 task 最终都落到 Vulkan 上：
│
Vulkan backend
├── transfer queue    （异步加载用）
├── graphics queue    （命令录制的 primary 最终提交到这里）
├── per-thread per-frame command pools    （F × T 组）
└── fence / semaphore / barrier    （跨 task / 跨 queue 同步）
```

**两条主线共享**：
- 同一个 `ITaskScheduler`（enkiTS 实现）
- 同一个 `IGpuJobOrchestrator`（[06](06-CPU-GPU分层架构草案.md) 的抽象）
- 同一个 fence/semaphore 对象池

## 7.11 对 LX Engine 的渐进实施路径

（对应 [04 · 演进路径](04-演进路径.md) 的阶段 3，更细化到录制并行）

### 阶段 1 · 基础设施：frame × thread 的 command pool 池化

当前 `src/backend/vulkan/details/commands/command_buffer_manager.cpp` 只按 frame 分 pool。

改造为：

```cpp
// F × T 个 pool
VkCommandPool pools[MAX_FRAMES_IN_FLIGHT][MAX_RECORDING_THREADS];

VkCommandPool& getPool(u32 frameIdx, u32 threadIdx) {
    return pools[frameIdx][threadIdx];
}

void resetFramePools(u32 frameIdx) {
    for (u32 t = 0; t < MAX_RECORDING_THREADS; ++t) {
        vkResetCommandPool(device, pools[frameIdx][t], 0);
    }
}
```

这一步**不启用并行**，只是把 pool 结构准备好。单线程继续跑没问题。

### 阶段 2 · 按 pass 起 primary 录制 task（保守版）

FrameGraph 每个 ready pass 投一个 `PassRecordingTask`：

```cpp
for (auto& pass : readyPasses) {
    scheduler.addTask([&pass](TaskContext& ctx) {
        auto& pool = cmdMgr.getPool(frameIdx, ctx.threadId());
        VkCommandBuffer cb = allocFromPool(pool);
        vkBeginCommandBuffer(cb, ...);
        pass.record(cb);  // 录这个 pass 的所有命令到 primary
        vkEndCommandBuffer(cb);
        pass.recordedCb = cb;
    });
}
scheduler.waitAll(readyPasses);

// 主线程按 DAG 顺序收集并提交
for (auto& pass : orderedPasses) {
    submitList.push_back(pass.recordedCb);
}
vkQueueSubmit(graphicsQueue, ...);
```

这一步收益：**多个独立 pass 可以并行录制**。

### 阶段 3 · 重 pass 拆 secondary（只给 draw-heavy pass）

识别 "draw call > 阈值" 的 pass，把它拆成 N 个 secondary：

```cpp
// 主线程：开 primary + render pass
vkCmdBeginRenderPass(primary, ..., VK_SUBPASS_CONTENTS_SECONDARY_COMMAND_BUFFERS);

// 投 N 个 secondary 录制 task
for (u32 i = 0; i < N; ++i) {
    scheduler.addTask([&, i](TaskContext& ctx) {
        auto& pool = cmdMgr.getPool(frameIdx, ctx.threadId());
        VkCommandBuffer sec = allocSecondaryFromPool(pool);
        vkBeginCommandBuffer(sec, ...);
        recordDrawChunk(sec, drawItems[i*chunk .. (i+1)*chunk]);
        vkEndCommandBuffer(sec);
    });
}
scheduler.waitAll();

// 主线程汇总
vkCmdExecuteCommands(primary, N, secondaryBuffers);
vkCmdEndRenderPass(primary);
```

**只在重 pass 上做**。第一版不要每个 pass 都拆。

### 阶段 4 · 保持 submit 集中在主线程

```
worker threads: 并行录制（可以发散）
main thread:    集中 submit（必须收敛）
```

理由：

- `vkQueueSubmit` 同一 queue 不能跨线程并发（Vulkan 硬约束）
- submit 顺序决定 GPU 执行顺序，主线程控制最稳
- 第一版不要追"并行 submit"，先把并行 recording 做对

## 7.12 并行录制 ≠ 并行提交

特别强调的一条纪律：

| | 并行录制 | 并行提交 |
|---|---------|---------|
| 是什么 | 多个 worker 同时写 command buffer | 多个线程同时调 `vkQueueSubmit` |
| Vulkan 约束 | 只要 pool 不共享就合法 | 同一 queue 不能跨线程并发 |
| 收益 | 大、明显（CPU 录制成本降低） | 小、复杂（要分 queue 才有意义） |
| 复杂度 | 中 | 高（queue 间 semaphore 依赖） |
| 第一版是否做 | ✅ 做 | ❌ 不做 |

第一版 **"并行 recording + 集中 submit"** 是最稳的起点。

## 7.13 总结：一帧完整渲染的并行模型

把这一章所有要点压成一张图：

```
Frame N 开始
  │
  ├── [阶段 1] FrameGraph.compile()                 (主线程)
  │            产出 pass DAG 和 GpuJob 描述
  │
  ├── [阶段 2] resetFramePools(frameIdx)             (主线程)
  │
  ├── [阶段 3] 按 DAG 并行录制                       (多 worker)
  │      ├── readyPasses: 投 PassRecordingTask
  │      ├── heavy pass: 内部再投 SecondaryTask
  │      └── 每个 task 用 (frameIdx, threadIdx) 拿 pool
  │
  ├── [阶段 4] 汇总                                  (主线程)
  │      ├── 按 DAG 顺序取 primary cb
  │      └── secondary cb 已通过 vkCmdExecuteCommands 插入
  │
  ├── [阶段 5] 集中 submit                           (主线程)
  │      └── vkQueueSubmit(graphicsQueue, primaryList, ...)
  │
  └── [阶段 6] present + 推进 frame slot
```

**加上异步加载那条主线**（来自 [05](05-enkiTS与Vulkan-queue的分工.md)），一个引擎的完整多线程画面就有了。

## 7.14 一句话

**命令录制并行分三层：帧（frame slot 流水线）、pass（FrameGraph DAG 内并行）、pass 内（secondary command buffer）**。每一层都是 enkiTS task，task 产物是 `VkCommandBuffer`。**第一版只做并行 recording，submit 保持集中在主线程**；重 pass 才拆 secondary，不要一上来就每个 pass 都拆。
