# 03 · LX Engine 当前实现

> 直白的现状报告：LX Engine 目前 **基本是单线程引擎**，没有 scheduler、没有 worker 池、没有 task 图。
> 这一章说明我们现在在哪，以及为什么这是当前合理状态。

## 3.1 一句话定位

**LX Engine 当前是一个纯单线程引擎**。主线程同时承担：

- 渲染 command buffer 录制与提交
- Scene update / transform 计算
- 资源加载（shader 编译、texture / mesh 读盘）
- 输入事件循环
- Gameplay 逻辑（未来 Phase 6）

没有任何子系统运行在独立线程上。

## 3.2 唯一的线程安全前置

全仓唯一使用并发原语的地方是 `GlobalStringTable`：

```cpp
// src/core/utils/string_table.hpp
#include <atomic>
#include <shared_mutex>

// ...
mutable std::shared_mutex   m_mutex;
std::atomic<uint32_t>       m_nextID;
```

原因：StringID 是跨子系统的基础设施，将来任何多线程改造都会首先触碰字符串驻留；提前给它装上 `shared_mutex` + `atomic`，避免未来改的时候污染大量代码。

**这不是"我们已经多线程了"的标志**，而是"**将来多线程时这一块肯定要安全，不如现在就写对**"的防御性设计。

## 3.3 帧循环的单线程结构

典型一帧目前大致是这样（pseudocode）：

```
main thread (唯一线程):
  pollInputEvents()              // SDL 事件
  update(dt):
    scene.updateTransforms()
    animation.tick()             // (Phase 4, 未做)
    physics.step()               // (Phase 5, 未做)
    gameplay.tick()              // (Phase 6, 未做)
  render(scene):
    frameGraph.build()
    pipelineCache.preload()       // 已实现，见 pipeline-cache/02
    recordCommandBuffers()        // 串行录制
    submitAndPresent()
```

所有 Vulkan 调用 —— 包括 `vkCmdBindPipeline`、`vkCmdDraw`、`vkQueueSubmit`、`vkQueuePresentKHR` —— 都在主线程。不存在"跨线程 Vulkan 资源"的约束问题，因为根本没有别的线程。

## 3.4 "能"和"不能"的现状

### 能做到的（单线程好处）

- ✅ **零同步开销**：没有锁、没有原子操作（除 StringTable）
- ✅ **调试直接**：stack trace 一眼能看到完整调用链
- ✅ **data race 不存在**：任何数据读写都是同步的
- ✅ **资源生命周期好管**：不需要考虑"是否有另一个线程还在用"
- ✅ **测试简单**：单元测试 / 集成测试不需要 thread fixture

### 做不到的（单线程限制）

- ❌ **资源加载卡主线程**：加载 100 MB 的 GLTF 会冻帧几百 ms
- ❌ **Shader 编译卡主线程**：SPIRV-Cross + SPIR-V 编译串行，首帧 stutter
- ❌ **Pipeline 创建卡主线程**：驱动编译 ISA（见 pipeline-cache/03）时一动不动
- ❌ **用不上多核**：14 核 CPU 跑到 1 核 100% + 13 核 idle
- ❌ **AI agent 的后台计算挤占 rendering**：将来 Phase 10 agent 跑模型推理时，主线程完全堵死

## 3.5 为什么现在这样是"合理的"

不是懒，是**当前阶段单线程性价比最高**：

1. **场景规模小**：demo 级别，几十个物体，主线程根本没满
2. **资产少**：加载耗时在 100ms 级别，用户可感但不致命
3. **功能在搭**：Phase 1 的 PBR / shadow / IBL 还没完成，现在引入多线程会把焦点分散
4. **架构在稳**：pipeline-key / pipeline-build-desc / render-signature 这些核心结构还在演进，过早引入锁会让每次重构都变贵
5. **调试成本低**：单线程下引擎问题排查是秒级，多线程后 data race / deadlock 是天级

**但这是"还没到需要的阶段"，不是"不需要"**。等 roadmap 走到 Phase 3 的资产管线、Phase 10 的 AI agent，单线程就撑不住了。

## 3.6 隐藏风险：单线程假设的"静默扩散"

当前最该警惕的，不是"多线程没做"，而是**代码里每天都在积累"只能在主线程调用"的隐式假设**：

- `VulkanResourceManager` 直接持有非线程安全容器
- `PipelineCache` 的 `unordered_map` 没有锁
- 资源加载直接阻塞调用，没有 "返回 future / 完成回调" 的概念
- Callback / observer 没考虑跨线程触发
- StringTable 之外所有单例都默认主线程独占

这些**不是 bug**，是合理的单线程设计。但将来每一条都是"改多线程时要付的债"。

**缓解策略**：在新增子系统时尽量保持"不与外部共享可变状态"的接口形状，将来包装成 task 的成本会低得多。参见 [04 · 演进路径](04-演进路径.md) 的"零成本预留"部分。

## 3.7 Vulkan backend 的多线程潜力

虽然引擎是单线程的，**Vulkan 本身已经准备好了多线程**：

| Vulkan 特性 | LX 当前是否用到 | 多线程切换成本 |
|------------|---------------|--------------|
| 多 queue（graphics + transfer + compute） | 单 graphics queue | 中 |
| 多 command buffer 并行录制 | 单 primary command buffer | 中 |
| 多 command pool（每线程一个） | 单 pool | 小 |
| Secondary command buffer（并行录制到 primary 下） | 未用 | 中 |
| Descriptor pool（UPDATE_AFTER_BIND） | 非 bindless，按材质分 | 大（见 bindless-texture） |

换句话说，**Vulkan 这一层是全准备好的，限制来自引擎侧的抽象**。

## 3.8 总结表

| 维度 | LX 当前 | 成熟引擎常见 | 差距 |
|------|---------|------------|------|
| Scheduler | 无 | enkiTS / TBB / 自研 | 🔴 大 |
| Worker pool | 无 | 4-16 线程 | 🔴 大 |
| 资源加载 | 同步阻塞 | 异步 pinned task | 🔴 大 |
| Shader 编译 | 同步 | 后台 task | 🟡 中 |
| Pipeline 创建 | 主线程 | 后台 task + pipeline cache | 🟡 中 |
| Command buffer 录制 | 单 primary, 主线程 | 每 thread 一个 secondary | 🟡 中 |
| Scene update | 主线程 | 按子系统 parallel for | 🟡 中 |
| StringTable | 已 thread-safe | — | 🟢 已对齐 |
