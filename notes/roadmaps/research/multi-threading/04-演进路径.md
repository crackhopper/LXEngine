# 04 · 演进路径

> 结论先行：**不急着做，但从现在起写新子系统时就要按"将来能并行"的约束走**。
> 真正要启动是在 Phase 3 资产管线 / Phase 10 AI agent 之前。

## 4.1 核心原则：零成本预留 > 一次性大重构

多线程最容易出的错是"等要用了再改"。那时候全引擎到处是单线程假设，改一处牵一片。所以：

- **现在**：把接口设计成"不假设单线程调用者"，即使内部仍然是单线程实现
- **中期**：补一层 scheduler + worker 池，按场景激活少数 task
- **后期**：全面 task 化，包含 command buffer 并行录制

## 4.2 触发条件（什么时候真正启动）

任意一条出现，就说明单线程够不了了：

1. **资产加载卡顿**：关卡切换 / scene 初始化可感知冻帧（> 200 ms）
2. **Shader 编译 stutter**：首帧 / 材质切换肉眼可见卡顿
3. **Phase 3 资产管线上线**：异步加载成为硬需求
4. **Phase 10 AI agent 启动**：agent 推理不能挤占渲染
5. **场景规模扩大**：物体数破千，transform / culling 单线程跑不过 60fps
6. **命令录制超 3 ms**：单 primary command buffer 录制时间成瓶颈

## 4.3 分阶段路径

### 阶段 0 · 零成本预留（**现在就做，无重构**）

这些不是"多线程工作"，是 **"写新代码时的习惯"**：

- 新增子系统时，公开接口**不假设主线程调用**（注释或 static_assert 标记）
- 新增资源加载接口时，**优先设计成"返回 future / 接收 callback"**，即使当前实现是同步的
  - 例：`TextureLoader::loadAsync(path) → TextureHandle`（未来返回时机通过 callback 通知，现在就阻塞同步完成）
- 新增缓存 / 注册表时，**数据结构选择上预留并发访问能力**（例：`unordered_map` + 接口层 mutex 占位）
- Scene 相关的 mutable state **集中在少数对象里**，便于将来加 single-writer/multi-reader

预期产出：**未来多线程改造时，改动面小**；短期 **性能几乎不变**（同步实现，0 额外开销）。

### 阶段 1 · 接入 enkiTS + 资源异步加载（**Phase 3 触发**）

- 引入 [enkiTS](https://github.com/dougbinks/enkiTS) 作为 `src/infra/external/` 下的 submodule
- 建立 `LX_infra::TaskScheduler` 单例，启动时 `Initialize(numTaskThreadsToCreate = N)`
  - N 策略：`std::thread::hardware_concurrency() - 2`（主线程 + I/O 线程预留）
- 最后一个 worker 专用于 I/O：按 [02 §2.3](02-enkiTS与异步IO模式.md#23-pinned-task异步-io-专用线程) 挂双 pinned task
- `AsyncLoader` 类维护加载请求队列；主线程投递请求，I/O 线程消费
- 完成通知：task 结束后主线程在下一帧开始前 poll 完成队列，统一 `vkUpdateDescriptorSets`（若用 bindless）或注册资源
- Vulkan 资源上传走 **transfer queue**（如果硬件有独立 transfer queue）

预期产出：关卡加载不再卡主帧；用户层保持同步 API 也能 work。

### 阶段 2 · Pipeline / Shader 编译异步化（**与 bindless 演进同期**）

- `PipelineCache::preload` 拆成两步：**派发 N 个 task 并行创建，主线程 wait**
- Shader 编译（GLSL→SPIR-V、反射）也走 task
- 关键约束：`vkCreateGraphicsPipelines` 本身 **线程安全**，但要给每条线程自己的 `VkPipelineCache`（或用唯一共享的加 spinlock）
- 书里讲的 `VkPipelineCache` 磁盘持久化可在此阶段一并加上（见 `pipeline-cache/04`）

预期产出：scene 初始化从 "等 pipeline" 变成 "等最后一个 pipeline"，耗时降 1/N（N = worker 数）。

### 阶段 3 · Command buffer 并行录制（**性能触发**）

- 每线程一个 `VkCommandPool`
- 把 pass 切分：一批 draw 一条 secondary command buffer，并行录制
- 主线程只负责 **汇总 secondary 到 primary** 和 `vkQueueSubmit`
- 前提：pass 内的 draw 之间无依赖（典型：forward pass 本身、shadow pass）

预期产出：command buffer 录制从 `O(draws)` 单线程变成 `O(draws/N)`。

### 阶段 4（可选）· Scene / culling 并行化

- Transform / AABB 更新：parallel-for 分块
- Culling：视锥体剔除按对象数切片
- 物理、动画：各自独立 task，通过依赖图表达前后顺序

预期产出：game thread 时长大幅缩短，和 render thread 解耦更彻底。

## 4.4 需要同步改动的周边

| 子系统 | 改动 |
|-------|------|
| `VulkanDevice` | 持有独立 transfer queue（如硬件支持） |
| `VulkanResourceManager` | 容器加锁或拆成每线程实例 |
| `PipelineCache` | 查询 / 插入加 shared_mutex（或走每线程本地 + merge） |
| `ResourceHandle` 生命周期 | 引入"延迟销毁 + frames-in-flight"策略（和 bindless 同）|
| Log / profile | 线程安全、非阻塞、带 thread id |
| 构建系统 | 加 enkiTS submodule / cmake 链接 |

## 4.6 风险矩阵

| 风险 | 影响 | 应对 |
|-----|------|-----|
| **Data race 调试成本高** | debug 变慢、ship 质量降 | 强制 sanitizer（TSan）跑集成测试；关键路径打线程 assert |
| **死锁** | 帧卡死，复现困难 | 避免嵌套锁；锁获取顺序固定；timeout 加假死检测 |
| **Cache false sharing** | 吞吐 30~50% 损失不可见 | 关键结构体按 cache line 对齐（`alignas(64)`） |
| **Vulkan 外部同步违规** | validation layer 过了、实机崩 | 对每个 Vulkan 对象标注"线程约束"，review 时检查 |
| **跨线程生命周期错误** | 引用悬空 | 统一走 handle + 延迟销毁；禁止裸指针跨线程 |
| **不同 CPU 核数行为不一致** | 测试通过、用户环境崩 | numTaskThreads 支持环境变量 override 做压力测试 |
| **enkiTS 依赖变化** | 未来如果要换 scheduler | 在 `LX_infra::TaskScheduler` 包一层接口，把 enkiTS 当实现细节 |

## 4.7 需要明确禁止的事

即使多线程上了，以下事项 **仍然必须主线程**：

- SDL 事件 poll 和窗口管理
- `vkQueuePresentKHR`（swapchain 呈现）
- ImGui 主线程构建（ImGui 本身不线程安全）
- Scene graph 结构性改动（添加/删除节点）—— 跨线程改会炸
- Global state 初始化与销毁（`main` 开始 / 结束）

文档 / 断言里都要写明。

## 4.8 一句话

**不急着做，但从现在起写新子系统时就要按"将来能并行"的约束走**。真正启动在 Phase 3 / Phase 10 前夕，第一步是 **enkiTS + 资源异步加载（书里的模式）**，然后依次 pipeline 并行创建、command buffer 并行录制、scene / culling 并行化。核心原则是**早做接口预留，晚做内部实现**，避免将来大重构。
