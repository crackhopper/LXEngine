# 03 · LX 演进与触发条件

> 阅读前提：[01](01-VRS基础.md) + [02](02-specialization-constants.md) 讲完了 VRS 和 specialization constants。本文给出 LX 的实施位置 — *Phase 1 后期 / 性能 tuning 阶段*。

## 3.1 LX 当前状态对照

| 维度 | LX 现状 | 本调研需要 |
|------|---------|----------|
| `VK_KHR_fragment_shading_rate` 检测 | 无 | extension query |
| Per-draw rate API 封装 | 无 | `vkCmdSetFragmentShadingRateKHR` |
| Per-primitive rate（vertex/mesh shader 输出） | 无 | shader 输出扩展 |
| VRS image attachment | 无 | 新 attachment 类型 |
| Sobel edge detection compute pass | 无 | 新 compute pass |
| Rate image format（R8_UINT） | 无 | format 表加项 |
| Layout transition 到 `*_FRAGMENT_SHADING_RATE_*` | 无 | barrier path |
| Hardware fallback（不支持 VRS 时） | 无 | 接口可关 |
| SPIR-V reflection 识别 spec constants | 待查 | 扩展 reflector |
| Pipeline 创建接受 spec values | 无 | `VkSpecializationInfo` 路径 |
| Subgroup size 自适应 | 无 | 启动期 query + spec constant 填值 |
| Pipeline cache 按 spec value 区分 | 无 | cache key 扩展（可选） |

## 3.2 候选 REQ

只 2 个 REQ，相对独立：

### REQ-A · Specialization Constants 反射 + Pipeline 集成

**目标**：让 LX shader system 支持 SPIR-V specialization constants。

**改动**：

- 扩展 `ShaderReflector` 解析 `SpvOpSpecConstant*`
- `IShader` 暴露 spec constants 元数据
- Pipeline 创建路径接受 `VkSpecializationInfo`
- 启动期 query subgroup size 等硬件特性
- 至少一个 demo shader 用 spec constant（如 compute culling 的 SUBGROUP_SIZE）

**验收**：

- Compute shader 在 NVIDIA / AMD 上 thread group size 自动适配
- 单元测试：相同 SPIR-V + 不同 spec values 创建出不同 pipeline，行为正确

**风险**：低。

**前置依赖**：无（独立）。

### REQ-B · Variable Rate Shading

**目标**：实现 image-attachment 模式 VRS + Sobel edge detection。

**改动**：

- 启用 `VK_KHR_fragment_shading_rate`
- 资源类型加 `VRS_ATTACHMENT` 角色
- Sobel 边缘检测 compute pass（用 shared memory + barrier 协作模式）
- VRS image attachment 接入 dynamic rendering
- 跟 frame graph 集成（VRS image 是新 attachment 类型）
- Hardware fallback：不支持 VRS 时关闭，画质回归

**验收**：

- Profiling 显示 fragment shading 时间下降（典型 -30% ~ -50%）
- 视觉对比：开 / 关 VRS 画面差异 < 1% pixel diff（边缘检测正确判定）
- 不支持硬件上自动关闭，画面正常

**风险**：低 - 中。Algorithm 简单，但需要硬件兼容性测试。

**前置依赖**：

- 至少一个 *已存在的 deferred lighting 或后期 pass*（VRS 优化的对象）。早于 [REQ-119 G-Buffer](../../main-roadmap/phase-1-rendering-depth.md#req-119--g-buffer--延迟渲染路径) 立项没意义
- Compute pipeline 路径就位（async-compute REQ-A）

## 3.3 时序定位：为什么是后期

参照 LX Phase 1 路线图，本调研的 REQ 应该 *在所有以下条目完成之后* 才考虑：

```
Phase 1 主路径（需先完成）:
  ✓ REQ-118 PBR 完整管线
  ✓ REQ-103 Shadow pass + REQ-104 CSM
  ✓ REQ-119 G-Buffer / 延迟渲染
  ✓ REQ-109 多光源合同
  ✓ gpu-driven-rendering REQ-A..F（资产 + culling + indirect）
  
Phase 1 后期性能 tuning（本调研在这里）:
  ↓ 候选 REQ-A: Specialization Constants
  ↓ 候选 REQ-B: VRS
```

只有主路径全部就位，才有：

- 足够多的 shader 让 specialization 价值显现
- 足够重的 fragment shading 让 VRS 收益显现
- 稳定的 frame graph 让 VRS attachment 集成自然

提前做没收益。

## 3.4 触发条件

**全部满足时立项 REQ-A**：

- Shader 数量 > 30
- #define 变体管理已经成为 build 痛点
- 想做硬件自适应（subgroup size / 特性 toggle）

**全部满足时立项 REQ-B**：

- Phase 1 deferred + lighting 主路径完成
- Profiling 显示 fragment shading > 30% frame time
- 目标硬件覆盖支持 VRS（>= NVIDIA Pascal / AMD RDNA 2 / Intel Arc）

## 3.5 跟其它调研的耦合

弱耦合，几乎独立：

| 调研 | 耦合点 | 强度 |
|------|--------|------|
| **frame-graph** | VRS image 是新 attachment 类型，frame graph 表达 | 中 |
| **gpu-driven-rendering** | per-primitive VRS 跟 mesh shader 协同（可选）；spec constant 跟 subgroup size 联动 | 弱 |
| **shadows** | shadow pass 通常 *不用 VRS*（depth 不能降速率），无关 | 无 |
| **multi-threading / async-compute** | Sobel compute pass 走 compute pipeline | 弱 |
| **clustered lighting** | 可选：低重要性区域 rate 降低 | 弱 |

本调研 *不阻塞* 任何其它调研。是 *叠加层*。

## 3.6 风险与不确定性

### 风险 1：硬件碎片化

VRS 跨硬件支持差异大：

- NVIDIA Pascal+：完整支持
- AMD RDNA 2+：完整支持
- Intel Arc：支持
- 移动 GPU：基本不支持
- NVIDIA Pre-Pascal / AMD GCN：不支持

LX 部分用户（移动端 / 老硬件）拿不到 VRS 收益，必须 fallback 路径正常。

### 风险 2：MSAA 协同

VRS + MSAA 在 *边界处* 可能有 corner case（per-sample shading 行为）。不打算 v1 处理这个交叉，留 follow-up。

### 风险 3：Sobel 阈值调参

`G > 0.1` 这个阈值是经验值，*可能不适合所有场景*：

- 暗场景：阈值太高，过多 1×1
- 高对比场景：阈值太低，过多 2×2

可以做成 spec constant 让场景按需调，或自适应（按 luminance 分布算阈值）。v1 先固定，根据实际场景调。

## 3.7 一句话总结

**VRS + Specialization 是 *Phase 1 主路径完成后* 的性能 tuning 工具。两个 REQ 独立可做、跟其它调研耦合弱、风险低。在主架构稳定前不要立项。**
