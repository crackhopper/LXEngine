# 04 · 演进路径

> 结论先行：**LX `PipelineCache` 不删、不改 API；`VkPipelineCache` 作为"miss 之后的加速器"补进去。**

## 4.1 核心判断

从 [02](02-LX当前实现.md) 和 [03](03-书本方案与VkPipelineCache.md) 的分析出发：

> **`VkPipelineCache` 不能替代我们的 `PipelineCache`**。它不缓存"引擎对象身份"，只加速"驱动创建过程"。
> **最好的方案**是保留现有接口和对象缓存，**再在 miss 创建时叠加使用 `VkPipelineCache`**。

所以关系不是二选一，而是：

1. **LX `PipelineCache`** 先决定"要不要新建 `VulkanPipeline`"
2. 真要新建时，把 `VkPipelineCache` 传给 `vkCreateGraphicsPipelines`
3. 让驱动少做重复编译工作

**`VkPipelineCache` 本质上是"我们 cache miss 之后的加速器"**。

## 4.2 目标结构

```
应用请求 pipeline X
    ↓
PipelineCache::find / getOrCreate（按 PipelineKey 查引擎对象）
    ├── 命中 → 直接返回 VulkanPipeline（零开销，今天已经是这样）
    └── miss → vkCreateGraphicsPipelines(device, nativeVkCache, ...)
                    ↓
                    驱动内部尝试用 VkPipelineCache 跳过 SPIR-V→ISA 编译
                    ↓
                返回 VkPipeline → 包装成 VulkanPipeline → 放入 m_cache
```

一句话：**对外 API 不变，内部在 create 一线挂一个长命 `VkPipelineCache` 对象**。

## 4.3 分步改动清单

按"每一步都可独立落地、每一步都有收益"的原则：

### 阶段 A · 最小集成（进程内加速）

| 步骤 | 动作 | 文件 |
|------|------|------|
| 1 | `VulkanDevice` 或 backend 级别持有一个长命 `VkPipelineCache` 成员 | `src/backend/vulkan/vulkan_device.*` |
| 2 | `VulkanPipeline::create` / `createGraphicsPipeline` 接受或从 device 获取 `VkPipelineCache`，作为参数传给 `vkCreateGraphicsPipelines` | `src/backend/vulkan/details/pipelines/*` |
| 3 | 对外 API 完全不变，`PipelineCache::getOrCreate` / `preload` 不感知新改动 | `src/backend/vulkan/details/pipelines/pipeline_cache.*` |

阶段 A 收益：**进程内同一批 pipeline 的首次创建加速**（驱动可跨 pipeline 复用编译结果）。不改磁盘，不改启动流程。

### 阶段 B · 磁盘持久化（冷启动加速）

| 步骤 | 动作 |
|------|------|
| 4 | renderer 退出或合适时机（scene 切换、editor 保存）调 `vkGetPipelineCacheData` dump 出 blob |
| 5 | 写入磁盘：约定路径（`cache/pipeline_cache.bin`），带简单元信息 |
| 6 | 启动时读取 blob：**必须先校验 header**（`vendorID` / `deviceID` / `pipelineCacheUUID`），任一不匹配视为无缓存 |
| 7 | 校验通过才喂给 `VkPipelineCacheCreateInfo::pInitialData` |

阶段 B 收益：**冷启动加速、跨关卡加速**。首次启动不省，后续启动能跳过大部分驱动编译。

### 阶段 C（可选）· 更精细的管理

这些属于"有需要再做"：

- 多个 `VkPipelineCache`（按 pass 或按 shader 类别分组）
- 缓存大小上限管理（超了截断）
- 后台线程做 merge（`vkMergePipelineCaches`）
- 带版本号的磁盘格式（LX 自己的外层 header 嵌套驱动 header）
- Scene / editor 显式触发 flush

## 4.4 关键细节：可以"收敛"到什么程度

> 可以收敛的是**接口入口**，不是**底层职责**。

对外仍然只暴露：

```cpp
PipelineCache::find(...)
PipelineCache::getOrCreate(...)
PipelineCache::preload(...)
```

但内部拆成两层：

1. **对象缓存层** —— 现在的 `PipelineCache`（按 `PipelineKey` 查 `VulkanPipeline`）
2. **驱动缓存层** —— `VkPipelineCache` + 可选磁盘持久化

`VulkanResourceManager` / renderer / scene 等上游完全不知道第二层存在。它们只看到"miss 现在变快了"。

## 4.5 一张类比图帮助理解

把两层缓存类比成：

- **LX `PipelineCache`** ≈ 应用自己的 `unordered_map`
- **`VkPipelineCache`** ≈ 编译器自己的 object cache（`ccache` 对 `g++` 那种关系）

前者解决"我有没有这个对象"，后者解决"如果要重新编译，能不能快点"。两者并不互斥。

## 4.6 什么时候启动这个 roadmap

以下信号出现时再推动：

1. **Pipeline 数量上百**，首次启动/切换场景可感知 stutter
2. **启动时间 profile** 显示 `vkCreateGraphicsPipelines` 占比 > 10%
3. **AOT pipeline 预编译流程** 成型（例如 build 时扫一遍所有 material 变体）
4. **shader 变体爆炸** 发生（见 bindless-texture/04 的 ubershader 讨论）
5. **关卡切换** 成为常用场景，不能每次都卡几秒

当前 LX Engine pipeline 数量还少、启动时间不敏感，这件事**值得加，但不紧迫**。

## 4.7 风险与边界情况

| 风险 | 应对 |
|------|------|
| 驱动升级 / 换 GPU 导致缓存失效 | Header 校验，失效就当无缓存重新生成；不能假设 blob 永远有效 |
| 磁盘 blob 损坏 / 异常退出写了半截 | 先写临时文件再 rename；读失败就降级到无缓存 |
| 恶意篡改 blob | Vulkan 驱动**不保证**检查 blob 合法性，可能崩溃。对外分发场景要考虑签名或仅本地使用 |
| 缓存无限增长 | Vulkan 不承诺 `vkMergePipelineCaches` 会压缩；必要时自己截断或按 size 上限丢弃 |
| 跨用户复用 | 原则上可以跨用户（只要 GPU/驱动一致），但包含的编译结果可能含平台指纹，不建议当成"随产品分发"的东西 |

## 4.8 一句话

**侵入最小、收益最大的做法**：保留现有 `PipelineCache` 接口和对象缓存，在 `vkCreateGraphicsPipelines` 调用处叠加一个长命 `VkPipelineCache`，退出时持久化到磁盘，启动时带 header 校验的加载。对外 API 不动，引擎语义层继续由 `PipelineCache` 负责，驱动编译加速由 `VkPipelineCache` 负责，两层各司其职。
