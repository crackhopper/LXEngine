# 01 · 两种缓存的区别

> 在 Vulkan 上说"pipeline cache"，至少有两个完全不同的东西。搞清楚它们的边界是后续决策的前提。

## 1.1 Pipeline 创建的完整开销

先看"从零创建一个 `VkPipeline` "要经过什么：

```
SPIR-V 字节码（引擎层已经准备好）
    ↓
[ 驱动步骤 A ] 解析 SPIR-V、做 legalization、优化
    ↓
[ 驱动步骤 B ] 把 SPIR-V 编译成目标 GPU 的二进制指令（shader ISA）
    ↓
[ 驱动步骤 C ] 根据 pipeline state（blend / raster / depth 等）生成硬件寄存器配置
    ↓
[ 引擎步骤 D ] 把结果包进一个 VkPipeline 句柄，交还给应用
    ↓
[ 引擎步骤 E ] 应用把句柄放进某个 map，避免下次再调 vkCreate
```

**步骤 A+B+C 是"驱动的工作"，通常占总耗时 90%+**；步骤 D 几乎零开销；步骤 E 是应用自己的数据结构。

两种 cache 分别针对不同位置：

- **应用层对象缓存** 砍掉步骤 E 之后的"再来一次 D 前面所有步骤"
- **驱动层二进制缓存（`VkPipelineCache`）** 砍掉步骤 B 甚至一部分 A 的时间

## 1.2 应用层对象缓存（Object cache）

**在做什么**：把 `PipelineKey → VkPipeline` 的映射存在引擎内存里。相同 key 的请求直接返回已有句柄。

**典型代码**：

```cpp
VkPipeline& PipelineCache::getOrCreate(const PipelineBuildDesc& info, VkRenderPass rp) {
    auto it = m_cache.find(info.key);
    if (it != m_cache.end()) return *it->second;     // ← 命中：零成本

    auto pipeline = createVulkanPipeline(info, rp);  // ← miss：真正去 create
    m_cache[info.key] = std::move(pipeline);
    return *m_cache[info.key];
}
```

**生命周期**：和引擎进程同寿。进程一退出，map 消失。下次启动仍要全部重建。

**节省的时间**：**进程内相同 pipeline 的重复 create**。对于一帧 drawcall 万级、pipeline 实际只有几十种的场景，这个缓存每帧省下几十到几百次 `vkCreateGraphicsPipelines`。

**没能节省的时间**：**首次创建每个 pipeline 的驱动编译开销**。应用层 cache miss 时，你仍然要等驱动把 SPIR-V 编译成 GPU 指令。

## 1.3 驱动层二进制缓存（`VkPipelineCache`）

**在做什么**：Vulkan 原生对象，传给 `vkCreateGraphicsPipelines`。驱动在创建 pipeline 过程中把编译的中间/最终产物写进去，下次创建相似 pipeline 时复用。

**典型代码**：

```cpp
// 启动时：从磁盘读上次保存的 blob
VkPipelineCacheCreateInfo info{};
info.initialDataSize = diskBlob.size();
info.pInitialData = diskBlob.data();
vkCreatePipelineCache(device, &info, nullptr, &vkCache);

// 创建 pipeline 时传进去
vkCreateGraphicsPipelines(device, vkCache, 1, &pipelineInfo, nullptr, &pipeline);

// 退出前：dump 出来写回磁盘
size_t size;
vkGetPipelineCacheData(device, vkCache, &size, nullptr);
std::vector<uint8_t> blob(size);
vkGetPipelineCacheData(device, vkCache, &size, blob.data());
writeToDisk(blob);
```

**生命周期**：通过磁盘持久化，**跨进程、跨启动**。

**节省的时间**：**驱动的 SPIR-V → ISA 编译过程**。这部分开销在首次启动时可能占一个 pipeline 创建耗时的 80~95%。有磁盘缓存意味着 pipeline 首次创建走"我以前见过它，查表即可"的快路径。

**前置约束**：驱动 blob 格式**不是跨驱动/跨硬件兼容**的。所以书里讲的 header 校验很关键：

```cpp
struct VkPipelineCacheHeaderVersionOne {
    uint32_t headerSize;
    VkPipelineCacheHeaderVersion headerVersion;
    uint32_t vendorID;           // 必须匹配
    uint32_t deviceID;           // 必须匹配
    uint8_t  pipelineCacheUUID[VK_UUID_SIZE];  // 驱动升级就会变
};
```

四个字段都得和当前设备一致，否则 cache 直接作废、重新编译。驱动升级、换 GPU、换驱动厂商都会让缓存失效。

## 1.4 两种缓存的叠加关系

把两种缓存画在同一张图上：

```
应用请求 pipeline X
    ↓
[应用层对象缓存] ──命中──→ 直接返回 VkPipeline 句柄（零开销）
    ↓ miss
vkCreateGraphicsPipelines(device, vkPipelineCache, ..., &X)
    ↓
    [驱动层二进制缓存] ──命中──→ 跳过 SPIR-V→ISA 编译（省 80%+）
    ↓ miss
    完整走驱动 A+B+C 步骤（最慢路径）
    ↓
返回 VkPipeline，应用层 cache 记录
    ↓
dump VkPipelineCache blob → 磁盘（下次启动可复用）
```

两层**不互斥，是串联的**：

| 场景 | 应用层缓存 | 驱动层缓存 | 总体耗时 |
|------|-----------|-----------|---------|
| 同一进程第 N 次请求同一 pipeline | 命中 | 不查 | **近零** |
| 首次启动，无磁盘缓存 | miss | miss | **最慢**（驱动完整编译） |
| 次次启动，有磁盘缓存 | miss（进程新起的） | 命中 | **中等**（跳过编译） |
| 驱动升级后首次启动 | miss | UUID 不匹配、失效 | **最慢**（等同首次启动） |

所以想把 "**冷启动时间**" 也压下去，只有应用层缓存是不够的，必须叠加驱动层。

## 1.5 哪个更值得先做？

看你关心哪个指标：

- **帧内性能**：应用层对象缓存是**硬需求**。没有它每帧几百次 `vkCreate` 会直接卡死
- **冷启动时间**：驱动层二进制缓存是**优化项**。首次启动没省，后续启动省几秒到几十秒
- **关卡切换时间**：两者都有用，取决于你是否在关卡切换时销毁 pipeline

**LX Engine 的现状**：前者已实现，后者未接入。详见 [02 · LX Engine 当前实现](02-LX当前实现.md)。
