# 02 · LX Engine 当前实现

> 这一章盘点 LX Engine 现在 **已经做了什么**、做到了哪一层、以及为什么它不是 `VkPipelineCache` 的替代品。
> 相关权威文档：`openspec/specs/pipeline-cache/spec.md`、`notes/subsystems/pipeline-cache.md`。

## 2.1 一句话定位

**LX Engine 的 `PipelineCache`：解决"引擎知道自己有没有这条 pipeline"。**

换句话说，它是**引擎对象身份管理层**，不是驱动编译加速层。这个定位和 Vulkan 原生 `VkPipelineCache` **不是一回事**（后者详见 [03](03-书本方案与VkPipelineCache.md)）。

## 2.2 类、位置、接口

**类**：`LX_core::backend::PipelineCache`
**位置**：`src/backend/vulkan/details/pipelines/pipeline_cache.{hpp,cpp}`
**契约**：`openspec/specs/pipeline-cache/spec.md`

对外三个入口：

```cpp
class PipelineCache {
public:
    explicit PipelineCache(VulkanDevice& device);

    // 只查不建：miss 返回 nullopt，size 不变。
    std::optional<std::reference_wrapper<VulkanPipeline>>
        find(const PipelineKey& key) const;

    // Miss 则按 buildInfo 新建并缓存。
    // Preload 阶段以外的 miss 会打印 warn 日志（含 toDebugString）。
    VulkanPipeline& getOrCreate(const PipelineBuildDesc& info,
                                VkRenderPass renderPass);

    // 批量预构建：抑制 miss 警告，循环调用 getOrCreate。
    void preload(const std::vector<PipelineBuildDesc>& infos,
                 VkRenderPass renderPass);

    size_t size() const;

private:
    VulkanDevice& m_device;
    std::unordered_map<PipelineKey, VulkanPipelineUniquePtr, PipelineKey::Hash> m_cache;
    bool m_suppressMissWarning = false;
};
```

**数据结构**：`std::unordered_map<PipelineKey, VulkanPipelineUniquePtr>`，cache 本身持有 pipeline 生命周期。

## 2.3 与周边系统的接法

LX Engine 的 pipeline 创建流程是 **FrameGraph 驱动的 preload**：

```
Scene 初始化
    ↓
VulkanRenderer::initScene(scenePtr)
    ↓
构建 FrameGraph
    ↓
FrameGraph::collectAllPipelineBuildDescs()
    ↓
PipelineCache::preload(descs, renderPass)       ← 批量预构建
    ↓
后续 draw loop
    ↓
VulkanResourceManager::getOrCreateRenderPipeline(item)
    → PipelineCache::getOrCreate(...)           ← 理想情况下命中
    → 漏预构建 → miss 补建并 warn
```

几条关键约束：

- `find` **绝不会偷偷创建 pipeline**；构造只在 `getOrCreate` / `preload` 路径
- `VulkanResourceManager` **不持有 pipeline map**，它只是转发给 `PipelineCache`
- `preload` 期间 miss 是**正常流**，不打 warn；运行时 miss 是**异常信号**，要打 warn 并带上 `PipelineKey` 的人类可读字符串

## 2.4 `PipelineCache` 真正做的工作清单

这些是 Vulkan 原生 `VkPipelineCache` **不管**的事情：

| 能力 | 说明 |
|------|------|
| **PipelineKey 稳定身份** | 用结构化 `PipelineKey::build(objSig, matSig)` 唯一标识 pipeline，不依赖 SPIR-V 字节相同 |
| **find / getOrCreate / preload 三分** | "预期命中"和"允许现场创建"是两个独立 API，避免混淆 |
| **运行时 miss 诊断** | 打印 warn 日志 + toDebugString，立刻知道哪条 pipeline 预构建漏了 |
| **VulkanPipeline 生命周期** | cache 本身拥有 pipeline，析构时统一清理 |
| **避免上层对象重复** | descriptor layout / pipeline layout / shader module 等都靠命中跳过 |
| **FrameGraph 集成** | preload 入口直接对接 scene 初始化流程 |
| **失败行为契约化** | spec 里写明每个 scenario（cold miss / 二次调用 / preload 幂等） |

## 2.5 LX 当前做到的 vs 没做到的

**做到了（对应"引擎对象身份管理"）**：

- ✅ 同一进程内，同 `PipelineKey` 的 pipeline 只创建一次
- ✅ Scene 初始化时预构建，避免首帧抖动
- ✅ 漏预构建的 pipeline 可被 runtime miss 警告捕获
- ✅ `VulkanResourceManager` 清爽解耦

**没做到（对应"驱动编译加速"）**：

- ❌ 调 `vkCreateGraphicsPipelines` 时传的是 `VK_NULL_HANDLE`，**未使用 `VkPipelineCache`**
- ❌ 没有磁盘持久化 / 跨启动复用
- ❌ 驱动每次启动都要从 SPIR-V 编译成 ISA，首次创建耗时全量

换言之：**cache miss 时我们让驱动完整走一遍编译流程**，损失了 [03 · 书本方案](03-书本方案与VkPipelineCache.md) 讲的那部分优化。

## 2.6 为什么 `PipelineCache` 不能被 `VkPipelineCache` 替代

LX 的 `PipelineCache` **命中时**：

- 根本不会再调用 `vkCreateGraphicsPipelines`
- 直接返回已经建好的 `VulkanPipeline`

而 `VkPipelineCache` **命中时**：

- 你仍然要调用 `vkCreateGraphicsPipelines`
- 只是驱动可能编得更快

**这是本质区别**。两种缓存的目标、粒度、生命周期都不同：

| 层次 | 缓存内容 | key 是谁定义的 | 生命周期 |
|------|---------|--------------|---------|
| LX `PipelineCache` | `VulkanPipeline` 对象本身 | `PipelineKey`（引擎侧） | 引擎/renderer 生命周期 |
| `VkPipelineCache` | 驱动内部的编译中间结果 | 驱动内部（不可见） | 进程内，或序列化到磁盘跨启动 |

如果只留 `VkPipelineCache`，去掉 LX 的 `PipelineCache`，引擎会退化成：

- 每次 draw 或 preload 都得重新调 `vkCreateGraphicsPipelines`
- 只是驱动内部可能加速
- **引擎侧失去对象缓存与身份管理，也失去 miss 诊断、预构建契约等工程手段**

所以两者是**互补**的，不是二选一。详细演进建议见 [04 · 演进路径](04-演进路径.md)。
