# 05 · LX 当前状态对照

> 阅读前提：[01](01-gpu-driven-范式.md) ~ [04](04-clustered-deferred-lighting.md) 已经讲完 Ch6 + Ch7 的全部技术细节。本文综合盘点 LX gap。

## 5.1 LX 渲染管线现状速查

```cpp
// 当前主要渲染流程（简化）
VulkanRenderer::renderFrame() {
    // 1. CPU 端 culling
    queue.buildFromScene(scene, pass, target);  // 简单 frustum on CPU
    
    // 2. CPU 端逐 mesh 提交 draw
    for (auto& item : queue.getItems()) {
        bind_pipeline(item.pipelineKey);
        bind_descriptors(item.descriptorResources);
        vkCmdDrawIndexed(...);
    }
}
```

特征：

- 单 forward pass
- 单 swapchain RT
- 单光源（最多 2-3 个）
- CPU 决策一切
- 无 indirect / 无 GPU culling / 无 mesh shader / 无 G-buffer / 无 light culling

## 5.2 字段级 gap 总表

按 Ch6 + Ch7 涉及的所有维度盘点：

### 资产层（Ch6 §1）

| 维度 | LX 现状 | 需要 | gap |
|------|---------|------|-----|
| Mesh 数据结构 | `Mesh : IRenderable { vertexBuffer, indexBuffer }` | + Meshlet 数组 + bounds | 大 |
| meshoptimizer 依赖 | 无 | CMake 依赖 | 小 |
| Vertex 压缩格式 | 全精度 vec3/vec2 | normal s8 / UV f16 / position f32 | 中 |
| Triangle index | u32 | meshlet 内 u8 | 小 |
| Meshlet bounds | 无 | per-meshlet sphere + cone | 中 |

### 渲染管线类型（Ch6 §2-3）

| 维度 | LX 现状 | 需要 | gap |
|------|---------|------|-----|
| Compute pipeline | 无（async-compute REQ-A 范围） | compute shader + dispatch | 大 |
| Mesh shader pipeline | 无 | task + mesh + fragment | 大 |
| Storage buffer 角色 | 没单独建模 | `IGpuResource::StorageBuffer` | 中 |
| Indirect draw 命令 | 无 | `vkCmdDrawIndexedIndirectCount` | 中 |
| Dispatch 命令封装 | 无 | wrapper | 小 |

### Culling 算法（Ch6 §3）

| 维度 | LX 现状 | 需要 | gap |
|------|---------|------|-----|
| Compute culling | 无 | frustum + back-face cone shader | 中 |
| Hi-Z 生成 | 无 | mip chain + max-reduce compute | 中 |
| Sphere → AABB 投影 | 无 | shader 工具函数 | 小 |
| Two-phase rendering | 无 | frame graph 多 pass + 跨 pass 资源 | 大 |
| Hardware path 选择 | 无 | mesh shader vs compute fallback | 中 |

### Render target 与 Light（Ch7）

| 维度 | LX 现状 | 需要 | gap |
|------|---------|------|-----|
| Multi-RT 渲染 | 仅 swapchain 单 RT | G-buffer 4 RT | 大 |
| Octahedral encoding | 无 | shader 工具 + descriptor 协议 | 小 |
| `VK_KHR_dynamic_rendering` | 无（用老 API） | 强烈推荐切换 | 中 |
| Light depth sort | 无 | 每帧 CPU qsort | 小 |
| Light AABB → screen | 无 | math 工具 | 小 |
| Tile bitfield | 无 | GPU buffer + 上传协议 | 中 |
| Z bins | 无 | 同上 | 小 |
| Light culling shader | 无 | clustered lighting shader | 中 |

## 5.3 LX 已经做对的、能直接对接的部分

| LX 已有 | 在本调研路线里的作用 |
|---------|--------------------|
| `IGpuResource` 统一契约 | 未来 storage buffer / indirect buffer / Hi-Z 都走同一接口 |
| StringID + interning | mesh / pipeline / pass 名字稳定 |
| `RenderingItem` | 改造成 indirect draw 输入数据源 |
| `RenderQueue::buildFromScene` | 改造成 *写 indirect command buffer*（GPU 不再用它当 draw list） |
| `Scene` 扁平容器 | 提供 mesh / light 列表给 GPU upload |
| Material system + reflection | 跟 mesh shader / compute shader 共用反射框架 |

## 5.4 跟现有 REQ 的对照

LX roadmap 里跟本调研直接对应的 REQ：

| REQ | 内容 | 跟本调研关系 |
|-----|------|-------------|
| [REQ-119](../../main-roadmap/phase-1-rendering-depth.md#req-119--g-buffer--延迟渲染路径) G-Buffer | deferred 路径 | 直接对应 Ch7 G-buffer + clustered lighting |
| [REQ-109](../../main-roadmap/phase-1-rendering-depth.md#req-109--pointlight--spotlight--统一多光源合同) 多光源合同 | N=8 上限 | 本调研用 clustered lighting 取代上限 |
| [REQ-110](../../main-roadmap/phase-1-rendering-depth.md#req-110--frustum-culling) Frustum Culling | CPU AABB × 视锥 | 本调研用 GPU compute 取代 |
| [REQ-101](../../main-roadmap/phase-1-rendering-depth.md#req-101--hdr-scene-color-target) HDR scene color | offscreen RT | dynamic rendering 简化它 |
| [REQ-103](../../main-roadmap/phase-1-rendering-depth.md#req-103--shadow-pass--depth-only-pipeline) Shadow pass | depth-only | 跟 G-buffer 协同 |
| [REQ-118](../../main-roadmap/phase-1-rendering-depth.md#req-118--pbr-完整管线) PBR 完整管线 | forward 路径 | 跟 deferred 并存 |

**关键观察**：REQ-119、109、110 当前的 *实现细节都没定*。本调研的演进路径就是给它们提供具体实施方案。

## 5.5 跟前面三个调研的耦合点

### 跟 multi-threading/08

| 耦合点 | 强度 |
|------|------|
| Hi-Z 是跨帧资源（上一帧 → 本帧），需要 `RetirePoint` 跟踪生命周期 | 中 |
| Indirect command buffer 跨帧也通过 RetirePoint 复用 | 中 |
| 不直接需要多 queue（除非加 async culling） | 弱 |

### 跟 frame-graph 调研

| 耦合点 | 强度 |
|------|------|
| 两阶段 occlusion = 复杂多 pass DAG，强依赖 frame graph 资源依赖（REQ-035） | **强** |
| Hi-Z 跨帧资源需要 frame graph 表达"persistent resource" | 中 |
| 自动 barrier 推导（REQ-038）必须先就位 | **强** |
| G-buffer 4 RT 需要 RenderTargetDesc 支持 MRT（REQ-034 R2 已预留） | 中 |
| Dynamic rendering 跟 frame graph 自动 framebuffer 创建（REQ-037）协同 | 中 |

### 跟 async-compute 调研

| 耦合点 | 强度 |
|------|------|
| Culling compute pass 是典型 async candidate | **强** |
| Light culling compute pass 跟 G-buffer pass 可并行 | 中 |
| Hi-Z 生成 compute pass 也可 async | 中 |
| Compute pipeline 路径直接复用 async-compute REQ-A | **强** |

## 5.6 一句话定位

**本调研是 *消费方* 而不是 *生产方***：它不引入新的同步原语 / pipeline 抽象 / 资源模型，全部依赖前面三个调研已经设计好的能力。

但它 *组合方式独特*，所以单独立调研是合理的：

- multi-threading/08 解决 *同步* 问题
- frame-graph 解决 *依赖调度* 问题
- async-compute 解决 *多硬件引擎并行* 问题
- 本调研解决 *如何用前三者实现现代渲染管线* 的问题

## 5.7 LX 当前位置

按 [01 §1.4](01-gpu-driven-范式.md#14-gpu-driven-能力层级) 给的 GPU-driven 能力层级：

```
Level 0: 纯 CPU-driven                        ← LX 在这里
Level 1: + indirect draw（CPU 还做 culling）
Level 2: + GPU frustum culling
Level 3: + GPU occlusion culling (Hi-Z 单阶段)
Level 4: + 两阶段 Hi-Z occlusion              ← Ch6 教科书目标
Level 5: + meshlet 级别 culling               ← Ch6 完整体
Level 6: + mesh shader                        ← Ch6 NV/EXT 路径
Level 7: + 全 GPU-driven dispatch chain
```

要从 0 推到 4 需要本调研全部 8 个候选 REQ。从 4 推到 5/6 需要进一步扩展。
