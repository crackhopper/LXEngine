# 06 · 演进路径

> 阅读前提：[05](05-LX当前状态对照.md) 盘清 LX gap。本文给出推进路径切分。
>
> 注：本文给出的 REQ-A..H 编号是 *候选编号*，正式立项时按 `notes/requirements/` 实际可用号顺延。

## 6.1 总体策略

**结论先行：8 个候选 REQ 分三族**，每族内可并行，跨族有时序依赖。

- **基础设施族（A / B / C）**：纯 backend 能力，可以早做
- **Cross-vendor 主路径族（D / E / F）**：跨硬件兼容的核心路径
- **优化路径族（G / H）**：mesh shader / fallback，性能优化层

每个 REQ：

- 输入输出明确
- 工程编译可测
- 风险点单独立 REQ

## 6.2 八步演进序列

### REQ-A · Meshlet 资产 pipeline

**目标**：把 LX 的 Mesh 加载流程加上 meshlet 切分。

**改动**：

- CMake 加 `meshoptimizer` 依赖
- `Mesh` 类加 `vector<Meshlet>` + `vector<MeshletBounds>` + meshlet vertex/triangle indices buffer
- OBJ / glTF loader 调 `meshopt_buildMeshlets` 切分
- Vertex 压缩格式（normal s8 / UV f16 / position f32）
- **Octahedral encoding 工具函数**（跟 Ch7 G-buffer 共用）

**验收**：

- 加载一个 glTF mesh，打印 meshlet 数 + 平均 vertex/triangle 数
- 离线检查每个 meshlet 的 bounding sphere + cone 合理

**风险**：低。纯资产层工作。

**前置依赖**：无。

### REQ-B · Indirect draw + storage buffer 基础

**目标**：让 LX 能跑 indirect draw，准备好 GPU 写 draw command 的能力。

**改动**：

- 扩展 `IGpuResource` 加 `StorageBuffer` 角色（或确认已有可复用）
- 加 `vkCmdDrawIndexedIndirectCount` / `vkCmdDispatchIndirect` 命令封装
- 资源创建时支持 `STORAGE_BUFFER` + `INDIRECT_BUFFER` usage flag
- 简单 demo：CPU 写 indirect command buffer → GPU 用它绘制

**验收**：用 indirect 命令画一个三角形场景，CPU 不直接调 vkCmdDraw。

**风险**：低。

**前置依赖**：async-compute REQ-A（compute pipeline 基础设施，因为 indirect 经常配合 compute 写命令）。

### REQ-C · G-Buffer + Multi-RT + dynamic rendering

**目标**：把 LX 的渲染 pipeline 从单 RT 升级为多 RT，同时切到 `VK_KHR_dynamic_rendering`。

**改动**：

- 启用 `VK_KHR_dynamic_rendering`（Vulkan 1.3 已 core）
- 替换 backend 的 RenderPass / Framebuffer 创建为 dynamic rendering 调用
- `RenderTarget` binding 端持有 attachment image views（呼应 REQ-034）
- G-buffer demo：4 RT（albedo / normal / ORM / emissive）+ depth
- Octahedral normal encoding 在 G-buffer pass shader 里使用

**验收**：

- 跑一个 deferred demo，G-buffer 4 RT 全部正确写入
- 用 RenderDoc 验证 dynamic rendering 路径无 validation error

**风险**：中。是 LX 渲染管线第一次重大改造。

**前置依赖**：[`REQ-034`](../../../requirements/034-render-target-desc-and-target.md)（RenderTargetDesc 拆分）。

### REQ-D · Compute culling 基础（frustum + back-face）

**目标**：mesh / meshlet 级别的 frustum + back-face cone culling，写到 indirect draw command buffer。

**改动**：

- Compute shader: `cullMeshlets.comp.spv`
- 输入：meshlet bounds buffer + mesh instance buffer + camera
- 输出：通过的 meshlet → indirect draw command buffer
- 单 pass culling，不做 occlusion
- 跟 REQ-B 的 indirect draw 命令配合

**验收**：

- 视锥外物体不画
- 完全背向相机的 meshlet 不画
- 性能对比 vs CPU culling：10×+ 速度提升

**风险**：中。algorithm 不复杂，但跟 frame graph 接口要清晰。

**前置依赖**：REQ-A、REQ-B。

### REQ-E · Hi-Z + 两阶段 occlusion culling

**目标**：把 occlusion culling 推到主流路径，用两阶段 Aaltonen 算法。

**改动**：

- Hi-Z 生成 compute shader（max-reduce mip chain）
- 跨帧 Hi-Z 资源（multi-threading/08 RetirePoint 跟踪）
- Sphere → screen-space AABB 投影 shader 工具
- 选 Hi-Z level + occlusion test shader
- 两阶段 frame graph 配置（phase 1 + Hi-Z update + phase 2）
- False negative 列表数据结构

**验收**：

- 室内场景：被墙挡住的物体被剔
- 性能对比：被遮挡部分越多收益越大
- 验证 false negative 列表正确（特定相机转向场景）

**风险**：**高**。是本调研的核心风险点。

**配套要求**：

- 必须配 Vulkan validation layer 强测试
- 跨帧资源管理（multi-threading/08 + frame graph 协同）
- 跨硬件验证（NV / AMD / Intel）

**前置依赖**：

- REQ-D
- **frame-graph REQ-038（barrier 推导）必须先完成** — Hi-Z 跨 pass 资源依赖必须自动管理
- multi-threading/08 backend 实现就位

### REQ-F · Clustered lighting

**目标**：把 LX 的多光源合同从"N=8 上限"升级为 *无上限 clustered lighting*（200+ lights）。

**改动**：

- CPU 端：light depth sort、bins 构建、tile bitfield 构建
- 上传到 GPU 的协议（每帧 ~260 KB 数据）
- GPU 端 lighting compute shader 用 1D bin + 2D tile 算法
- 跟 G-buffer pass 配合（deferred 模式）
- 也支持 forward+ 路径（同样的 light culling，shader 在 forward 里查）

**验收**：

- 渲染 100+ 光源场景，性能可接受（vs 朴素方案 50×+ 加速）
- 验证 false positive / false negative 行为
- Forward+ 和 deferred 两条路径都可用

**风险**：中。algorithm 已成熟，工程实施细节多。

**前置依赖**：

- REQ-C（G-buffer / dynamic rendering）
- REQ-B（compute pipeline）
- 替代 [REQ-109](../../main-roadmap/phase-1-rendering-depth.md#req-109--pointlight--spotlight--统一多光源合同) 的"N=8 上限"实施细节

### REQ-G · Mesh shader pipeline + 实战

**目标**：在支持 mesh shader 的硬件上启用，性能优于 compute culling 路径。

**改动**：

- 扩展 Pipeline 抽象：第三种 type（task + mesh + fragment）
- Shader loader 支持 `*.task.spv` / `*.mesh.spv`
- Shader reflection 扩展支持 mesh / task stage（local group size / storage block）
- `vkCmdDrawMeshTasksIndirectCountNV` 或 EXT 版本封装
- Task shader 实现 cull filter（subgroup ballot 压紧）
- Mesh shader 实现 vertex/primitive emit
- 跟 REQ-A 资产层的 meshlet 数据对接

**验收**：

- mesh shader 路径渲染同样场景，画面跟 compute 路径像素一致
- 性能对比 vs compute 路径：5-30% 提升

**风险**：中。Vulkan extension 路径，跨硬件验证多。

**前置依赖**：REQ-A，可选依赖 D + E 让两条路径一致输出。

### REQ-H · Path selection + fallback

**目标**：启动期检测硬件，自动选 mesh shader 或 compute culling 路径。

**改动**：

- `VulkanDevice` 查 mesh shader extension 支持
- `Renderer` 持 `path_selector` 配置
- 调用方不感知（统一接口下挂两条 backend 实现）
- 启动 log 标注当前路径

**验收**：

- NVIDIA Pascal+：自动走 mesh shader
- Intel Arc / AMD RDNA 2+：自动走 mesh shader（用 EXT 版本）
- 老 GPU / 移动 GPU：自动走 compute culling
- 强制 override flag 可手工切换调试

**风险**：低。

**前置依赖**：REQ-D + REQ-G 都就位。

- G 等待 A
- H 等待 D + G

按 [README §"何时触发本调研立项"](README.md#何时触发本调研立项) 的判断：

强触发：

- [REQ-119 G-Buffer / 延迟渲染](../../main-roadmap/phase-1-rendering-depth.md#req-119--g-buffer--延迟渲染路径) 立项 → 触发 C + F

- 性能瓶颈分析显示 draw call CPU 时间 / shading 时间是 hotspot

中触发：

- [REQ-110 Frustum Culling](../../main-roadmap/phase-1-rendering-depth.md#req-110--frustum-culling) 立项 → 触发 D

## 6.5 中间里程碑

| 阶段 | 完成 REQ | 可交付 / Level |
|------|---------|----------------|
| M-GD.α | REQ-A + REQ-B | 资产能切 meshlet，indirect draw 跑通；Level 1 |
| M-GD.β | + REQ-C | G-buffer + dynamic rendering；deferred 路径打通 |
| M-GD.γ | + REQ-D | GPU compute culling 取代 CPU culling；Level 2 |
| M-GD.δ | + REQ-E | 两阶段 Hi-Z occlusion；Level 4（教科书目标） |
| M-GD.ε | + REQ-F | Clustered lighting；100+ 光源 real-time |
| M-GD.ζ | + REQ-G | mesh shader 路径在支持硬件启用；Level 6 |
| M-GD.η | + REQ-H | 自动 fallback；产品级跨硬件 |

每个里程碑都是独立可验证的 demo。

## 6.6 风险与不确定性

### 风险 1：Hi-Z 两阶段 跨帧资源管理

跨帧资源是 LX 之前没碰过的领域。需要：

- frame graph 表达 "persistent resource" 概念
- multi-threading/08 RetirePoint 模型扩展到跨帧资源
- 第一帧的 Hi-Z 是空的，要有 fallback 路径

### 风险 2：Compute culling vs Mesh shader 维护两套 shader

两条路径输出相同画面，但实现是两套 shader 代码。维护成本：

- shader 修改要改两套
- 验证两套画面一致（pixel-level diff）
- CI 跑两套硬件路径

可接受，但要 *组织代码避免重复*（共享 cull math 工具函数）。

### 风险 3：Activision 算法在小场景下 overhead

Tile bitfield 构建是 CPU 工作 ~0.5-1 ms。光源 < 20 时不如朴素方案。

需要 *阈值切换*：

```cpp
if (light_count < THRESHOLD) {
    use_naive_lighting();   // 朴素全光源遍历
} else {
    use_clustered_lighting();
}
```

或永远用 clustered（接受小场景的小开销）。

## 6.7 工业级延伸（不在本路线范围）

完成 REQ-H 后离工业级还有：

- **多视角 / 多 swapchain**（多窗口编辑器、VR）
- **Streaming / virtualization**（无限场景，资产按需加载）
- **GPU-side material binding cache**（变体合并）
- **Software rasterization**（小三角 mesh shader 太慢，UE5 Nanite 风格）
- **Shader-driven LOD selection**
- **Persistent thread / persistent kernel** 调度

这些都是 *优化层 / 高级特性*，等 v1 跑稳定再考虑。

## 6.8 一句话总结

**GPU-driven + Clustered = 现代渲染管线的核心。LX 当前 Level 0；推到 Level 4 需要 8 个候选 REQ + frame graph / async-compute / multi-threading 三个调研协同。基础设施 REQ-A/B/C 可以早做，REQ-E 是高风险节点（依赖 frame-graph REQ-038）。Phase 1 REQ-119 + REQ-109 立项是真正的强触发。**
