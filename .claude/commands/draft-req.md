description: Interactive discussion to form a new requirement doc and save it to notes/requirements/
category: Requirements

---

和用户进行**交互式讨论**，把一个改动想法打磨成正式的需求文档，最终写入 `notes/requirements/`。命名、编号、段落结构都对齐已有的 `finished/*.md`。

**Input**: 可选一句话 brief
- `/draft-req` — 完全交互，从零开始问
- `/draft-req "支持 instanced rendering"` — 有初始种子，直接进入 discovery

**IMPORTANT**: 这个命令**只产出文档**。不生成代码、不触发 `/opsx:propose`。默认只新增一份需求文件；只有当"需求编号必须符合实施顺序"要求触发编号调整 / 拆分旧需求，并且用户确认了调整计划时，才允许调整已有的活动需求编号、拆分活动需求文档和更新相关文档引用。完成后由用户决定下一步（`/finish-req` 或 `/opsx:propose`）。

**REQ queue invariant**:
- `REQ 文件号 = 实施顺序`。用户应能按 `notes/requirements/NNN-*.md` 从小到大实施，不需要另记隐藏优先级。
- `一个 REQ 文件 = 一个连续实施周期`。同一文件不能同时包含"现在就做"和"等另一个新需求之后再做"的两段工作。
- 新需求改变旧需求顺序时，优先拆分旧需求；同一实施槽内用 `NNN-a`、`NNN-b` 后缀族表达拆分顺序，不要让一次局部插入牵动所有后续编号。

---

## Steps

### 1. 扫描现有需求库

先建立上下文：

- `Glob notes/requirements/*.md` — 活动中的需求
- `Glob notes/requirements/finished/*.md` — 已归档的需求

对每个文件，读头部 + "实施状态" 段，提取：
- 编号（`NNN`、`NNN-a/b/c`，或历史 `NNNa/b/c`）
- 标题
- 状态（未开始 / 进行中 / 已完成）
- 实施顺序线索（依赖、下游、状态日期、归档位置）

这份清单用于：
- 判断新需求应该插入的实施顺序位置（step 9）
- 如顺序变化，生成活动需求的编号调整方案（step 9，优先用 `NNN-a` / `NNN-b` 后缀族）
- 如旧需求只有一部分需要后置，生成旧需求拆分方案（step 9）
- Phase 5 讨论依赖 / 下游时提示给用户
- Phase 6 冲突扫描

### 2. 没有 brief 时先问主题

若 `$ARGUMENTS` 为空，用 **AskUserQuestion** 提一个开放问题：

> 这次的需求大致是关于什么？用一两句话描述你想要建或改的东西。

把用户回复作为 seed brief。

### 3. Phase 1 — 动机

用 **AskUserQuestion** 依次问（可合并成一条多填问卷）：

1. **当前痛点**: 现在哪里不对？不做这件事接下来的工作会在哪里卡住？
2. **触发时机**: 为什么现在做？是否有刚落地的上游 REQ 让这件事变可行？
3. **失败模式**: 不做的后果具体是什么？（bug / 维护负担 / 无法扩展某特性）

把答复结构化记在心里，后面喂给"背景"段落。

### 4. Phase 2 — 用代码验证现状

基于用户描述的痛点，主动 **Grep / Read** 定位 `src/` 里相关的类、文件、行号。目的是：

- 确认痛点描述准确（代码里确实是这样）
- 找出受影响文件的完整清单（后面的 R 要列出来）
- 发现描述和代码脱节时**停下反问**，不要在错误前提下写需求

报告给用户：

```
## 现状定位

你描述的问题对应到代码里：
- src/core/asset/foo.hpp:42 — `class Foo` 的 `bar()` 方法
- src/infra/loaders/foo_loader.cpp:15 — 唯一调用方
- src/test/integration/test_foo.cpp — 相关测试

这是你想改的全部范围吗？还有别的地方吗？
```

用户补充的文件也过一遍 Grep，保持一致。

### 5. Phase 3 — 目标状态

用 **AskUserQuestion**：

1. **成功标准**: 落地后"好"的状态是什么？给出 2-4 条**可验证**的语句（"某类存在" / "某函数被删" / "某测试通过"）
2. **不变的东西**: 哪些现有行为必须保持不变？（兼容性边界）
3. **API 变化**: 调用方需要改代码吗？范围多大？

### 6. Phase 4 — 分解为 R1..Rn

基于前三个 phase，**你**起草第一版 R 分解，给用户看：

```
## 初步分解

R1: <具体交付项>
R2: <具体交付项>
R3: <具体交付项>
...
```

和用户讨论直到认可。每个 R 必须：

- 有明确的验收标准，能被 `/finish-req` 的 verification 流程独立检查
- 颗粒度适中（太粗 → 不好验证；太细 → R 数量爆炸）
- 属于同一个连续实施周期；如果 R 集合天然分成"前置 cleanup"和"后置重写"，拆成两份 REQ
- 代码引用带行号（`src/core/foo.hpp:42`），行号从 Grep 结果拿，不要编造
- 如果涉及接口签名，给出 C++ 代码片段示例（遵循 finished/004-007 的风格）

### 7. Phase 5 — 边界、依赖、下游

用 **AskUserQuestion**：

1. **明确不做**: 哪些相关但不在本次范围的事情要留给以后？
2. **上游依赖**: 需要哪些已完成 / 进行中的 REQ 先落地？（用 step 1 的清单提示用户）
3. **下游消费者**: 本需求落地后解锁哪些未启动的 REQ？
4. **实施顺序**: 这份需求应排在哪些已知 REQ 之后、哪些活动 / 未启动 REQ 之前实施？如果它应该早于某个当前编号更小的活动需求，要在 step 9 里调整编号。

### 8. Phase 6 — 冲突扫描

写文件前主动扫描：

- `notes/requirements/finished/*.md` — 是否有归档项重叠（风险: 重复实现或重复废弃）
- 活动中的 `notes/requirements/*.md` — 是否有当前进行中的需求与本需求冲突
- `openspec/changes/**`（含 archive）— 是否有变更已经覆盖了本需求的一部分范围

报告发现，示例：

```
## 冲突扫描

⚠ REQ-003a（finished）R2 声明了 `X` 的废弃 — 本需求 R2 也涉及 X。
  → 建议本需求 R2 改为"在 X 已废弃的基础上扩展 Y"，避免重复。

⚠ openspec/changes/archive/2026-04-13-unify-material-system 涉及 MaterialInstance
  → 本需求对 MaterialInstance 的改动需要在此之上。
```

冲突严重时**停下**让用户决定是否调整范围或放弃本次需求。

如果发现新需求会改变某个活动需求的实施顺序，不要只把旧需求整体后移。先判断旧需求是否可拆，并优先用同一数字前缀的后缀族表达：
- 已经可以按原顺序实施的 R 项保留在原位置
- 被新需求阻塞、依赖新需求、或更适合后置的 R 项拆到新的后续需求
- 例如 `020-foo.md` 扩展后，改成 `020-a-foo.md`、`020-b-bar.md`；标题对应 `REQ-020-a`、`REQ-020-b`
- 拆分后每份需求都必须仍是一件可独立实施、可独立验证的事
- 拆分后的新旧文档都要留下简短追踪说明：旧文档说明哪些 R 已移出；新文档说明它从哪个 REQ 拆出、为什么后置

### 9. 计算实施顺序编号 + 文件名

编号原则：

- **REQ 编号表示实施顺序，不表示创建顺序。** 新需求必须插入到它预计真正落地的位置。
- **一个 REQ 文件表示一个连续实施周期。** 如果一个文件横跨两个实施窗口，先拆文件，再排编号。
- 先根据 Phase 5 的上游 / 下游关系确定插入点：上游依赖必须排在前面；本需求解锁的下游需求必须排在后面。
- 已归档 `finished/` 的 REQ 默认视为已发生历史，不主动改号；除非用户明确要求做一次编号迁移。
- 活动需求的编号只是计划顺序。如果本需求应该排在某个活动需求前面，必须调整活动需求编号，不能为了省事给新需求分配最后一个号。
- 如果新需求只导致某个活动需求的一部分后置，必须拆分旧需求文档，而不是让一个 REQ 同时覆盖前置和后置两段工作。
- 局部拆分 / 补充优先使用带连字符的字母后缀，顺序为 `020-a`, `020-b`, `020-c`, `021`。一旦 `020` 被拆分，原文件也要改成 `020-a-*`，不要让 active 目录同时存在 `020-*` 和 `020-a-*`。
- 后缀只表达同一数字槽内的局部顺序；如果是完全独立的全局插队，且不能自然归入相邻数字槽，才生成数字重编号计划。

操作：

- 从 step 1 的扫描结果提取形如 `NNN-*.md`、`NNN-a-*.md` 或历史 `NNNa-*.md` 的前缀
- 按当前编号建立活动需求计划队列，并把 `finished/` 作为已完成前缀参考
- 根据依赖、下游、用户确认的实施顺序，判断新需求插入点
- 对每个会被插队影响的活动需求，检查其 `R1..Rn` 是否需要拆分：
  - 前置部分保留在 `NNN-a`，标题和范围收窄
  - 新插入 / 后置部分写成 `NNN-b`、`NNN-c` 等新的活动需求
  - 原需求的依赖 / 下游 / 测试 / 修改范围要随拆分同步收窄
  - 新旧文档都写一段 blockquote 追踪说明，说明拆分来源、拆分日期、保留 / 移出的范围
- 如果新需求排在所有已知需求之后，编号 = `max(numeric prefix) + 1`
- 如果新需求要插入到活动需求队列中间，生成调整计划：必要时先拆分受影响的旧需求；能用同一数字槽后缀表达时，新需求拿 `NNN-b` / `NNN-c`，后续活动需求编号保持不变；只有全局顺序变化才让后续活动需求依次后移；同步更新文件名、文档标题、正文中的 `REQ-NNN` / `REQ-NNN-a` 引用
- 重编号只覆盖 `notes/requirements/*.md` 中尚未归档的活动需求，以及引用这些编号的 `notes/roadmaps/*.md` / `openspec/changes/**/*.md`；不要修改 `notes/requirements/finished/*.md`，除非用户明确要求历史迁移
- 文件名: `<NNN>-<kebab-case-title>.md` 或 `<NNN-a>-<kebab-case-title>.md`（标题小写 + 连字符，对齐已有文件命名）
- 文档标题: `# REQ-<NNN>: <中英文标题>` 或 `# REQ-<NNN-a>: <中英文标题>`

用 **AskUserQuestion** 让用户确认编号 + 文件名 + 标题。若存在拆分 / 编号调整计划，必须同时展示完整拆分表、改名表和引用更新范围，让用户确认后才能继续。

### 10. 按模板起草文件

严格遵循现有 `notes/requirements/finished/` 的结构：

```markdown
# REQ-<NNN 或 NNN-a>: <Title>

## 背景

<3-8 句，来自 Phase 1 的痛点 + 触发时机。带上 src/... 的具体引用。>

## 目标

1. <来自 Phase 3 的成功标准 1>
2. <成功标准 2>
...

## 需求

### R1: <简短名称>

<详细描述 + 可验证的交付项 + C++ 代码片段示例（如涉及接口签名）>

### R2: <简短名称>

...

### Rn: ...

## 测试

- <Phase 3 验收标准对应的测试入口，指向具体 `src/test/integration/test_*.cpp`>
- <若需要新建测试文件，说明期望覆盖的场景>

## 修改范围

| 文件 | 改动 |
|------|------|
| `src/core/...` | <简短描述> |
| `src/infra/...` | ... |
| `src/test/integration/test_*.cpp` | ... |

## 边界与约束

- <Phase 5 的"明确不做"逐条列出>
- <其他技术约束（性能 / 兼容性 / 层级依赖）>

## 依赖

- **REQ-NNN 或 REQ-NNN-a**（已完成 / 进行中）— 为什么硬依赖
- 外部条件（如 shader 工具链版本）

## 下游

- <本需求解锁的后续工作，点名具体的未启动 REQ 或概念>

## 实施状态

未开始。
```

每段都从前几轮讨论的具体答复中提炼，不要写占位符。

### 11. 预览 + 迭代

把完整草稿展示给用户：

```
## 草稿就绪

文件: notes/requirements/<NNN 或 NNN-a>-<title>.md
顺序调整: <无 / 拆分表 + 改名表 + 引用更新范围>

[完整内容]

操作: yes / edit <section> / redo
```

- `yes` — 如有已确认的拆分 / 编号调整计划，先拆分和重命名活动需求并同步引用，再调用 Write 落地新需求
- `edit <section>` — 只重写指定段（背景 / 目标 / R1 / 依赖 / ...），然后再次预览
- `redo` — 回到 step 6 重新分解 R

### 12. 写入 + 总结

如有已确认的拆分 / 编号调整计划：
- 先拆分受影响的活动需求：收窄原文件范围，新增后置需求文件
- 再对活动需求执行 `Move` / 重命名
- 同步更新被拆分 / 编号调整需求的标题、正文引用，以及 `notes/roadmaps/*.md` / `openspec/changes/**/*.md` 中的 `REQ-NNN` / `REQ-NNN-a` 引用
- 再 `Write notes/requirements/<NNN 或 NNN-a>-<title>.md`

如无编号调整计划，直接 `Write notes/requirements/<NNN 或 NNN-a>-<title>.md`

报告：

```
## draft-req 完成

已创建: notes/requirements/<NNN 或 NNN-a>-<title>.md
- 编号: REQ-<NNN 或 NNN-a>
- 标题: <title>
- 需求项: R1..R<N>
- 顺序调整: <无 / 拆分 REQ-020 为 REQ-020-a、REQ-020-b；或 REQ-010→REQ-011, ...>
- 上游依赖: <REQ-XXX, REQ-YYY>
- 下游解锁: <REQ-ZZZ, ...>

下一步可选:
- /opsx:propose — 把需求转成 openspec change 开始实施
- /finish-req <NNN 或 NNN-a> — 代码落地后用它做 verification + 归档
```

## Guardrails

- **讨论价值在前半段**: Phase 1-5 是这个命令的核心，不要为了快把它压缩成"你要什么我就写什么"。用户如果想直接跳到生成，让他用 `/opsx:propose`。
- **代码引用必须真实**: 每次写 `src/... :LINE` 之前必须用 Grep 确认过行号，禁止编造。
- **中文为主**: 匹配现有 `notes/requirements/` 风格。代码符号 / 类名 / 文件名保留英文原形。
- **只落在顶层**: 新需求永远写到 `notes/requirements/`，不写 `finished/`。只有 `/finish-req` 有权移入 `finished/`。
- **不随意改已有需求**: 发现冲突时报告 + 建议。只有实施顺序不匹配、已展示拆分 / 编号调整计划并得到用户确认时，才允许修改活动需求；禁止静默拆分或改号。
- **优先降低管理心智**: 活动需求列表必须按编号就是实施队列。如果一个旧需求横跨多个实施阶段，先拆成多个顺序清晰的需求，而不是让用户记住"REQ-010 的一半要等 REQ-011 之后"。
- **一次只写一份**: 发现讨论范围过大覆盖了两个独立话题，停下建议拆分成两次 `/draft-req`。
- **不代用户做权衡**: Phase 5 的"明确不做"必须由用户点头，不要自己默默砍范围。
- **不触发代码修改**: 输出只限文档；除已确认的实施顺序拆分 / 编号调整外，默认只新建一个需求 `.md` 文件。
- **若用户中途问别的问题**: 答完后明确回到"我们刚才在 Phase <N>，要继续吗？"
- **尊重已归档的历史**: 发现本次讨论涉及的模块被某个 finished REQ 明确规定过，优先引用而非改写历史。

## 编号与实施顺序的处理

Step 9 不能默默用 `max+1`。先判断本需求在实施顺序里的位置，再给出编号。如果用户提出的编号和实施顺序不一致，说明冲突并给出调整方案；如果旧需求的一部分需要后置，先拆旧需求并优先使用 `NNN-a` / `NNN-b` 后缀族。如果用户坚持保留冲突编号，停止写入，直到用户调整实施顺序、接受拆分 / 编号调整，或明确要求做一次历史迁移。

用户要求插入一个紧贴旧需求的补充或拆分（例如 `REQ-020` 讨论后扩成两个连续阶段）时，允许并优先用字母后缀，但必须：

- 先确认基础 REQ 存在且状态合适
- 若基础 REQ 仍在 active 且要拆分，把原文件也改成 `NNN-a-<title>.md`
- 在新文档的"依赖"或追踪说明里明确写"作为 REQ-NNN 或 REQ-NNN-a 的补充 / 拆分"
- 文件名用 `NNN-b-<title>.md` 这类带连字符后缀；历史 `NNNa-<title>.md` 可保留但新文件不用这种格式

## 使用场景

- **新功能想法成熟到可以写需求了** → `/draft-req "支持 instanced rendering"`
- **重构想法还在雏形** → `/draft-req` 从零开始
- **发现某个大 REQ 应该拆成子项** → 先在原 REQ 讨论，然后用 `/draft-req` 形成拆出来的子需求
- **代码实施过程中发现漏了一环** → `/draft-req "REQ-XXX 补充: 处理边界条件"`
